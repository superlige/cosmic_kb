# Cosmic KB 项目理解

理解既有苍穹项目，不默认生成业务代码。使用 `cosmic_kb` MCP 获取确定性证据，读取本地源码补全上下文，并把结果整理成可复核的业务结论。

## 核心纪律

1. 先取证，后结论。不要按字段名、类名、包名或训练记忆猜测中文名、绑定关系、事件语义和保存结果。
2. 为每个判断标记 `confirmed`、`likely` 或 `unknown`，并说明依据。没有证据时明确写 `unknown`。
3. 通过 MCP 查询字段、单据、插件绑定和苍穹语义；直接读取本地源码全文，保留文件和行号。不要把源码、KB 或报告上传到公网站点。
4. 不要默认生成或改写 Java 业务代码。用户明确要求修改时，再把已确认的理解作为修改依据。
5. 把 `kd.bos.*`、`SaveServiceHelper` 等未知平台符号视为外部 SDK，按需调用 `cosmic_semantics` 核实，不要当成本项目缺陷。
6. 工具返回冲突、歧义或信息不足时不要自行挑选答案；缩小标识范围重新查询，仍无法确认则保留存疑。

## 工具使用原则

以 MCP 当前 schema、description 和 server instructions 为参数事实源，不在本 Skill 复制完整签名。按已知起点路由：

| 已知起点或问题 | 首选工具 |
|---|---|
| 精确字段坐标；谁读写、事件、落库证据 | `trace(kind="field")` |
| 精确操作坐标；绑定插件、程序化上游、未解析入站、跨单据下游 | `trace(kind="operation")`，不先调 `bill` |
| 单据整体操作集、全部插件、插件车道、本单据插件对外触发的影响面(outbound_triggers) | `bill` |
| 字段/实体/单据标识或插件类名反查 | `resolve_fields` |
| 插件事件、SDK、DynamicObject、事务和入库语义 | `cosmic_semantics` |

- 对源码中尚未核对的标识批量调用 `resolve_fields`；MCP 已返回的精确坐标无需重复解析。
- `trace` 或 `bill` 返回 `pagination.complete=false` 时，按 `pending.next_cursor` 查询至 `complete=true`。
- 出现 `need_clarification`、`mismatched_*`、`invalid_request` 时，按候选或真实归属修正参数。
- 出现 `coarse_only`、`unlocated`、`dynamic_writers`、`unresolved_inbound`、未绑定源码类或动态字段时，读取对应源码并单列仍无法确认的部分；这些状态不等于“不存在”。

## 工作流

### 建立项目全貌

1. 确认 KB 和 MCP 已可用；若不存在、版本不符或未连接，停止取证并转用 `cosmic-kb-setup`。
2. 用户需要项目地图时，运行 CLI-only 的 `cosmic_kb report map`。
3. 用户需要规模、排障入口或风险热点时，运行 CLI-only 的 `cosmic_kb report overview`。
4. 从用户关注的模块、单据、字段或插件进入下列单点工作流，不要仅凭全貌报告推断字段行为。

### 字段追溯

1. 从源码上下文识别可能的单据、实体和字段坐标。
2. 用 `resolve_fields` 批量核对本轮出现的陌生标识；确认中文名、主实体/分录/子分录/基础资料路径，以及枚举或引用语义。
3. 用已确认的最精确坐标调用 `trace`。存在歧义时按候选重新查询，不要把裸字段的多个归属合并成一个结论。
4. 翻完所有分页，按源码类和方法整理读取点、写入点、事件与保存证据。
5. 对粗粒度命中、动态写入和未定位来源读取完整方法；仍缺保存链路时标为 `unknown`。
6. 按“字段排障结论”模板输出。

### 插件或方法解释

1. 只有插件类名时，先用 `resolve_fields(kind="plugin")` 反查绑定；需要完整单据上下文时再调用 `bill`。
2. 核对绑定单据、操作和启用状态。`enabled=null` 只表示未确认，不要解释为已禁用。
3. 已确认具体操作时，用 `trace(kind="operation")` 补充绑定插件、程序化上游、未解析入站和跨单据下游；不要无理由先调 `bill`。
4. 读取插件源码，定位触发方法、条件分支、字段读写和项目内服务调用。
5. 把源码中新出现且尚未核对的字段、实体和单据标识批量交给 `resolve_fields`。
6. 对不确定的插件事件、SDK 或保存语义调用 `cosmic_semantics`，再按“插件/方法作用解释”模板输出。

### 操作影响分析

1. 已知 `单据.操作key` 时，直接调用 `trace(kind="operation")`，不把 `bill` 当前置步骤。
2. 翻完所有分页，检查 `plugins`、`triggered_by`、`unresolved_inbound`、`triggers_downstream`；空的明确入站不代表不存在动态嫌疑。
3. 根据 `plugins` 读取操作插件源码，确认事务事件、条件分支和实际读写的实体字段。
4. 批量 `resolve_fields` 核对源码中新出现的标识，再对实际涉及字段分别调用 `trace(kind="field")` 核对读写和入库证据。
5. 对跨单据创建、下推、反写和审核回写继续追踪目标操作与字段，不把调用名称直接当成已落库证据。
6. 只有需要单据整体操作集、全部插件车道或本单据插件对外触发的影响面（`outbound_triggers`）时，
   才补充调用 `bill`；操作 key 或目标单据解不出的入站嫌疑（含挂不上操作坐标的表单插件外发）已
   并入 `trace(kind="operation")` 的 `unresolved_inbound`，对该操作的调用查一次即完整，不必为此
   再补查 `bill`。
7. 按“操作影响分析”模板汇总；明确程序化取证不覆盖人工、工作流和设计器的全部入口。

## 回答格式

遵循“结论 → 证据 → 存疑 → 下一步”。省略与问题无关的栏目，但不要省略会改变可信度的存疑信息。

### 字段排障结论

```text
实体：<单据标识>（<中文名>）
字段：<字段标识>（<中文名>）｜路径：<主实体/分录/子分录/基础资料>

结论：<谁读取或修改、是否有入库证据、主要疑点>

写入点（按置信度排序；没有则写“未发现写入点”）：
  ✅ confirmed  <类>#<方法>  事件=<event>  行号=<file:line>  依据=<保存或事务证据>
  ~  likely     <类>#<方法>  事件=<event>  行号=<file:line>  依据=<间接证据>
  ?  unknown    <类>#<方法>  事件=<event>  行号=<file:line>  原因=<缺失证据>

未定位/存疑：
  - <歧义、动态写入、粗粒度命中、缺少保存链路等；没有则写“无”>

读取点：<仅在相关时列出类、方法和行号>
下一步建议：<具体到要查询的坐标、工具或源码位置>
```

### 插件/方法作用解释

```text
类：<全限定名>｜绑定：<单据/操作/启用状态>

结论：<业务作用及触发事件>

写入字段（按置信度排序；没有则写“无”）：
  ✅ confirmed  <字段标识（中文名）｜路径>  行号=<file:line>  依据=<保存或事务证据>
  ~  likely     <字段标识（中文名）｜路径>  行号=<file:line>  依据=<间接证据>
  ?  unknown    <字段标识（中文名）｜路径>  行号=<file:line>  原因=<缺失证据>

读取字段：<相关时列出>
调用的项目内方法/服务：<目标类或服务；没有写“无”>
未定位/存疑：<绑定、动态路径、事件或保存链路问题；没有写“无”>
风险点：<有证据支持的风险；没有写“无明显风险”>
下一步建议：<具体查询或源码位置>
```

### 操作影响分析

```text
操作：<单据.操作key（中文名）>
绑定操作插件：<类名、启用状态和源码位置；没有则写“未发现”>

结论：<该操作的业务影响和已确认的调用方向>
明确上游 triggered_by：<调用类、方法、上游单据和行号；没有则写“未发现明确上游”>
未解析入站 unresolved_inbound：<逐条嫌疑及未确认原因；没有则写“无”>
跨单据下游 triggers_downstream：<目标操作、调用位置和 next_trace；没有则写“无”>
影响字段：<按 confirmed/likely/unknown 列字段、路径和入库证据>
能力边界：<未覆盖的人工、工作流、设计器入口；表单插件外发等挂不上操作坐标的嫌疑已并入
  unresolved_inbound，不算能力边界>
下一步建议：<需要继续追踪的操作坐标、字段或源码位置>
```

## 完成检查

输出前确认：

- 所有陌生标识都已核对，未用字面含义替代元数据。
- 插件绑定来自 `bill` 或插件反查，不来自类名猜测。
- 所有相关分页均已完成。
- 已知操作坐标时直接使用 `trace(kind="operation")`，没有无理由先调 `bill`。
- 已检查并单列 `unresolved_inbound`，没有把它或明确入站为空解释成不存在程序化嫌疑。
- 每条关键结论都有工具结果或 `文件:行号`。
- 动态、粗粒度和未定位证据已单列，没有被解释成“不存在”。
- 入库结论符合 `cosmic_semantics` 的当前规则；证据不足时使用 `unknown`。
