## 扩展元数据 · 底层库元数据源（只读直连）✅（2026-07-02）

**背景**：ISV 导出的 dym 只含扩展部分，原厂标准单据（`bd_customer` 等）的完整元数据不在包里，
导致扩展单据字段级排障"结构性半盲"。平台底层库 `t_meta_formdesign` / `t_meta_entitydesign`
两张设计表按 `fnumber` 各存一份明文 fdata XML，组合解析即可拿回**全部**原厂/扩展元数据。

**目标**：新增只读 DB 元数据源（新包 `cosmic_kb/dbmeta/`），把两表 fdata 合成现有 `MetaModel`，
解析器零改动；严格只读、可扩展多数据库、配置化。**红线 #1 受控例外**：仅只读取元数据，连接串
不落公网，结果照旧只进本机 KB。只读三重防线：SQL 白名单 + 服务端只读会话 + 永不 commit。
本次只做"DB→MetaModel"纵切 + CLI，暂不接入 `build`/`bridge`（下一步）。

**验收结论**：✅ 通过。`tests/test_dbmeta.py` 31 例：真实样例 `bd_customer` 合成正确（中文名/
落库字段/原厂插件/操作全部解析到）；降级、白名单拒写、配置装载 + 口令环境变量覆盖 + 脱敏均验证。

## 扩展元数据 · 原厂元数据并入 KB（build/bridge 接入 + 选择性摄取）✅（2026-07-02）

**背景**：上条 `db-meta` 是"死胡同"——不接入 `build`/`bridge`，KB 里没有任何原厂字段，
`trace bd_customer.name` 永远空。同一原厂实体会有两份元数据：DB 原厂（标准字段/操作，
插件是标准产品插件、无源码对 KB 无意义）+ 本地扩展 dym（`cqkd_bd_customer_ext` 这类，增量
字段/插件/操作）。苍穹原厂实体页面极多，不能全量摄取——用户拍板只摄取**代码库真访问到**
的非 ISV 前缀实体，二开故障排查用途，不做项目普查。

**目标**：① 用户拍板的合并范围——原厂**字段+操作**并入，原厂**插件**（无源码）一律不要；
② 发现信号取"源码字面量粗扫"∪"`field_access` 已解析出的非本地 form_key"，各自记来源，
不自动拉取/合并（红线 #4：证据先摆出来）；③ 合并后以**原厂 key**（如 `bd_customer`）为主，
本地扩展合并进它名下，查扩展 key 能重定向提示到原厂 key。

新增：`metadata/extension.py::detect_extension`（InheritPath 非空 + `<isv>_<fnumber>_ext`
命名双信号推候选，DB 查得到才算确认，不臆造）；`metadata/merge.py`（`strip_vendor_plugins`/
`merge_vendor_extension`——重写扩展表头实体 parent_id 与字段 entity_key 指向原厂表头，
字段冲突保留扩展侧+记 warning/`build_extension_alias`——扩展原 key 留空壳别名行）；
`dbmeta/discover.py`（两路候选发现，CLI `db-meta --discover <源码根>`）；
`dbmeta/integrate.py::apply_vendor_metadata`（编排：detect→DB 拉取→合并/引用并入，
查不到就跳过如实提示，不中断整体 build）；`build`/`bridge` 加 `--vendor <fnumber...>
--db-config <配置>`；schema **v12**（`form` 加 `is_extension`/`extends`）；`trace`/`bill`
命中扩展别名行时顶层提示重定向到原厂 key。

**堵住的一个风险**（红线 #4）：只选择性合并少数原厂实体后，`id`/`name`/`number` 这类
苍穹平台通用系统字段（`kind='platform'`）会在字段反查索引里"看似唯一"——其实只是没并入
其它同样带这字段的原厂实体，若不排除会让毫不相关的孤立 helper 被误判定位。
`java/analyze.py::_field_form_index` 已排除 `kind='platform'`，追不到来源就诚实留 None，
不是回归。

**验收结论**：✅ 通过（Claude 开发 + 写用例；Codex 执行全量测试 + 提交）。新增
`tests/test_metadata_extension.py`（11 例：detect_extension 双信号/合并端到端用真实样例
`samples/db_xml/bd_customer_*` + `samples/bill/cqkd_bd_customer_ext.dym`——实体重挂/字段
entity_key 重写/字段冲突保留扩展侧/别名行清空）、`tests/test_dbmeta_discover.py`（8 例：
两路候选 + 并集排序）、`tests/test_dbmeta_integrate.py`（4 例：合并/引用并入/查不到跳过/
空列表 noop，假 DbMetaReader 不连真库）、`tests/test_vendor_extends_redirect.py`（6 例：
trace/bill 命中别名行的重定向提示，rich+compact+文本三路）；`test_form_key_backfill.py`
新增 1 例（`_field_form_index` 排除 platform 字段）。本地全量 `pytest -q` **407 passed**
零回归。真实样例端到端验证：`bd_customer` 合并出 163 个字段（原厂 123 + 扩展 40，冲突 0）、
插件只剩扩展的 1 个（原厂 26 个平台插件全部排除）、操作 43 个（原厂+扩展）；`db-meta
--discover` 对真实库跑通，候选列表按 `field_access`/`coarse` 来源分列。

```powershell
pytest tests/test_metadata_extension.py tests/test_dbmeta_discover.py tests/test_dbmeta_integrate.py tests/test_vendor_extends_redirect.py -q  # 29 例
cosmic_kb db-meta --discover "<项目源码根>" --meta <dym|zip|目录> [--db cosmic_kb.db]  # 候选原厂 key（不自动合并）
cosmic_kb build "<源码根>" "<dym|zip|目录>" --vendor bd_customer --db-config cosmic_db.json  # 重建 KB（schema v12），并入原厂字段/操作
cosmic_kb trace bd_customer.name          # 应能查到（原厂字段并入）
cosmic_kb trace cqkd_bd_customer_ext.name # 提示"已合并进 bd_customer，请改查"
```

**后续（范围外）**：`null_reason` 新增 `standard-field-not-local` 码（本方案已让合并实体的
原厂字段直接可定位，未选中的原厂字段仍落现有通用成因桶，非回归，暂不需要）；trace 对扩展
别名 key 的自动转发查询（而非仅提示）；除 PostgreSQL 外的其它数据库驱动。

## 扩展元数据 · `--discover` 真实项目降噪（field_access 补过滤 + `--meta` 免传）✅（2026-07-02）

**背景**：用户在真实项目（626 个本地表单）上跑 `db-meta --discover` 做全量验证，候选列表
4584 个里九成以上是自家 `cqkd_` 前缀噪声，且不少根本不是实体 key（如 `cqkd_htid` 其实是
"合同id"字段、`"createtime desc"` 是排序子句片段）。排查发现根因：`from_field_access`
（`_resolve_entity_arg` 解析 ORM 调用参数产的候选）这条信号路径**完全不做任何过滤**——
不排除已知 key，也不检查形状，只做了 SQL 侧"不在 form 表"这一层；而此前的已知 key 集合
（`known_keys_from_models`）又只覆盖 form/entity，不含 **field**，导致大量实为字段 key 的
噪声被误判成候选表单。此外 `--meta` 是过滤生效的必要条件，KB 已经建好后还要重新传一遍
元数据包，操作繁琐。

**目标**：① `known_keys_from_db`（新增）直接从已建好 KB 的 `form`+`entity`+**`field`**
三表取全量已知 key，`--db` 一给就自动生效，不再依赖 `--meta`；② `isv_prefixes_from_db`
（新增）同理从 KB 的 `form.key` 前缀分布现算 ISV 前缀，逻辑与 `namespace.discover_meta_prefixes`
一致（数据源换成 KB）；③ `from_field_access` 的结果在 `discover_candidates` 里统一套用
"已知 key 排除 + `_KEY_SHAPE_RE` 实体形状校验"（此前只有 `from_source_scan` 一路在做）。

**验收结论**：✅ 通过。真实项目（626 表单）验证：`--discover` 候选总数 **4584 → 211**
（降噪 95%），且不再需要 `--meta` 即可拿到干净结果；`field_access` 命中数最高的前几名
（`bos_attachment`/`bos_org`/`sch_job`/`sim_vatinvoice`/`bos_user`/`bd_customer` 等）
全部是真实平台标准实体，无 `cqkd_` 噪声混入前排。`known_keys_from_db` 对缺表（旧
schema/精简测试库）容错跳过，不报错。新增 `tests/test_dbmeta_discover.py` 6 例：
form+entity+field 并集、缺表容错、ISV 前缀现算、`field_access` 排除已知字段 key/非实体
形状、`--db` 免传 `known_keys` 自动生效。本地全量 `pytest -q` 409 passed 零回归。

```powershell
pytest tests/test_dbmeta_discover.py -q  # 14 例
cosmic_kb db-meta --discover "<源码根>" --db cosmic_kb.db   # 不用再传 --meta，噪声大幅下降
```

**后续（范围外）**：`field_access` 剩余的少量 `cqkd_` 前缀候选（KB 里 form/entity/field
三表都查不到，但确实被 ORM 调用引用到）不是噪声，是"本地元数据疑似有解析/摄取缺口"的
信号，暂保留在候选列表里供人工判断，不做进一步抑制（抑制会把真缺口也一并藏起来）。

### 增补 · 排除物理库前缀 `fk_`/`tk_` 噪声 ✅（2026-07-02）

**背景**：用户拿到 211 候选后人工抽查，发现 `fk_` 开头的其实是实体表**字段的物理列名**、
`tk_` 开头的是实体的**物理表名**（如源码里原始 SQL 拼接
`"...JOIN tk_cqkd_assetcard_soe asset ON entry.fk_cqkd_assetname = asset.fid"`），
和实体 fnumber 形状完全一样（小写下划线）但语义是苍穹**物理库**命名，不是逻辑实体标识，
之前的过滤（已知 key + `_KEY_SHAPE_RE`）挡不住这类。全部来自 `coarse` 一路（原始 SQL
字符串字面量），`field_access` 一路（走 ORM 调用参数解析）本身没有这类噪声。

**目标**：新增 `_PHYSICAL_SCHEMA_PREFIX_RE`（`^(?:fk|tk)_`），在 `from_source_scan` 与
`discover_candidates` 的 `field_access` 过滤里统一套用（不区分项目自家前缀，`fk_`/`tk_`
是苍穹平台级物理库保留前缀，与 `isv_prefixes` 分开处理）。

**验收结论**：✅ 通过。真实项目候选 **211 → 164**，`fk_`/`tk_` 前缀候选清零。新增
`tests/test_dbmeta_discover.py` 2 例（`from_source_scan` 排除物理前缀字面量、
`discover_candidates` 的 field_access 路径同理排除）。本地全量 `pytest -q`
**411 passed** 零回归。

```powershell
pytest tests/test_dbmeta_discover.py -q  # 16 例
cosmic_kb db-meta --discover "<源码根>" --db cosmic_kb.db
```

### 增补 · 三类确定性信号取代"形状过滤候选" + build/bridge 自动摄取 ✅（2026-07-03）

**背景**：`4584→211→164` 两轮降噪后候选依旧不可信——`from_source_scan`（粗扫字符串字面量+
形状过滤）与 `field_access.form_key`（ORM 调用参数"抓全部字面量取首个"）本质都只按"字符
形状"把关，字段名/map·json 构造 key/操作编码常量/排序子句等任意小写下划线字符串都会混进
候选。用户拍板（替代 2026-07-02 的"只列候选不自动拉取"）：改用**三类确定性信号**，命中即
"必定摄取"，且不再要求人工手敲 `--vendor`。

**目标**：`dbmeta/discover.py` 重写为三信号扫描——① `from_extensions`（本地 `_ext` 扩展 →
母体必摄取，复用 `detect_extension`）；② `from_orm_calls`（`BusinessDataServiceHelper` /
`QueryServiceHelper` 查询，按接收者+方法名**重载参数位规则表**精确取 entityName，常量引用
经 `ConstantTable` 解析，歧义一律跳过不猜）；③ `from_operation_calls`
（`OperationServiceHelper.executeOperate` / `DeleteServiceHelper.delete` /
`SaveServiceHelper.save` 系列，`save(var)` 仅同方法体回溯变量的 ORM 来源，跨方法诚实跳过）。
`build`/`bridge` 给了 `--db-config` 即自动跑三信号发现并入 `--vendor`（并集，去重），旧两路
宽松候选彻底删除；`db-meta --discover` 改为同一套信号的干跑预览（不连库、不摄取）。

**验收结论**：开发 + 用例已完成，`tests/test_dbmeta_discover.py` 重写为 32 例（三信号逐条
消歧 + 反例：map/json 构造 key、字段名、操作编码常量、排序子句均不再命中）、
`tests/test_dbmeta_integrate.py` 增补 5 例（CLI 编排层 auto∪manual 并集、零 `--db-config`
不碰 DB、只给 `--vendor` 报错、无候选无手动指定不连库）。已用最小 Java 样例逐条手动核对
三信号取值/回溯/跨方法跳过行为与预期一致，CLI `db-meta --discover` 端到端跑通。
全量回归 `pytest -q` 待 codex 执行确认零回归；真实项目端到端（候选噪声应清零）待用户环境验证。

```powershell

pytest -q                                                                 # 全量回归（codex 执行）
```

### 增补 · 修复扩展候选 fnumber 截断导致原厂元数据查不到 ✅（2026-07-03）

**背景**：真实项目实测三信号自动摄取，信号①（扩展母体）命中的候选里有一批查库返回
"底层库未查到"，人工核对发现是**解析错误**而非底层库真无此数据——本地扩展 dym 的 key
（如 `cqkd_cas_bankjournalf_ext`）本身因平台标识长度限制被**截断**过，`detect_extension`
按 `<isv前缀>_<候选>_ext` 命名规律反推出的候选（`cas_bankjournalf`）也就跟着错了，真实
原厂标识其实是 `cas_bankjournalformrpt`。按错的候选去 `t_meta_formdesign`/`t_meta_entitydesign`

### 增补 · 批量查库替代逐候选循环查库（自动摄取变慢）✅（2026-07-03）

**背景**：用户反馈真实项目自动摄取"执行速度很慢，是不是循环查库了"。排查确认：上一条
增补修好 fmasterid 关联后，`apply_vendor_metadata` 仍是 `for fnumber in fnumbers:` 循环里
逐个调 `read_model`/`read_model_via_local_ext`，每个候选各发 2 条单行 `WHERE fnumber = %s`
SELECT——一次自动摄取常有几十个候选（用户样例 29 个），就是 50~100+ 次同步网络往返；DB
若不在本机（本地优先红线 #1 的受控例外，见 `project_dbmeta_source` 记忆），每次往返的网络

**目标**：把"N 个候选 = N 次查询"改成"无论 N 多大，固定几条 SQL"。有本地扩展命中和无

分两组，分别批量查一次（各自固定 2 条 SQL，合计 4 条，不随候选数增长），查完在内存里按

### 增补 · 修复无扩展命中分支同 key 重复并入 ✅（2026-07-03）

**背景**：用户真实项目验收原厂元数据拉取时反馈：同一单据同一字段的 `trace` 结果返回了

**根因**：`apply_vendor_metadata` 的"无本地扩展命中"分支（`fnumber` 没被 `detect_extension`

**修复**：该分支 append 前先在 `result` 里找同 key 的本地模型；找到则整体移除、改走已有的

`coarse.coarse_only`。

**验收结论**：`tests/test_trace_compact.py` 更新两处旧断言（`test_compact_default_has_writers_and_reader_overview`

## `read_source` 常量条目截断优先级修复（复盘二：真实翻车仍复现）✅

**背景**：上条上线后用户拿一份真实 394 行文件（`InvoiceWriteBackTask.execute`，命中 58 个 KB 已知
key，其中含 `FacturateCon.TEMPORARYID`/`FacturateCon.ENTRY5_TOTALAMOUNT`/`TemporaryStopCon.ENTITY`
三处限定常量引用）实测，MCP `read_source` 返回的 `field_names` 只展示了 25 条（`field_names_next_cursor:
"field_names@25"`），且三处常量条目全部不在展示范围内——大模型据此判断"常量没解出来"，又退回按
`TemporaryStopCon` 的英文语义猜。排查发现**不是解析失败**，是**截断顺序**坑了：`read_source()` 构建
`field_names` 字典时先塞普通字面量 key（`found` 按字母序），常量条目最后才 `field_names[expr]=...`
追加进去；紧凑投影 `_build_rs_compact` 按预算裁剪时取的是 `dict.items()` 的**前** N 条（`fn_full[:cap]`）
——大文件普通 key 一多，预算总在够到常量条目之前就耗尽，常量标注被稳定截断在翻页线之外，模型即使遵守
"`field_names_next_cursor` 非 null 不下覆盖性结论"的纪律也大概率不会为了几个不起眼的 key 硬翻到底。

**目标**：让常量条目（本来就少、且是"字面值不在正文里、工具帮不了就只能瞎猜"的最高价值标注）在预算
紧张时**优先存活**，不因普通字面量 key 数量多而被挤出首页。

**实现**：`report/read_source.py::read_source()` 把 `field_names` 字典的组装顺序倒过来——先建
`const_field_names`（常量条目），再 `{**const_field_names, **{普通字面量 key 的分类结果}}` 合并，常量
条目排在字典**最前**。下游 `_build_rs_compact` 的 `fn_full = list(rich["field_names"].items())` 天然
继承这个顺序，只要预算 ladder 选中的 cap ≥ 常量条目数（几乎总成立——常量引用受 `_MAX_CONST_REFS` 封顶
且实际出现频率远低于普通字段字面量）就不会被截断；即便预算紧到 ladder 兜底到最窄档 cap=1，留下的也是
常量条目而非随手一个字母序靠前的普通 key。CLI 富投影 `render_read_source` 顺带受益（常量标注展示在前）。

**验收结论**：✅ 通过，Claude 本轮自测（用户临时授权，非常规分工）：`tests/test_read_source.py` 新增
`test_read_source_orders_constant_entries_before_plain_keys`（锁字典插入顺序）+
`test_read_source_compact_keeps_constant_entry_under_tight_budget`（预算压到 1600 字节、逼 ladder 兜底
到 cap=1，断言存活的仍是常量条目）；`pytest -q` 全量 **453 passed, 4 skipped**。CLAUDE.md 已同步记录
"截断顺序坑"这类根因，供日后排查同类"工具返回不完整"问题时先查截断/分页选取逻辑。


## `read_source` 游标翻页强制提醒（复盘四：数据是对的，但模型没翻到那一页）✅

**背景**：复盘三修好表单标识兜底后，用户拿同一份真实样本第四次实测（`tests/error_return/第四次测试.txt`）。
这次不一样——大模型自己复盘了完整时序：`read_source` 第一次调用返回时 `field_names_next_cursor`
非 null（`"field_names@25"`），`cqkd_zkd`「转款单」其实已经在被截掉的那一页标注里；模型只调用了
`read_source(cursor="content@88")` 续读源码正文，却没调 `read_source(cursor="field_names@25")`
续取标注，于是把 `load("cqkd_zkd")` 凭变量名 `srctransid` 谐音猜成"转账单"。模型自己给出的诊断：
①`content` 游标的翻页说明给了完整因果链（游标值→怎么传→续读什么），`field_names` 游标只有"同法
翻页"四字，没有等价的因果链被激活；②"表单标识也在 field_names 里"这条关键信息写在文末独立段落，
离游标说明很远，决策那一刻没被激活；③警戒措辞是"非 null 时不能下覆盖性结论"这种否定式约束，容易
被"少调用"倾向压过；④两个游标并列描述，没有告诉模型它们同等重要（都必须翻到 null）。

**关键判断**：这轮与前三轮（复盘一/二/三）性质不同——`field_names` 里的数据本身是对的（"转款单"
已经算出来了，只是在下一页），不是数据/候选池/截断顺序的 bug，是**返回值的信息呈现方式没能让模型
在正确的时刻执行正确的动作**。加更多说明文字解决不了注意力分散，得把行动指令焊进模型一定会看到的
地方——返回值本体（游标字段旁）和分页返回（page 本体），而非只留在长文本 note 里。

**实现**：`report/read_source.py`——① `_rs_overview()` 里 `field_names_next_cursor`/
`content_next_cursor` 非 null 时各自新增一个同级 `*_next_cursor_action` 字段，直接给出"必须调
read_source(cursor=...) 续取，取全前不得下结论/不得凭谐音猜"的具体指令（含 `cqkd_zkd`→"转款单"
真实翻车案例复述）；② `_rs_page_field_names`/`_rs_page_content` 翻页返回的 `page` 里，`next_cursor`
非 null 时同样带 `page.reminder`，翻到最后一页（`next_cursor=null`）后消失，逐页续翻都不会丢提醒；
③ `_RS_COMPACT_NOTE` 从单段长文字改写为编号行动清单（①content②field_names③都读全才能下结论），
两个游标并列给出同等权重的指令，否定式"不能下结论"改成肯定式"必须先做 X"，并补一句反制"少调用"
倾向（翻页省的 token 远少于猜错返工的成本）；④ MCP `tool_read_source` 文档段落同步改写为对齐结构，
把"表单标识也在 field_names 里"这条信息挪到游标说明旁边而非只留在文末独立段落。

**验收结论**：✅ 通过，Claude 本轮自测：`tests/test_read_source.py` 新增 4 条——
`test_field_names_next_cursor_carries_action_reminder`（field_names 游标带可执行 action，非只有

逐页存在、到末页消失）、`test_content_next_cursor_carries_action_reminder`（content 游标同样带
action）、`test_content_page_reminder_present_until_exhausted`（content 翻页 page.reminder 到末页
消失）；`pytest -q` 全量 **460 passed, 4 skipped**。

## MCP 描述精简（复盘四·续：结果对了，但描述太占上下文）✅

## read_source 两步取证协议（keys 显式核对，堵全文盲扫噪音）⬜（待 codex 跑测试确认）

**背景**：用户实测反馈 `read_source` 返回"太过臃肿……超多待选字段，token 消耗大"。查代码定位到
根因——`read_source` 默认对正文做**全文盲扫**：把正文里每个 token 与 KB **全量**已知 key（field+
entity+form，成百上千个）取交集，而苍穹很多字段 key 本身是 `remark`/`amount`/`org` 这类**通用英文
词**，会和完全无关的局部变量撞名，每个撞中的 key 还要走三档分类+坐标列表，`field_names` 因此又大
又多是噪音。这不是格式问题，是"扫描面"选错了——盲扫天然分不清"这是本文件真正引用的业务标识"还是
"凑巧撞词的无关变量"，只有读得懂代码语境的模型自己能分辨。

**方案**（用户确认：MCP 首次调用完全留空，不留自动信号）：`read_source()`/`read_source_compact()`
加 `keys: list[str] | None` 参数，形成三态：
- `keys=None`（缺省）——保留老的全文盲扫，**CLI/Web 富投影不受影响**（终端/HTTP 无 32KB 限制，人眼
  看全量标注更省事，且不碰默认档，既有测试零回归）。
- `keys=[]`（MCP 压缩投影 `read_source_compact` 的默认状态，未传 `keys` 时自动落到此档）——**不做任
  何标注**，只给源码正文，`field_names` 为空；note 明确提醒"为空≠本文件没有业务字段，只代表还没
  核对"，防止模型误读。
- `keys=[...]`（显式列表）——**只核对调用方传入的这些标识**，不扫正文，天然零噪音；列表里形如
  `Xxx.CONST_NAME` 的条目按限定常量引用处理（查 `java_constant` 表，逻辑与盲扫档完全复用），其余
  按普通字段/容器/表单 key 走原有三档分类（`_classify_key`/`_classify_form_key`，含本文件
  `field_access` 消歧——这是显式传 `resolve_fields` 拿不到的增量价值，故未把两条路径合并）。

两步取证流程：① 模型首次调 `read_source(relpath)` 只拿源码正文；② 读完自己把不认识的业务标识/
限定常量引用列出来，`keys=[...]` 二次调用才拿到核对结果。`INSTRUCTIONS`、`tool_read_source`/
`tool_resolve_fields` docstring、模块内 `_RS_COMPACT_NOTE` 同步改写为这套两步协议的措辞。

**实现**：`cosmic_kb/report/read_source.py`——`read_source()` 按 `keys is None` 分支legacy 全文盲扫

## `resolve_fields` 补齐"分录.字段"/"单据.分录.字段"复合限定符 ⬜（待 codex 跑测试确认）

**背景**：上条改动落地后，用户拿真实排障对话验证——发现两次 `resolve_fields` 调用全部返回
`null`：一次传了 `"cqkd_invoiceentry.cqkd_invoiceid"`（分录限定，不带单据），一次照搬 `trace`
的三段式坐标写法传了 `"cqkd_invoic_apply.cqkd_invoiceentry.cqkd_invoiceid"`（单据.分录.字段）。
根因：`_split_qualified` 只认 `"单据.字段"` 两段式，且限定符必须命中 `form` 表——`cqkd_invoiceentry`
是分录(entity)key 不是单据 key，两次调用都不满足这个前提，落到"当裸 key 查、查无此字面 key"
分支返回 `null`，模型误判"这个字段元数据里没登记"，其实是工具的限定符语法比 `trace` 窄，模型
理所当然地照搬了 `trace` 的坐标写法却踩空。

**实现**：`cosmic_kb/report/resolve_fields.py::_split_qualified` 改写为按段数解析（与
`field_trace.parse_locator` 同一套坐标惯例）：两段式前段先试 `form` 表、不中再试 `entity` 表
（按分录/子分录限定，两处都不命中才当裸 key 查）；三段及以上首段=单据（须命中 `form` 表）、
倒数第二段=分录/子分录（不强制校验，交给候选过滤+`mismatched_form`诚实兜底，中间段仅供阅读，
与 `parse_locator` 的"中段=父分录"同一取舍）。`resolve_fields()` 主循环按 `(form_key, entry_key)`
两个维度过滤候选；过滤后为空但全局候选非空时，`mismatched_form[key]` 视限定符类型给
`given_form`/`available_forms`、`given_entry`/`available_entities`（可同时出现），不悄悄回退

## `method_calls` 紧凑投影防 MCP 32KB 截断 ⬜（待 codex 跑测试确认）

**背景**：用户真实排障复现——`ask` 问「`InvoiceWriteBackTask.execute` 是做什么的？在哪个类
里？」经 MCP 返回被 host 从中段截断（实测 62415 字节，elided 29647）。排查发现这不是回归，是
2026-06-27 那次 `trace`/`bill` 防截断修复（`efca38e`/`3f6d35a`，加 cap + 字节 governor + 游标
分页）当年只覆盖了 `field_who_changed`/`bill_drilldown` 两个意图，`report/method_calls.py`
从未补上同款投影——`tool_method_calls`（MCP 直调）和 `tool_ask` 里 `method_calls` 分支都是把
`method_calls()` 的富 dict（重载方法多、或方法体项目内调用多时无界）原样透传，第一次撞上大方法
就破了 host 32KB 硬上限。

**实现**：`cosmic_kb/report/method_calls.py` 新增 `method_calls_compact()`，与
`field_trace.trace_compact`/`bill_view.bill_compact` 同一套治法：
1. `_wire_len`：按 host 真实序列化口径（`json.dumps(indent=2, ensure_ascii=True)`）度量字节——
   与 `field_trace._wire_len` 同口径，避免只量无缩进版本导致低估。
2. `_cap_method`/`_build_compact`：每个方法节点的 `calls`、`fields.writes`、`fields.reads`
   各自按 cap 裁剪，真实总数留 `calls_total`/`writes_total`/`reads_total`，截断量留
   `calls_capped`/`writes_capped`/`reads_capped`；重载方法数本身也有 `methods_total`/
   `methods_capped` 兜底（多重载罕见，但不假设不会发生）。
3. `_MC_LADDER`：`(方法数, 单方法 calls, 单方法 fields)` 七档由宽到窄，governor 逐档收紧重建
   直至序列化 ≤ 预算，保证永不被截断（硬底档也能塌到单方法 2 calls + 1 field）。
4. `_mc_page`/`_mc_parse_cursor`：被 cap 的段带 `next_cursor`（`"calls:<idx>@offset"`/
   `"writes:<idx>@offset"`/`"reads:<idx>@offset"`/`"methods@offset"`），消费方逐页可**取回
   全部被截条目**，不只报计数（红线 #4）。
5. `mcp/server.py`——`tool_method_calls` 加 `cursor` 形参、改调 `method_calls_compact`；
   `tool_ask` 的 `method_calls` 意图分支同 `field_who_changed`/`bill_drilldown` 一样，在
   `status=="ok"` 时把 `evidence` 换成 `method_calls_compact(conn, rq.class_fqn, rq.method_name)`。
   两处 docstring 同步补充 `next_cursor` 翻页纪律。
6. CLI `cosmic_kb calls`（`cli/main.py`）与 resolver/builder 的富 dict 路径**不受影响**——
   只有 MCP 出口换紧凑投影，模块顶部注释同步标注这层新分工（同 `field_trace` 的既有约定）。

**测试**：Claude 已写用例（未自跑，按分工交 codex 执行）——新增 `tests/test_method_calls_compact.py`
（8 自调用 + 4 处字段写入 + 2 处字段读取的真实可解析 Java fixture，覆盖：小数据默认 budget 不
裁剪、未命中/歧义原样透传、极小 budget 逼出 calls/fields 双重裁剪、`calls`/`writes` 游标分页
逐页取回全部被截条目不丢数、未知 section 报错而非静默空结果）；改写 `tests/test_method_calls.py`
的 `test_mcp_tool_method_calls`——MCP 出口不再与富 dict 逐字节相等（同 trace/bill 的既有约定），
改断言语义一致（`calls`/`fields.writes` 的 key 集合相同）且小结果 `calls_capped == 0`。

**验收结论**：⬜ 待 codex 执行 `pytest -q` 确认零回归（新增 `test_method_calls_compact.py` +
改写 `test_mcp_tool_method_calls`）。改 MCP 源码后需重连/重启 MCP 才生效（本次未改 schema，无需
`cosmic_kb build`）。
