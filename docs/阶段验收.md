## 阶段 10 增补 · 方法出向调用导航 method_calls ✅

**背景**：原 `read_method`（方法级深读：复述方法源码 + 语义摘要）在交付人工测试时被否——段二
形态拍板为**大模型直接读本机源码 + 挂苍穹 skill**，复述源码 / 列平台·`equals`·常量调用 / 做自然
语言解释这些大模型自己做得更好，静态层在这块零增量甚至刷屏噪声（如 `super.propertyChanged()`、
`e.getProperty()`、常量类 `equals` 全被列成调用）。确定性扫描层对一个会读源码的大模型唯一不可
替代的，是**野生、多 ISV 前缀、不可编译码上的"跳转到定义"**。故把方法级能力收窄重置为纯调用导航。

**目标**：给定 类全限定名 + 方法名 → 只回该方法调用的**项目内**方法（调用名 + 目标类全限定名
`target_fqn` + 目标源文件 `target_relpath` + 调用行号），供大模型顺调用链逐层读源码下钻。**不回
源码全文**（大模型自读）、**不列平台/外部/落库 sink 调用**（噪声）、**不做字段落库取证**（归
`trace`）；接收者类型解不出 → 不收录（宁缺毋滥，不臆造）。schema 不变 v9。四入口：CLI `calls`
+ MCP 工具 `method_calls` + `ask` 意图 `method_calls` + `trace`/`bill` 跳转提示。

**验收结论**：⏳ 待人工测试。`pytest -q` **180 passed**（`tests/test_method_calls.py` 9：项目内调用
解析+噪声不收 / 跨类下钻 / 源码根自动解析 / 三类未命中候选 / resolver+builder+MCP 同口径）。诚实
边界：tree-sitter 未装或源码读不到 → 给空清单 + 明确提示、不臆造；同名类/重载列候选反问。

```powershell
cosmic_kb calls "<类全限定名>" "<方法名>"   # 该方法调了项目内哪些方法→目标类/源文件/行，供顺调用链读源码下钻
```

## 阶段 10 增补 · 字段名核对 `resolve_fields` ✅

**背景**：段二大模型读 Java 源码时靠**命名惯例猜字段中文名**翻车——`cqkd_zjjnqk` 被猜成"资金缴纳情况"，
真实是"租金缴纳情况"（一字之差），整轮解释没调过任何取证工具核对。根因两件事同时缺位：① 没有"便宜"的
字段名查询工具（现有 `trace`/`bill`/`ask` 都"重"：payload 大、语义是"谁改了它"而非"它叫什么"，模型不愿
为核一个名去调，于是走阻力最小的路：猜）；② MCP `INSTRUCTIONS` 没有硬反射（只说"绝不臆造"，但模型主观
把"按命名惯例推断"当识别而非臆造）。用户 2026-06-25 拍板：补轻量工具 + 加纪律，把"按命名惯例臆断"挤掉。

**目标**：① 新增 O(1) 取证 `resolve_fields(keys)`——标识 → 真实元数据中文名+实体坐标，**同时覆盖
`field`+`entity` 两表**（分录容器 key 也能解析，避免读到 `getDynamicObjectCollection("cqkd_zdfl")` 解不出），

## 信任优先 · trace 防截断补强（governor 按字节 + 坐标组可裁剪，堵全量调的洞）✅

`@ContractBillAnnotation(value="cqkd_qs", name="未缴金额", type=…)`，再用反射方法
`convertToBillDynamicObject()`（循环 `getDeclaredFields()` 做 `bill.set(annotation.value(), field.get(this))`）
把整批字段写进 DynamicObject。字段 key 藏在**注解里的静态字面量**、不出现在 `.set()` 实参位置，旗舰
`trace` 的字面量扫描完全看不见 → `trace cqkd_qs` 一条 access 都不产生、真实写入隐形（false negative，踩
红线 #4）。用户实测命中：`cqkd_qs 未缴金额` 被 `BillDo.convertToBillDynamicObject` 反射写入却查不到。
**与已搁置的「动态写入」本质不同**：动态写入的 key 由运行时/配置决定、静态钉不出唯一字段；注解里的
`value()` 是静态字面量且能用 KB 字段表反验证，属确定性可解的盲点——该补。

**目标**：识别这类映射，让 `trace`/`bill`/`ask` 查这些 key 时不再隐形（看到映射类 + 反射写方法 +
调用它的插件入口 + 已核对中文名）；A 档先消灭「完全隐形」，**精确单据坐标/落库三态无法静态确定的留
unknown**（B 档后续）。硬约束：**绝不另起无界数组/新 section**，合成写入作为普通 `field_access` 行流经
trace 既有「按类合并 + cap + 字节 governor」压缩管线，膨胀有界 by construction。

**实现**：① `cosmic_kb/java/ast_index.py` 新增 `FieldAnnotation` + `iter_field_annotations`（抽字段注解及
成员字面值）；② 新模块 `cosmic_kb/java/annotation_map.py`——**用法自动发现注解类型**（注解 `value` 命中
KB 字段 key 集才采纳，KB 反验证、不硬编码注解名、不臆造）+ 反射 bulk-write 方法分类（getDeclaredFields/
getAnnotation + 对非 `field` 接收者 `.set()`）+ `synth_accesses(fqn,method)` 产合成 `FieldAccess`
（via=`annotation-map`、key_resolution=`annotation-map`、entity=None、conf 低档）；③ `cosmic_kb/java/analyze.py`
建索引挂 `pg.annmap`，`_walk_event`/`_analyze_standalone` 走到映射类 bulk-write 方法时注入合成写入
（plugin_fqn=调用它的插件入口、access_class=映射类），`_emit_event` 对 annotation-map 写入一律
`persists=unknown`（落库取决于调用方是否保存转换产物，不蹭链路 sink 结论）；合成行经现有
`_backfill_form_key` 自动按字段归属诚实回填 form_key（唯一→metadata_unique，歧义无收敛→None）。
④ `cosmic_kb/report/field_trace.py` 仅新增 `summary["annotation_writers"]` 标量 + CLI 文本一行提示，
**不新增 section/数组、不改 cap/ladder/budget**。零 schema 改动（仍 v10）。

**验收结论**：✅ 通过（Claude 开发 + 写用例 + 自测；Codex 执行全量测试 + 提交）。真实源码直跑
`annotation_map.build_index`：发现 2 个写映射类 **BillDo**（34 映射，写方法 convertToBillDynamicObject/
updateDoToDynamicObject）、**BillSubDo**（29 映射）；`cqkd_qs` 由 `BillDo.unpaidAmount`(line 62) 映射、经两反射
写方法合成——隐形问题消除。term DO（CostTermDo/FloatTermDo/RentDetailDo）只在构造器从 DynamicObject 读、
无 bulk-write，**正确不登记**（A 档只覆盖写、读方向在 `new X()` 构造器不入调用图，诚实留白）；`id` 等非

## 信任优先 · 修复变量名作用域冲突：真实写入被错盖 basedata「无需追」 ✅

**背景**：用户 2026-06-29 顺 `basedata-ref` 成因核查真实库，发现 `level=basedata` 的写仅 21 条（读 655 条，
基础资料确以读为主，符合直觉）。逐条读源码核实：**多数是误判**——把真实业务字段写入错盖成
`null_reason=basedata-ref`「正确 None·无需追」，落入 `CORRECT_NONE_REASONS`，trace/coverage/web/MCP 都告诉段二
"无需追"，**静默假阴性**（比 unknown 更糟，直接违背红线 #4）。铁证 `AssetCardCleanQuantityTask.java`（14 条）：
内层 `asset = card.getDynamicObject(基础资料字段)` 先把 `asset` 钉成 basedata；外层
`for (DynamicObject asset : BusinessDataServiceHelper.load(资产卡片实体))` 想重绑表头却被旧绑定挡住，于是该循环里
`asset.set(闲置面积/资产状态…)`（264–290 行）全被错归 basedata。`AssetMergeOpPlugin`/`AssetSplitOpPlugin` 的
`contract.set(总承租面积…)` 同模式。**根因**：`field_access.py` 的 `doc_ctx` 方法级扁平、无块级作用域，同名局部
变量「先到先得」——for-each 绑定 `if row_var not in doc_ctx` 守卫、lambda 绑定 `setdefault` 都拒绝覆盖已有绑定。

**目标**：for-each / lambda 循环变量是**新作用域的新绑定**，当来源集合带真实坐标（header/entry/subentry）时
应**覆盖**同名变量早先被 `getDynamicObject` 误钉的 basedata 旧绑定，让真实写入回到正确层级、不被错盖"无需追"。

**实现**：`field_access.py` 不动点循环两处——① for-each 绑定（原 `not in doc_ctx`）改为「未绑定→绑定，或
旧绑定是 basedata 且本集合非 basedata→覆盖」；② lambda 形参绑定（原 `setdefault`）同口径。仅当目标值真变化才置
`changed=True`，覆盖后不再触发，不动点收敛安全（原硬上限不变）。**只放宽 basedata→有坐标这一种覆盖**，
其余绑定一律不动（最小改动、对症）。

**验收结论**：✅ 通过（Claude 开发 + 写用例 + 自测；Codex 执行全量测试 + 提交）。真实源码 `AssetCardCleanQuantityTask`
复跑：`level=basedata` 写入 **14→0**，264–290 行写入全部回到 `level=header / entity=cqkd_*`，全量重建后 form_key
可定位、不再标"无需追"。有意识取舍：内层那条真·基础资料读（如 `asset.getLong(id)`）会随之归到 header——读是 `na`
危害小，远轻于"藏写入"；护栏用例 `test_c4_genuine_basedata_write_still_basedata` 确认**无同名循环覆盖时真·基础资料
内部写入仍判 basedata**（不过度修正）。新增 `tests/test_field_scan_inline.py` 3 例（C4：for-each 覆盖 / lambda 覆盖 /
真 basedata 不误伤）；相关文件自测 `test_field_scan_inline + null_reason + java_field + field_trace_header` **67 passed**
零回归。需 `cosmic_kb build` 重建 KB 生效；真实库整库重建复算 `level=basedata` 与 null_reason 分布交 Codex。

```powershell
pytest tests/test_field_scan_inline.py -q              # C4 三例 + 既有 C1/C2/C3 回归
cosmic_kb build "<源码根>" "<dym|zip|目录>"            # 重建 KB（修复后 basedata 写入误判清零）
# 真实库复算：SELECT access,COUNT(*) FROM field_access WHERE level='basedata' GROUP BY access
```
