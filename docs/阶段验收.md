## 阶段 2 · 元数据解析 + 整包/多包处理 ✅

**背景**：元数据（globalsdk 导出的 dym/zip）是 KB 的另一半来源。需把三类 dym
（单据/基础资料/动态表单）统一解析，并能批量吃下整包 zip；生产项目一个 zip ≈ 一个业务模块、
常多包并存，故还需多包跨模块汇总。

**目标**：
1. 三类 dym 统一为 `MetaModel`：字段不遗漏、层级正确（表头/分录/子分录）、类型可解析、
   下拉项带出、插件三类（界面/列表/操作）齐收。
2. 继承覆盖型操作的 hex oid → 标准操作，经继承根模板回填。
3. 整包双层 zip 解析，列出全部表单。
4. 多包 / 目录解析 + 跨模块（appKey）汇总。

**验收结论**：✅ 通过（含多包补强）。
- **单样例**：资产卡片 390 字段（表头 306 / 分录 75 / 子分录 9，无悬空）、插件 form4/list4/op17、
  4 个继承操作经模板译出；基础资料、动态表单分支均正确（动态表单字段解出且 DB 列为空）。
- **整包**：单包 315 dym **0 失败**；多包目录一条命令汇总 **2 包共 344 表单 0 失败**，
  按模块 `cqkd_assets×315 + cqkd_bd×29`。
- 测试 **36 passed**。
- 已知留接口（带入后续阶段）：反写插件无样例、占位未产出；动态表单继承根模板未提供、
  覆盖操作记 unknown；`.dymx` 多语言本阶段跳过。

**相关命令**：
```powershell
pip install -e ".[parse,encoding,dev]"
pytest -q                                          # 36 passed
cosmic_kb meta samples\bill\cqkd_assetcard.dym     # 单 dym
```

3. 两套前缀（代码包前缀 / 元数据标识前缀）自动发现统计，仅作报告产物。
4. 孤儿类打轻量角色标签（常量类 / 真孤儿），常量类喂阶段 6/9、不进阶段 4 风险热点。

3. 识别继承苍穹插件基类（**传递闭包**，含经过项目中间基类）的孤儿，打 `role='plugin'`。

## 阶段 5+6（类内）+7 · 字段级排障引擎 🔧（待人工验收）

**背景**：阶段 4 的「项目地图 / 理解报告」是管理者视角的项目普查，对接手开发者无价值。
苍穹真实排障流程的痛点在最后几步——开发要把元数据里注册的一堆插件**全路径逐个复制到
代码里翻**，找对应事件函数、判断它是否真改了某字段、是否落库。用户 2026-06-17 拍板：工具
核心价值是消灭「逐个翻代码」，把产品方向从「项目普查」重定向为「字段级排障导航」。

**目标（旗舰能力）**：输入字段标识 → 列出**所有读/写该字段的插件 + 对应事件函数 + 是否
落库 + 行号 + 源码路径**，点一下直达。本轮一次实现 阶段 5（事件/字段）+ 阶段 6 类内
（调用链路径）+ 阶段 7（落库判定）。

**关键决策（已拍板）**：
- 调用链做到**类内 + 跨类回溯**（事件→本类私有方法链 + 可解析的跨类项目方法）；解析不到
  的（kd.bos.* 等平台/外部）标 unresolved，落库据此保守标 unknown。
- 多单据复用 → **入口单据锁定路径**，同一类/helper 被多单据绑定各自成记录、归各自单据。
- 落库判定 = **事件相位 × 操作类型 × 调用链到 sink**：入库类操作(save/submit/audit)的
  事务内事件 setValue → 落库；donothing 需显式 save sink；界面 propertyChanged 仅内存。
- 字段引用 = 字面量 + **全局常量值表**（多套常量类）；`setValue` 按实参个数判表头/分录/
  子分录；DynamicObject 树形赋值靠方法内轻量数据流定层级 + 所属分录。
- **字段定位以实体坐标 (单据·层级·分录) 为准**（不是裸标识）；查询按坐标分组、可按
  form/entry/level 过滤（用户 2026-06-17 验收反馈）。

**首轮验收不通过 → 返工（2026-06-17）**。用户指出三处硬伤，均已修复：
1. *字段定位不可靠*：同名字段可能跨单据/层级/分录，按裸标识列「所有插件」毫无价值。
   → `field_trace` 改为**按实体坐标 (form, level, entry) 分组**，先列元数据定义坐标作消歧
   菜单，支持 `单据.分录.字段` 点号定位与 `--form/--entry/--level` 过滤。
2. *前端文字超长/挤压、非以实体为单位*：→ Web 重做——定宽表格(table-layout)+长 FQN/路径
   换行截断+悬浮全名，字段结果与单据视图均**以实体为分组块**展示。
3. *部分类的字段读写漏统计*（如 `CollateralService` 写 `cqkd_collateralstatus`）：根因有二
   ——① 只分析「绑定到元数据的插件」，service/工具类整片漏；② `field_access` 只认硬编码
   接收者名，service 方法 `param.set(字段,值)` 的 DynamicObject 入参不识别。→ ① 新增
   `java/project_graph.py` 全项目类索引 + 跨类回溯，插件事件调 service 方法时跟进归因
   （form_key + 落库判定）；未被任何插件调用的项目类**全量补全**（form_key 空、落库 unknown）；
   ② `field_access` 识别 DynamicObject 形参/局部变量为数据包（层级按表头保守推断、降置信）。

**二轮验收不通过 → 再返工（2026-06-17）**。用户指出仍有「代码操作了单据字段但漏检」，并给出
数据包来源判定口径（事件入参=绑定单据/转换目标源单；ORM load=实参实体；树形结构分录/子分录/
基础资料）。根因：数据包来源识别规则太窄 —— ORM `BusinessDataServiceHelper.load("实体",…)` 的
结果数组不认 → 其 for-each 循环变量无上下文 → `bill.set(...)` 整片丢弃；转换插件
`getTargetExtDataEntitySet/getDataEntity/getValue(SOURCE)` 不认；且即便抓到也一律归「插件绑定
单据」而非数据包真实来源实体。已修：
1. `field_access` 大改 —— `_Ctx` 增 `entity`（数据包来源实体），沿集合→行→子集合→子行/for-each
   一路传播；新识别 ORM load/loadSingle/newDynamicObject/query（来源=实参实体，经常量解析+已知
   实体校验）、`getModel().getEntryEntity`、转换 `getDataEntity`(目标单)/`getValue(...SOURCE)`(源单)；
   修正 `getDataEntities` 误判（它**不是** `getDataEntity` 子串）。
2. `analyze` 用**来源实体归因**：field_access 行 `form_key = 数据包来源实体`（判不出=None 归

## 信任优先 · 可信度报告（手段一字段覆盖率 + 手段二粗/高精度对比）🔧（待人工验收）

**背景**：红线 #4「覆盖率/可信度报告是一等功能」。字段级排障引擎答案再准，接手者也要先问

又复杂的高精度扫描，会不会**静默漏掉**字段？」——两问分别由「手段一」「手段二」回答。

**目标**：
1. **手段一·字段覆盖率**（`report/coverage.py`）：以**元数据业务字段为分母**（entity/dynamic/
   basedata_prop）、代码观测到被读/写为分子算覆盖率；配四维质量分解（字段标识解析、来源单据
   定位、落库判定、命中元数据），让「覆盖率」这个数字能被正确解读（未覆盖 ≠ 漏扫）。
2. **手段二·粗精度 vs 高精度对比**（`report/scan_compare.py`）：**高精度** = `field_access`
   （AST + 跨类 + 数据流 + 落库，带坐标）；**粗精度** = 纯正则把业务字段标识当**源码字符串
   字面量**直接搜出来（`coarse_field_hit` 表，建库期算，不解析坐标/落库/调用链——召回底线）。
   两侧锚定同一业务字段全集对比：**两者都见**（互证）/**仅粗扫见 = 疑似盲点**（强信号 rw-idiom
   =字面量作 get/set 首参；弱信号 literal=多为常量定义/注释）/**仅高精度见 = 精度增量**（常量
   解析触达，纯 grep 抓不到）。诚实声明「盲点是候选非确诊」，verdict 以强信号盲点定红绿灯。

**验收结论**：🔧 待人工验收。真实项目 `asset_management_sys` × `samples\appzip`+`samples\trans`：
