## 阶段 10 增补 · 方法出向调用导航 method_calls ✅

**背景**：原 `read_method`（方法级深读：复述方法源码 + 语义摘要）在交付人工测试时被否——段二
形态拍板为**大模型直接读本机源码 + 挂苍穹 skill**，复述源码 / 列平台·`equals`·常量调用 / 做自然
语言解释这些大模型自己做得更好，静态层在这块零增量甚至刷屏噪声（如 `super.propertyChanged()`、
`e.getProperty()`、常量类 `equals` 全被列成调用）。确定性扫描层对一个会读源码的大模型唯一不可
替代的，是**野生、多 ISV 前缀、不可编译码上的"跳转到定义"**。故把方法级能力收窄重置为纯调用导航。

**目标**：给定 类全限定名 + 方法名 → 只回该方法调用的**项目内**方法（调用名 + 目标类全限定名
`target_fqn` + 目标源文件 `target_relpath` + 调用行号），供大模型顺调用链逐层读源码下钻。**不回
源码全文**（大模型自读）、**不列平台/外部/落库 sink 调用**（噪声）、**不做字段落库取证**（归
`trace`）；接收者类型解不出 → 不收录（宁缺毋滥，不臆造）。schema 不变 v9。四入口：CLI `calls`
+ MCP 工具 `method_calls` + `ask` 意图 `method_calls` + `trace`/`bill` 跳转提示。

**验收结论**：⏳ 待人工测试。`pytest -q` **180 passed**（`tests/test_method_calls.py` 9：项目内调用
解析+噪声不收 / 跨类下钻 / 源码根自动解析 / 三类未命中候选 / resolver+builder+MCP 同口径）。诚实
边界：tree-sitter 未装或源码读不到 → 给空清单 + 明确提示、不臆造；同名类/重载列候选反问。

```powershell
cosmic_kb calls "<类全限定名>" "<方法名>"   # 该方法调了项目内哪些方法→目标类/源文件/行，供顺调用链读源码下钻
```

## 阶段 10 增补 · 字段名核对 `resolve_fields` ✅

**背景**：段二大模型读 Java 源码时靠**命名惯例猜字段中文名**翻车——`cqkd_zjjnqk` 被猜成"资金缴纳情况"，
真实是"租金缴纳情况"（一字之差），整轮解释没调过任何取证工具核对。根因两件事同时缺位：① 没有"便宜"的
字段名查询工具（现有 `trace`/`bill`/`ask` 都"重"：payload 大、语义是"谁改了它"而非"它叫什么"，模型不愿
为核一个名去调，于是走阻力最小的路：猜）；② MCP `INSTRUCTIONS` 没有硬反射（只说"绝不臆造"，但模型主观
把"按命名惯例推断"当识别而非臆造）。用户 2026-06-25 拍板：补轻量工具 + 加纪律，把"按命名惯例臆断"挤掉。

**目标**：① 新增 O(1) 取证 `resolve_fields(keys)`——标识 → 真实元数据中文名+实体坐标，**同时覆盖
`field`+`entity` 两表**（分录容器 key 也能解析，避免读到 `getDynamicObjectCollection("cqkd_zdfl")` 解不出），

## 阶段 10 增补 · 模式 B 语义增强（把核对结果焊进工具返回值）✅

**背景**：上一节加了 `resolve_fields` + `INSTRUCTIONS`「字段名纪律」后，2026-06-25 一次真实段二调用显示
模型**仍绕过** `resolve_fields`/`cosmic_semantics`：凭命名惯例猜字段名、凭训练知识臆断 `propertyChanged`
的触发时机与是否入库，整轮没核对。关键发现——模型**知道**规则（能准确说出该查 `plugin-form`、该
`resolve_fields`），但还是没执行：`INSTRUCTIONS` 是一次性系统提示常量，离决策时刻远，压不过自信先验；且
模型"读源码→直接答"的默认回路**一个取证工具都不碰**，软约束施加的位置不对。结论：不该再去"强制模型
调用"（对自主 agent 几乎不可达），应换目标。用户拍板：段二是"用户直接开 agent 聊、且不一定是 Claude
Code"，故宿主钩子（§2.3）不可移植——**通用 host 上唯一所有客户端都必读的硬信息，是我们自己工具的返回值**。

**目标**：模式 B——不强制调用，把"已核对的事实"**焊进模型本来就要走的导航工具返回里**，让它必定读到、
想按命名惯例改写都没机会：① 字段旁内联 `field_name` 已核对中文名（`trace` 顶层 + `bill` 的 `field_touch`
+ `ask` 插件/操作证据；覆盖 field+entity 两表；钉不出留 `None` 不臆造）；② 事件方法旁内联 `semantics_topic`
（`propertyChanged`→`plugin-form`、`beforeDoOperation`→`plugin-operation`…），提示"判触发时机/是否入库前
先 `cosmic_semantics(topic)`，勿凭训练知识臆断"；③ 新增中立复用模块 `semantic/hints.py`（事件→主题映射 +
字段名索引，report/context/method_calls 复用、不反向依赖 mcp）；④ `INSTRUCTIONS` 加「返回值已带证据，
直接采用」。host-agnostic，零 schema 改动（v9）。

**验收结论**：✅ 通过。`pytest -q` **211 passed**（新增 `tests/test_hints.py` 10：事件→主题映射 / 回落
plugin_type / 提示串 / 字段名索引覆盖 field+entity / 同 key 多名歧义留 None 不替选 / trace 顶层名+写入行
topic / bill `field_touch` 名+事件 topic / ask 插件解释名+topic / ask 操作解释名 / method_calls 事件方法
带 topic）。**已知天花板（诚实记录）**：若模型既不调任何导航工具、又用宿主原生 reader 纯读源码后直接答，
这条全绕过的路 host-agnostic 手段一个都拦不住——那一截缝留待**模式 A**（自带自动标注的 `read_source`
工具，用野生编码正确解码等硬价值把原生读拉回来）后续评估。

```powershell
cosmic_kb trace "<单据.字段>"     # 写入行已带 ⚑ 事件语义路由（cosmic_semantics('plugin-...')）
cosmic_kb bill "<单据>"           # 字段触达已带「中文名」+ 事件 semantics_topic
cosmic_kb ask "<某类/某操作>"     # 证据包字段带 field_name、事件带 semantics_topic
```

## 阶段 10 增补 · 堵两条"读源码猜字段名"路径：method_calls 带字段 + read_source（模式 A）✅

**背景**：模式 B 落地后，2026-06-25 第二次真实样本仍翻车——模型走的路径是「`method_calls`（方法导航，
**不带字段名**）→ 宿主原生 reader 直接读源码 → 字段 key（`KEY_TZRQ`/`cqkd_zdfl`）只在源码正文里出现、
从没进过任何工具返回值 → 按拼音猜」，并**主动选择省事跳过 `resolve_fields`**。这正是模式 B 的覆盖天花板：
内联只在模型流经带名工具（trace/bill/ask）时生效，而这条路一个都没碰。用户拍板「两个都做，先 method_calls
再模式 A」。

**目标**：① **method_calls 延伸模式 B**——`field_access` 无"访问方法"列，但有 `source_relpath`+`line`，故按
方法 AST 行范围 `start_line..end_line` 圈出该方法体读写的字段，返回 `fields`（已核对中文名 + 是否落库 +
事件语义路由），导航到方法就拿真名；钉不出具体字段的动态写入只计数（交 dynwrites/读源码定性）。② **新增
`read_source`（模式 A）**——让模型读源码走我们的工具而非宿主原生 reader：按建库同款编码探测**正确解码**野生
码（GBK/GB2312/UTF-8±BOM，原生 reader 易乱码=拉它过来的硬理由）、行号对齐 KB，并**自动标注** `field_names`
（扫文本里出现的已知字段/容器 key→真实中文名；常量 `KEY_X="cqkd_x"` 的字面值就在源码正文，无需常量表，
直接复用 `resolve_fields`）；支持 `--lines A-B` 区间、防 `../` 越界。新增 `report/source_read.py`（源码读取
公共件，`method_calls` 委托复用）+ `report/read_source.py`。`INSTRUCTIONS` 加「读源码优先用 read_source」。

**验收结论**：✅ 通过。`pytest -q` **220 passed**（method_calls +2：方法返回自身读写字段+已核对名+语义路由、
按行范围归属正确不串跨类方法；`tests/test_read_source.py` +7：标注已知字段名(含常量字面值)、不臆造非字段
token、行区间切片、防路径越界、缺文件、render、CLI/MCP 同口径；`test_mcp.py` TOOLS 加 `read_source`）。
**天花板仍在（诚实记录）**：`read_source` 是"引导非强制"——若模型坚持用宿主原生 reader 读源码、连
`read_source` 都不调，host-agnostic 仍拦不住；能做的是把"正确解码野生编码 + 自带字段真名 + 行号对齐 KB"
做成它用我们 reader 的硬理由。

```powershell

cosmic_kb source "cqspb/am/Foo.java"        # 读源码(正确解码)+顶部「已核对字段名」标注块；--lines 30-60 读区间
```

## 阶段 10 增补 · read_source 字段名三档置信消歧 ✅

**背景**：真实样本发现 `read_source` 把同一 `<isv>_xxx` key 在**多张单据**的同名候选**全平铺**返回（中文名
可能不同），段二模型只能脑补归属——这是一种严重误导。诊断：`read_source` 明明知道"在读哪个文件"，却没用
本文件的解析上下文收敛候选，直接把 `resolve_fields` 的裸全候选倒出去。

**目标**：用本文件的确定性解析结果收敛同名候选，归属能定就定、定不了就**显式标歧义**（红线 #4），绝不平铺成
"像已确定"。

**实现**：仅改 `report/read_source.py`（其余返字段名的工具 trace/bill/ask/method_calls 早已用
`FieldNames.get(key, form_key)` 按数据包来源单据取名，本无此病）。依据是本文件 `field_access.form_key`——
它存的是数据包**实际来源实体**（经 ORM load / 事件入参 / 跨实体传播解析，`analyze.py` 中 `form_key=acc.entity`；
解不出为 NULL）。`field_names` 按 key 归三档：✅`unique`（元数据唯一）/ ✅`resolved`（本文件 field_access 已把它
解析到具体单据——含 `loadSingle` 跨实体取到别张单据字段的情形——收敛 + 附依据行号 + 标明其余单据）/
⚠️`ambiguous`（多单据同名又没解析到实体 → 列候选 + 指出消歧方向「看接收变量来源 dataEntity/loadSingle/
getAllSonList」，`names` 留空、绝不替选、不默认当前单据）。返回形状变为 `{key:{tier,names,coordinates,note}|None}`；
MCP `INSTRUCTIONS`/工具说明补「ambiguous 别默认当前单据」。零 schema 改动（v9）。

**验收结论**：✅ 通过。`pytest -q` **222 passed**（test_read_source 适配新形状 + 新增 resolved/ambiguous 两档
用例：`cqkd_fee` 经本文件 loadSingle 收敛到合同单「手续费」不平铺资产卡「服务费」、附依据行号；`cqkd_rate`
无 field_access 解析 → 标歧义、列两单候选、给消歧方向）。

```powershell
cosmic_kb source "cqspb/am/Foo.java"        # field_names 三档：unique/resolved 照抄 names；⚠️ambiguous 顺调用链消歧
```
