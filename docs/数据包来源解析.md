# 数据包来源解析 —— 字段修改来源追踪能力清单

> 本文盘点**字段级排障引擎**（旗舰能力：输入字段标识 → 列出谁读/写它、事件函数、是否落库、
> 行号、源码路径）当前**能追踪到的字段修改写法**与**会遗漏/判不准的写法**，作为返工与回归的
> 对照基准。实现见 `cosmic_kb/java/field_access.py`（识别）、`analyze.py`（跨类编排与来源
> 传播）、`constants.py`（字段 key 常量解析）。
>
> 核心哲学（沿用项目红线）：**处处置信度 + 证据行号 + 判不出标 unknown / 未定位，绝不臆造**。
> 来源判不出的字段读写不会被丢弃——它进「未定位」/「可能命中（存疑）」桶，留行号证据。

---

## 1. 什么是"数据包来源"，为什么要追

苍穹里同一个字段标识（如 `cqkd_amount`）可能出现在多张单据 / 多个分录里。排障要回答的不是
"哪些插件出现了这个字符串"，而是 **"这个字段，在『哪张单据·哪个层级·哪个分录』上，被谁改了"**。
所以每一处字段读写都要带四元坐标：

| 维度 | 字段 | 取值 | 判不出时 |
|------|------|------|----------|
| 来源实体 | `form_key` / `entity` | 单据/实体标识 | `None`（未定位） |
| 层级 | `level` | `header` / `entry` / `subentry` / `basedata` / `unknown` | `unknown` |
| 所属分录 | `entry_key` | 分录/子分录 key | `None` |
| 字段 | `field_key` | 字段标识 | `None`（dynamic/unknown） |

"数据包来源"即上表第一行的 **来源实体**：这处写入操作的 DynamicObject / 模型，到底承载的是哪张单据。

## 2. 能识别的两套写入/读取习语

字段读写只通过这两套 API 进入引擎（`field_access.py` 第 38-46 行）：

### 习语 A · 界面/模型 API `getModel().setValue(...)`
- **写**：`setValue` / `setItemValueByID` / `setItemValueByNumber`
- **读**：`getValue` / `getItemValueByID` / `getItemValueByNumber`
- **接收者**必须是 `xxx.getModel()` 或已识别的模型局部变量（`_is_model_receiver`）。
- **层级靠实参个数判**（操作的就是插件当前绑定单据）：
  - 写：`setValue(key,val)`→表头；`(key,val,row)`→分录；`(key,val,row,sub)`→子分录。
  - 读：`getValue(key)`→表头；`(key,row)`→分录；`(key,row,sub)`→子分录。
- **来源实体** = `default_entity`（插件绑定单据 / 转换目标单），跨类时取传播来的模型形参绑定单据。

### 习语 B · DynamicObject 数据包 `obj.set(...)` / `obj.get(...)`
- **写**：`set`
- **读**：`get` / `getString` / `getInt` / `getLong` / `getBigDecimal` / `getDecimal` / `getDate` /
  `getBoolean` / `getDynamicObject`
- 接收者的**层级 + 分录 + 来源实体**由「方法内轻量数据流」追出来（见第 3、4 节）。

> 字段 key（首个实参）一律经 `constants` 解析：字面量、`类.常量`、裸常量名、跨方法传来的
> String 形参都能解析回字段标识（第 5 节）。

## 3. 数据包来源识别规则（这是本文的核心）

`_build_contexts`（`field_access.py` 第 273-463 行）用方法内轻量数据流，把每个局部变量 / 形参 /
循环变量绑定到一个数据包来源。规则按以下顺序识别：

| # | 来源写法（init 表达式 / 上下文） | 判定结果 | 置信 |
|---|----------------------------------|----------|------|
| 1 | 事件入参 `e.getDataEntity()` / `getDataEntities()[i]` / `getBizDataEntity()` / `extendedDataEntity.getDataEntity()` / `this` | 表头包，来源=**绑定单据**(`default_entity`) | 高 |
| 2 | `e.getDataEntities()`（数组、无下标） | 表头**集合**，来源=绑定单据，供 for-each/lambda 取行 | 高 |
| 3 | ORM 集合 `BusinessDataServiceHelper.load/loadFromCache(...)`、`QueryServiceHelper.query(...)` | 表头行**集合**，来源=**实参里的实体标识** | 高/中 |
| 4 | ORM 单包 `loadSingle/loadSingleFromCache/newDynamicObject/queryOne(...)` | 单条表头包，来源=实参实体 | 高/中 |
| 5 | 转换插件源单 `getValue(CONVERT_SOURCE)` / `getValue(...SOURCE...)` | 表头集合，来源=**源单**(`convert_source`) | 高 |
| 6 | 分录集合 `x.getDynamicObjectCollection(k)` / `getModel().getEntryEntity(k)` | entry 集合（base 已是分录行 → subentry），来源继承 base，分录 key=解析 k | 高/中 |
| 7 | 基础资料包 `x.getDynamicObject(字段)` | `basedata`，归该基础资料实体 | 中 |
| 8 | 集合取行 `coll.get(i)` / `coll.iterator()` / `coll.addNew()` | 元素行，继承 coll 的层级/分录/来源 | 高 |
| 9 | 数组/集合下标 `entities[i].set(...)` | 元素行，继承集合坐标 | 高 |
| 10 | 增强 for `for (DynamicObject row : coll)` | row 继承 coll 元素坐标 | 高 |
| 11 | Lambda 行 `coll.forEach(o -> ...)` / `coll.stream()...forEach` / `Arrays.stream(coll).forEach` | o 继承 coll 元素坐标 | 高 |
| 12 | 流式派生集合 `Arrays.stream(X)...collect` / `X.stream()...collect/toList/toSet` | 继承 X 的元素坐标（**先于取分录判定**，避免误吞 lambda 体内的 getCollection） | 高 |
| 13 | 根包硬编码名 `bill`/`dataentity`/`billobj`/`info`/`obj`/`dynamicobject`/`data` 调 `.set()` | 表头，来源=绑定单据 | 中 |

**ORM 实体实参解析**（`_resolve_entity_arg`，第 228-243 行）：取字符串字面量 + 常量风格标识符
做候选，**优先落在元数据已知实体集里的**；都不在则取首个并标注"可能跨模块/外部实体"；全是
动态表达式则来源=`None`，标"无法静态解析来源单据"。

## 4. 层级与分录 key 判定

- 层级值域：`header` / `entry` / `subentry` / `basedata` / `unknown`。
- **entry vs subentry**：若取集合的 base 变量本身已是 entry/subentry 行，则其再取的集合判为
  `subentry`，否则 `entry`（第 378-382 行）。
- **分录 key** 经 `_resolve_key_token`（第 246-261 行）：字符串字面量 / 跨方法传来的 String 形参 /
  常量引用都能解析回 key；解不出则 key=`None`，**层级仍可信**（API 名已证明是分录集合），只是
  分录 key 未定位 → 降置信、进存疑桶。

## 5. 字段 key 解析（常量表）

`constants.py` 先扫全工程所有 `static final String`（含接口常量），建「常量名 / 类.常量名 →
字面值」表：

| 解析方式 | kind | 置信 |
|----------|------|------|
| 字面量直解 `setValue("cqkd_amount", …)` | literal | 1.0 |
| `类.常量` 精确命中 `Const.AMOUNT` | constant | 0.95 |
| 裸常量名全工程唯一命中 `KEY_AMOUNT` | constant | 0.85 |
| 同名常量多个不同字面值 | ambiguous | 0.3（标 unknown 留证据） |
| 查不到 | unknown | 0.0 |
| 实参是表达式/拼接 | dynamic | 0.3（key=None） |

## 6. 跨方法 / 跨类的来源传播

`analyze.py` 从事件方法 BFS 跨类回溯（`_walk_event`），按「**实参 ↔ 形参位置**」把来源信息
传进被调方法（`_callee_prop`，第 257-289 行）：

- **DO / 集合形参** → 传完整坐标 `(level, entry_key, entity, is_collection)`；
- **模型/视图形参**（`IDataModel`/`IBillView`/… 见 `_MODEL_TYPES`）→ 传绑定单据，使 service 里的
  `model.getDataEntity()` 也能定位来源单；
- **String 形参** → 传字面值（调用方传来的分录/字段 key 常量）；
- **方法返回值** `bill = this.helper(...)` → `_RetResolver` 预解析 helper 的「返回数据包坐标」，
  把 bill 绑到该坐标（helper 在模型上造行后 `return rows.get(i)` 的常见写法）。

只传"有信息"的项；裸 `(header,None,None)` 不传，让被调方按默认推断。

---

## 7. 落库判定（来源追踪的下游，简述）

写入记录再叠加落库结论（`persistence.verdict`）：入库类操作（save/submit/audit）事务内事件的
setValue → 落库；donothing 需显式 save sink；界面 propertyChanged 仅内存；跨类解析不到的外部
调用 → `unresolved` → unknown。孤立 service 类写入按本方法是否含 sink 判 yes/unknown。

## 8. ✅ 当前能稳定追踪的字段修改写法

| 场景 | 示例 | 来源/层级 |
|------|------|-----------|
| 界面插件改表头 | `getModel().setValue("cqkd_amt", v)` | 绑定单据·表头 |
| 界面插件改分录/子分录 | `getModel().setValue(K, v, row)` / `(K,v,row,sub)` | 绑定单据·分录/子分录 |
| 操作插件事件包改字段 | `e.getDataEntities()[0].set(K, v)` | 绑定单据·表头 |
| 事件包取分录改字段 | `bill.getDynamicObjectCollection("entry").get(i).set(K,v)`（经中间变量） | 绑定单据·该分录 |
| for 循环遍历分录改字段 | `for (DynamicObject r : coll) r.set(K, v)` | coll 来源·分录 |
| lambda 遍历改字段 | `coll.forEach(o -> o.set(K, v))` | coll 来源·分录 |
| stream 过滤后改字段 | `coll.stream().filter(..).collect()` 再遍历 set | coll 来源 |
| ORM 加载别的实体改字段 | `BusinessDataServiceHelper.load("cqkd_other",…)` 再 set | **load 实参实体**（不是绑定单据） |
| 转换插件改目标/源单 | `extendedDataEntity.getDataEntity()` / `getValue(SOURCE)` | 目标单 / 源单 |
| service/工具类被插件调用 | 插件把 model/包传进 service，service 里 set | 跨类传播来的来源 |
| helper 造行再回写 | `r = this.newRow(); r.set(K,v)` | helper 返回坐标 |
| 字段 key 是常量 | `set(Const.AMT, v)` / `set(KEY_AMT, v)` | 常量表解析 |
| 调度/WebApi 等未绑定插件 | `AbstractTask.execute` 内 ORM load→改→save | ORM 实参实体 |
| 基础资料包改字段 | `bill.getDynamicObject("cqkd_org").set(K,v)` | basedata 实体 |

## 9. ⚠️ 会遗漏 / 判不准的写法（已知盲区）

下表区分两类：**A. 来源/坐标判不出但记录仍在**（进未定位/存疑桶，留行号），
**B. 整条写入识别不到（真正遗漏，引擎不产出记录）**。返工时应优先收 B 类。

### A. 记录在、但来源或坐标标 unknown / 未定位（诚实降级，非遗漏）

| 写法 | 现状 | 影响 |
|------|------|------|
| 字段 key 为字符串拼接/方法返回 `set("cqkd_"+t, v)` / `set(getKey(), v)` | key=None，kind=dynamic，置信 0.3 | 字段查询匹配不到，但记录可见 |
| 字段 key 用了多值同名常量 | ambiguous，标 unknown | 同上 |
| DynamicObject 局部变量非 ORM/非事件/非传播 | 来源=None，层级 header，标"来源未识别" | 进「未定位」桶 |
| DynamicObject 形参但调用方未知 | 来源/层级未定位 | 进未定位桶 |
| ORM `load(变量, …)` 实体为动态变量 | 来源=None | 进未定位桶 |
| ORM `load("外部实体",…)` 不在元数据 | 来源=该串，标"可能跨模块/外部" | 来源存疑 |
| 分录 key 为外部常量/变量 | 层级可信、分录 key=None | 进存疑桶 |
| 跨类调用解析不到目标（外部 jar / 反射 / 接口多实现） | 标 unresolved → 落库 unknown | 链路断点 |

### B. 整条写入识别不到（真正的遗漏，需补）

| 写法 | 为何漏 | 严重度 |
|------|--------|--------|
| **模型/包存为实例字段** `this.model = getModel();` 后在别的方法 `model.setValue(...)` | `model_vars` 只在本方法内由 `xxx.getModel()` 局部初始化时识别；实例字段接收者 `_is_model_receiver` 判 false | 高（界面插件常见） |
| **DynamicObject 存为实例字段**跨方法使用 | 数据流只在方法内，实例字段不在 doc_ctx/coll_ctx | 高 |
| **for-each 头里内联取集合** `for (DynamicObject r : bill.getDynamicObjectCollection("e"))` | `_foreach_pairs` 取 iter 文本 `split(".")[0]`=`bill`（非集合变量），row 继承不到分录坐标 | 中（写法常见，会归错/漏分录） |
| **lambda 接收者内联取集合** `bill.getDynamicObjectCollection("e").forEach(o->o.set..)` | `_lambda_collection_var` 取 base=`bill`，非集合变量 | 中 |
| **链式一行多级** `bill.getDynamicObjectCollection("e").get(0).set(K,v)`（无中间变量） | `_resolve_do_ctx` 里 base=`bill` 命中 doc_ctx 返回表头，正则又命中 getDynamicObjectCollection——层级/分录可能误判 | 中 |
| **DynamicObject.set 接收者完全无上下文** 且非根包硬编码名 | `_resolve_do_ctx` 返回 None → 整条丢弃 | 中 |
| **反射 / Map 批量赋值 / `BeanUtils` 拷贝 / `setValues`** 等非标准 API 写字段 | 不在 `_MODEL_WRITE` / `_DO_WRITE` 名单 | 中 |
| **调用链深度 > 8 层** | `_MAX_DEPTH=8` 截断，更深的写入不再回溯 | 低/中 |
| **同一被调方法被多入口以不同来源调用** | `_walk_event` 的 `seen` 去重，第二个来源上下文不再传播 | 中（多单据共用 service 时来源可能只记到首次） |
| **递归 / 环形调用里的写入** | `_RetResolver` 遇环保守不解 | 低 |
| **三元/条件表达式接收者** `(flag?a:b).set(K,v)` | base 解析取不到单一变量 | 低 |
| **其他 ORM 取数包入口**（`BillEntityType`、自研 DAO、`ORM.create`、缓存服务等） | 不在 load/query 正则名单 | 中 |

## 10. 改进方向（建议优先级）

1. **实例字段级数据流**（B 类前两项，覆盖面最大）：扫类字段声明 + `initialize()/构造` 里的
   `this.model = getModel()` / `this.bill = ...load(...)`，把字段也纳入 `model_vars`/`doc_ctx`。
2. **for-each / lambda / 链式内联取集合的分解**：在 `_foreach_pairs`、`_lambda_collection_var`、
   `_resolve_do_ctx` 里对 iter/接收者文本做内联 `getDynamicObjectCollection(k)` 识别，而不是只取
   `split(".")[0]`。
3. **同方法多来源去重放宽**：`seen` 改为按 `(fqn, method, 来源签名)` 去重，让多单据共用的
   service 各来源都被归因。
4. **扩充 ORM/写入 API 名单**：补常见自研 DAO、缓存服务、`setValue` 变体到正则与 API 集合。

> 所有改进必须守住红线：**判不出宁可标 unknown / 未定位，绝不臆造来源单据**。每补一类写法都要
> 在 `tests/test_java_field.py` 加样例（含正例 + 判不出的负例），跑 `pytest -q` 防回归。
