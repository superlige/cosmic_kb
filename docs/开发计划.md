# 苍穹 AI 工程知识底座 — 开发计划

> 定位修订：本工具不是"一个分析库"，而是 **"接手陌生苍穹老项目时，在本机上跑的项目理解工具"**。
> 本计划基于该定位重做，配套文档 `项目企划.md`（9 层目标架构）。
> 不含开发时间估算，按阶段交付与验收推进。

---

## 一、使用场景（设计基准）

```
入职 / 接手一个已存在的苍穹老项目
   ↓
在自己的电脑上运行本程序（纯本地）
   ↓
喂两类输入：
   ① 元数据：从 globalsdk 手动导出的 dym / zip（同示例）
   ② Java 源码：公司项目原始目录，多文件、结构复杂、5~6 个业务模块
   ↓
程序做确定性扫描 → 建立"项目知识库 (KB)"
   ↓
借助 AI（Claude / Codex，经 CLI 或 MCP 查 KB）
   ↓
理解项目：这项目干嘛的、有哪些模块、某字段谁改的、某 bug 先查哪、改一处影响哪
```

---

## 二、六条硬约束（重塑整个设计）

| 约束 | 含义 |
|------|------|
| **本地离线** | 公司代码不能外传。扫描器纯本地；AI 只查 KB，不把整库源码上传。 |
| **代码是"野生的"** | 可能不可编译、缺依赖 jar、混合编码（中文项目常见 GBK/GB2312）、风格不一、多 ISV 前缀。解析器**绝不依赖编译或依赖解析**。 |
| **规模大** | 多模块成百上千文件 → 要性能、进度、缓存、增量重扫。 |
| **信任优先** | 解析覆盖率/可信度报告必须是**一等功能**，而非事后补。用户先信任它，才会给更多代码。 |
| **接手者视角** | 第一需求不是查单字段，而是 **"这项目长什么样"** —— 要能先产出项目地图/理解报告。 |
| **两段式解耦** | 「确定性扫描器(建 KB)」与「AI 理解层(查 KB)」必须解耦，**KB 是契约**。 |

## 三、场景必须补上的能力（原企划未覆盖）

1. **源码摄取层 (Ingestion)** —— 把工具指向一坨乱目录：项目清单 `project.toml`、目录递归发现、编码自动探测（GBK/UTF-8/带 BOM）、排除 `target/ build/ .git/ *.jar`、符号链接处理。野生代码能跑起来的前提。
2. **元数据 ↔ 代码桥接层** —— 全项目最关键的桥：把元数据里的 ClassName（如 `cqspb...AssetCardFormPlugin`）解析到源码树里真实 `.java`。处理三态：① 绑定且有源码（正常）；② 绑定但为平台插件 `kd.bos.*`，无源码（标为外部）；③ 源码里大量 service/util/webapi 未被任何元数据绑定（独立 Java 类，也要纳入）。
3. **解析鲁棒性策略** —— 用 tree-sitter 的错误恢复治"不可编译"；不做类型解析、不解析依赖；遇到不认识的符号（`SaveServiceHelper`、`kd.bos.*`）一律当外部已知符号、用内置 SDK 目录解释，而非报错。
4. **解析可信度 / 覆盖率报告** —— 直接回应"得确认能正确解析"的顾虑。输出：扫描 N 文件、成功解析 M、识别 K 个苍穹插件类、X 个 ClassName 在源码中找到/找不到、哪些文件 AST 有错误片段。是"信不信它、给不给更多代码"的依据。
5. **SDK / 平台符号目录** —— 复用现成 `ok-cosmic-docs.db` 和 `references/`，让工具"认识" `kd.bos.*` 这类无源码平台 API，从而解释入库、事务、F7 等语义。
6. **模块识别 & 项目地图** —— 模块划分以**代码包路径前缀**（如 `cqspb.sysint.assets`）聚类为准，**不用元数据 ISV**（ISV 与包前缀经常对不上，见下「ISV 决策」）；辅以 bizapp 分组，自动推断那 5~6 个业务模块，先给一张"项目地图"。孤儿类（无元数据绑定）也按代码包前缀归模块。
7. **接手者「项目理解报告」** —— 一键生成：项目概览、模块清单、实体清单、插件清单、跨模块依赖、风险热点（无人绑定的孤儿类、超大插件、可疑入库点）。接手第一天最想要的就是这个。
8. **两套前缀命名空间（务必分清）** —— 项目里有**两类互不相干的前缀**：① **元数据标识前缀**（`cqkd_assetcard`、`cqkd_qz` —— 实体/字段标识，防字段串实体）；② **代码包路径前缀**（`cqspb.sysint...` —— Java 包，管类归哪个模块）。同一实体的标识前缀（`cqkd_`）与其绑定插件类的包前缀（`cqspb`）对不上是常态，二者**分别建、不可混**。
9. **处处置信度 + 证据行号 + unknown** —— 老项目分析天生不完整，宁可标 unknown 也不臆造。
10. **增量与可重建** —— 源码/元数据变了只重扫差异；KB 任何时候可从零重建（幂等）。
11. **AI 层"按需取证"** —— AI 不一次吞整库，按问题查 KB 取最小证据集（省 token、保隐私）。
12. **不强依赖 GitNexus** —— 它是增强项；没有它，工具在一个文件夹上也要能独立工作。

## 四、修订后架构（两段式）

```
┌─────────────── 段一：本地确定性扫描器（Python） ───────────────┐
│  Ingestion 摄取  →  Metadata 解析  →  Java 静态分析  →  桥接/路径/入库 │
│  (扫目录/编码探测)   (dym/zip→结构)    (tree-sitter)     (ClassName↔file)│
│                          ↓                                            │
│                 Cosmic KB（SQLite 图谱 + FTS5 + JSON 快照）            │
│                          ↓                                            │
│                 覆盖率/可信度报告 + 项目理解报告                       │
└──────────────────────────────────────────────────────────────────────┘
                          ↓ KB 是契约
┌─────────────── 段二：AI 理解层（CLI / MCP） ───────────────┐
│  语义解析(NL→意图) → 查 KB 取证 → Context Builder → 挂苍穹 Skill 语义  │
│                    → AI 输出带证据的解释 / 排查建议                    │
└──────────────────────────────────────────────────────────────────────┘
              (可选) GitNexus 调用链增强叠加在 Context Builder 上
```

## 五、分阶段开发计划

> 重排原则：先打通"指向乱目录 → 证明解析可信 → 出项目地图"这条**接手者最先需要**的链路，再逐层加深。
> 每阶段标注 **【我开发】【你测试 ✅】【你需提供】**。

### 阶段 0 · 脚手架 + 资产复用
- **【我开发】** 建 `cosmic_kb/` 包（`ingest/ metadata/ java/ bridge/ graph/ semantic/ context/ report/ cli/`）；接入现成 `references/` 与 `ok-cosmic-docs.db` 为 `skill_assets/`；pytest + 冒烟命令。
- **【你测试 ✅】** `pip install -e .` 成功；`cosmic_kb --version` 有输出。

### 阶段 1 · 源码摄取 + 解析可信度报告（先攻信任顾虑）✅ 已完成
- **【我开发】**
  - ✅ `ingest/scanner.py`：指向项目根，递归发现 `.java`，编码探测（GBK/UTF-8/BOM，CJK 偏置 gb18030），排除 `target/build/jar`，输出文件清单。
  - ✅ `java/parser.py`：tree-sitter-java 装载 + 错误恢复，逐文件产出"解析状态(ok/partial/skipped/unavailable)"与错误片段行号。
  - ✅ `report/parse_coverage.py`：覆盖率报告 —— 总文件数 / 读取与解析成功率 / 编码分布 / 低置信编码 / 含错误片段文件清单 + 信任结论；文本与 JSON 双输出。
  - ✅ CLI：`cosmic_kb ingest <路径> [--json] [--follow-symlinks]`；tests/test_ingest.py 全绿（18 passed）。
- **【你需提供】** 一批真实 Java 文件（哪怕一个模块）—— 即"得确认能正确解析"。
- **【你测试 ✅】** `cosmic_kb ingest <项目路径>` → 看覆盖率报告：是否几乎全部成功解析？中文编码有无乱码？**这是第一个信任门槛**，过了再决定给全量代码。

### 阶段 2 · 元数据解析 + 整包处理
- **【我开发】** `metadata/dym_parser.py`（实体/字段/分录/子分录/基础资料/操作/按钮/插件绑定/中文名↔标识/层级）；`metadata/package_loader.py`（双层 zip 整包批量解析）；统一 `MetaModel` + JSON 快照。
  - **设计决策（ISV）**：解析 `<Plugin>` 时**完整保留 `<ClassName>` 全限定名**（包路径 + 类名，如 `cqspb.sysint.assets.formplugin.assetcard.AssetCardFormPlugin`），不得只截末段类名 —— 它是阶段 3 桥接到源码的**唯一钥匙**。
- **【你需提供】** 导出的 dym / 整包 zip（已有示例可先用）。
- **【你测试 ✅】** 解析 `cqkd_assetcard.dym` 字段数对得上、绑定插件类名对；整包能列出全部表单无报错；抽查字段层级正确。

### 阶段 3 · 元数据 ↔ 代码桥接（全项目枢纽）
- **【我开发】** `bridge/linker.py`：以**元数据 `<ClassName>` 全限定名为唯一定位键**直接映射到源码 `.java`（`<ClassName>` 一定等于源码包路径）；分类三态（有源码 / 平台 `kd.bos.*` 无源码 / 孤儿未绑定）；产出桥接报告。
  - **设计决策（ISV）**：桥接**不依赖 ISV / 前缀映射去猜源码位置**（ISV 与包前缀常对不上，不可靠）。前缀（无论几个、叫什么）由工具从元数据标识与代码包路径**自动发现并统计**，仅作报告产物，不作定位手段。
- **【你测试 ✅】** 桥接报告里：元数据绑定的插件类命中率多少？平台类是否正确归"外部"？源码里未绑定的 service/util 是否被收录？

### 阶段 4 · 知识图谱存储 + 项目地图 / 理解报告（接手者首个大价值）
- **【我开发】** `graph/schema.sql` + `store.py`（entity/field/plugin/class/method/module/edge + FTS5，幂等重建）；`report/project_map.py` 模块识别（**按代码包路径前缀聚类，不用 ISV**；孤儿类同理）；`report/overview.py` 一键项目理解报告（概览/模块/实体/插件/风险热点）。
- **【你测试 ✅】** `cosmic_kb report overview` → 是否正确分出那 5~6 个业务模块？实体/插件清单是否齐全？风险热点是否合理？**接手第一天最直接的回报。**

### 阶段 5 · Java 行为分析（类型/事件/字段访问）
- **【我开发】** `java/plugin_classifier.py`（类类型）、`event_extractor.py`（propertyChanged/beforeDoOperation/afterExecuteOperationTransaction/validate/beforeF7Select…）、`field_access.py`（getValue/setValue、DynamicObject 读写、行号）。
- **【你测试 ✅】** 对真实插件 `cosmic_kb java scan <File>` → 类型/事件/字段读写/行号准确，行号能定位源码。

### 阶段 6 · DynamicObject 路径追踪（核心难点）
- **【我开发】** `java/path_tracer.py`：变量来源追踪，把 `row.set("cqkd_qz",v)` 还原为 `cqkd_assetcard.cqkd_entryentity.cqkd_qz`；基础资料内部属性区分；结合元数据消歧同名字段；跨方法不可解时标 unknown。
- **【你测试 ✅】** 含分录写入的真实插件：分录字段还原为全路径、不串实体、基础资料属性可分。

### 阶段 7 · 持久化判断
- **【我开发】** `java/persistence.py`：confirmed/likely/unknown + 原因（SaveServiceHelper.save / 封装 save / executeOperate / 事务 e.getDataEntities / 仅改参数）。
- **【你测试 ✅】** 抽查结论合理；构造"只 set 不 save"应判 unknown。

### 阶段 8 · 业务流分析
- **【我开发】** `graph/biz_flow.py`：基础资料引用 / BOTP 下推 / 审核回写 / 跨单据创建，写入图谱边。
- **【你测试 ✅】** 资产相关单据的"引用/下推/回写"至少一类被正确识别。

### 阶段 9 · 语义解析 + Context Builder + Skill 集成（端到端问答）
- **【我开发】** `semantic/`（中文名↔标识词典、NL→意图、RapidFuzz 候选、低置信反问）；`context/builder.py`（组装实体/写入点/事件时机/入库/证据/风险/排查建议）；挂 `skill_assets`。
- **【你测试 ✅】** `cosmic_kb ask "资产卡片抵押状态是谁改的？"` → 结构化答案带类/方法/事件/行号/入库判断，无臆造字段；模糊词给候选。

### 阶段 10 · MCP 封装
- **【我开发】** 参考现成 `mcp_bill_metadata_server.py`，把字段影响/插件解释/操作影响/入库/业务流/项目地图封装为 MCP 工具，供 Claude 与 Codex 直接调用。
- **【你测试 ✅】** MCP 客户端注册后调用结果与 CLI 一致。

### 阶段 11 · 增量重扫 + GitNexus 增强（可选收尾）
- **【我开发】** 文件指纹增量重扫；接 GitNexus 补调用链/影响范围进 Context Builder。
- **【你测试 ✅】** 改几个文件只重扫差异；写入点能附上游调用链。

## 七、分工与纪律

- **Claude Code**：摄取 / 桥接 / 图谱 / 路径追踪 / 业务流 / Context —— 跨模块、需架构判断的硬骨头。
- **Codex**：编码探测器、事件识别规则、正则兜底、单测样例、references 整理 —— 边界清晰、可独立验证的任务。
- **纪律**：每阶段产物 + 一份 `docs/阶段N-验收.md`（含测试命令），两个工具都以它为准。

## 八、待拍板的点（不阻塞，想到再回）

1. **第一个信任门槛（阶段 1）**：能先给一个业务模块的真实 Java 目录做解析验证吗？带敏感逻辑也只在本机跑、只出覆盖率统计、不外传内容。
2. **起步路线**：建议先走 **M1（阶段 1–4）** —— 最快在"接手第一天"看到项目地图，也最快验证解析可信度。认可这个优先级，还是想先纵向打通"资产卡片某字段谁改的"一条线？
3. ~~**多 ISV 前缀**：项目除 `cqkd_ / cqspb` 外大概还有哪些前缀？~~ ✅ **已拍板（2026-06-14）**：
   - **无需预先提供前缀清单。** ISV 是苍穹的开发者标识，理论上应与代码包前缀一致，但实际项目包名常随意取、**对不上**，故 ISV 不可作定位依据。
   - 真正可靠的桥是元数据 `<ClassName>` **全限定名**（一定等于源码包路径），桥接只认它。
   - 区分**两套前缀**：元数据标识前缀（`cqkd_`，管字段/实体）与代码包前缀（`cqspb`，管模块归属），分别建、不混。
   - 前缀由工具**自动发现**，仅作报告产物。详见三-6 / 三-8 与阶段 2/3/4 的「设计决策（ISV）」。
