# 苍穹 AI 工程知识底座 — 开发计划

> 定位修订：本工具不是"一个分析库"，而是 **"接手陌生苍穹老项目时，在本机上跑的项目理解工具"**。
> 本计划基于该定位重做，配套文档 `项目企划.md`（9 层目标架构）。
> 不含开发时间估算，按阶段交付与验收推进。

---

## 一、使用场景（设计基准）

```
入职 / 接手一个已存在的苍穹老项目
   ↓
在自己的电脑上运行本程序（纯本地）
   ↓
喂两类输入：
   ① 元数据：从 globalsdk 手动导出的 dym / zip（同示例）
   ② Java 源码：公司项目原始目录，多文件、结构复杂、5~6 个业务模块
   ↓
程序做确定性扫描 → 建立"项目知识库 (KB)"
   ↓
借助 AI（Claude / Codex，经 CLI 或 MCP 查 KB）
   ↓
理解项目：这项目干嘛的、有哪些模块、某字段谁改的、某 bug 先查哪、改一处影响哪
```

## 二、六条硬约束（重塑整个设计）

| 约束 | 含义 |
|------|------|
| **本地离线** | 公司代码不能外传。扫描器纯本地；AI 只查 KB，不把整库源码上传。 |
| **代码是"野生的"** | 可能不可编译、缺依赖 jar、混合编码（中文项目常见 GBK/GB2312）、风格不一、多 ISV 前缀。解析器**绝不依赖编译或依赖解析**。 |
| **规模大** | 多模块成百上千文件 → 要性能、进度、缓存、增量重扫。 |
| **信任优先** | 解析覆盖率/可信度报告必须是**一等功能**，而非事后补。用户先信任它，才会给更多代码。 |
| **接手者视角** | 第一需求不是查单字段，而是 **"这项目长什么样"** —— 要能先产出项目地图/理解报告。 |
| **两段式解耦** | 「确定性扫描器(建 KB)」与「AI 理解层(查 KB)」必须解耦，**KB 是契约**。 |

## 三、场景必须补上的能力（原企划未覆盖）

1. **源码摄取层 (Ingestion)** —— 把工具指向一坨乱目录：项目清单 `project.toml`、目录递归发现、编码自动探测（GBK/UTF-8/带 BOM）、排除 `target/ build/ .git/ *.jar`、符号链接处理。野生代码能跑起来的前提。
2. **元数据 ↔ 代码桥接层** —— 全项目最关键的桥：把元数据里的 ClassName（如 `cqspb...AssetCardFormPlugin`）解析到源码树里真实 `.java`。处理三态：① 绑定且有源码（正常）；② 绑定但为平台插件 `kd.bos.*`，无源码（标为外部）；③ 源码里大量 service/util/webapi 未被任何元数据绑定（独立 Java 类，也要纳入）。
3. **解析鲁棒性策略** —— 用 tree-sitter 的错误恢复治"不可编译"；不做类型解析、不解析依赖；遇到不认识的符号（`SaveServiceHelper`、`kd.bos.*`）一律当外部已知符号、用内置 SDK 目录解释，而非报错。
4. **解析可信度 / 覆盖率报告** —— 直接回应"得确认能正确解析"的顾虑。输出：扫描 N 文件、成功解析 M、识别 K 个苍穹插件类、X 个 ClassName 在源码中找到/找不到、哪些文件 AST 有错误片段。是"信不信它、给不给更多代码"的依据。
5. **SDK / 平台符号目录** —— 复用现成 `ok-cosmic-docs.db` 和 `references/`，让工具"认识" `kd.bos.*` 这类无源码平台 API，从而解释入库、事务、F7 等语义。
6. **模块识别 & 项目地图** —— 模块划分以**代码包路径前缀**（如 `cqspb.sysint.assets`）聚类为准，**不用元数据 ISV**（ISV 与包前缀经常对不上，见下「ISV 决策」）；辅以 bizapp 分组，自动推断那 5~6 个业务模块，先给一张"项目地图"。孤儿类（无元数据绑定）也按代码包前缀归模块。
7. **接手者「项目理解报告」** —— 一键生成：项目概览、模块清单、实体清单、插件清单、跨模块依赖、风险热点（无人绑定的孤儿类、超大插件、可疑入库点）。接手第一天最想要的就是这个。
8. **两套前缀命名空间（务必分清）** —— 项目里有**两类互不相干的前缀**：① **元数据标识前缀**（`cqkd_assetcard`、`cqkd_qz` —— 实体/字段标识，防字段串实体）；② **代码包路径前缀**（`cqspb.sysint...` —— Java 包，管类归哪个模块）。同一实体的标识前缀（`cqkd_`）与其绑定插件类的包前缀（`cqspb`）对不上是常态，二者**分别建、不可混**。
9. **处处置信度 + 证据行号 + unknown** —— 老项目分析天生不完整，宁可标 unknown 也不臆造。
10. **增量与可重建** —— 源码/元数据变了只重扫差异；KB 任何时候可从零重建（幂等）。
11. **AI 层"按需取证"** —— AI 不一次吞整库，按问题查 KB 取最小证据集（省 token、保隐私）。
12. **不强依赖 GitNexus** —— 它是增强项；没有它，工具在一个文件夹上也要能独立工作。

## 四、修订后架构（两段式）

```
┌─────────────── 段一：本地确定性扫描器（Python） ───────────────┐
│  Ingestion 摄取  →  Metadata 解析  →  Java 静态分析  →  桥接/路径/入库 │
│  (扫目录/编码探测)   (dym/zip→结构)    (tree-sitter)     (ClassName↔file)│
│                          ↓                                            │
│                 Cosmic KB（SQLite 图谱 + FTS5 + JSON 快照）            │
│                          ↓                                            │
│                 覆盖率/可信度报告 + 项目理解报告                       │
└──────────────────────────────────────────────────────────────────────┘
                          ↓ KB 是契约
┌─────────────── 段二：AI 理解层（CLI / MCP） ───────────────┐
│  语义解析(NL→意图) → 查 KB 取证 → Context Builder → 挂苍穹 Skill 语义  │
│                    → AI 输出带证据的解释 / 排查建议                    │
└──────────────────────────────────────────────────────────────────────┘
              (可选) GitNexus 调用链增强叠加在 Context Builder 上
```

### 阶段 2 · 元数据解析 + 整包处理
> 已对全部样例（单据/基础资料/动态表单/整包/两份模板）做完结构勘探，**勘探结论与逐项口径见
> `docs/阶段验收.md` 阶段 2 小节**；本节是据此排定的"接下来要干什么"。

**目标**：把三类 dym 统一解析成一个 `MetaModel`（表单→实体→分录层级→字段→类型→中文名↔标识→插件→操作），
能从单个 dym 起步、最终批量吃下整包；产出 JSON 快照 + 分类计数报告。

**输入与样例**（`samples/`）：`bill/cqkd_assetcard.dym`、`basedata/cqkd_bill_calculation.dym`、
`form/cqkd_adjusttolcontract.dym`、`appzip/*.zip`（整包，758 dym）；
模板 `bos_temp/bos_billtpl.dym`（单据）、`bos_temp/bos_basetpl.dym`（基础资料，已重导带操作定义）。

**分两步走（拍板的开工顺序）**

- **阶段 2.1 · 单 dym 三类解析器**【我开发】
  - `metadata/dym_parser.py`：统一解析 `DeployMetadata → DesignMetas → {DesignFormMeta, DesignEntityMeta}`。
    - 三类 `ModelType`（`BillFormModel`/`BaseFormModel`/`DynamicFormModel`）走同一入口，
      主实体标签按集合识别（`BillEntity`/`BaseEntity`/`MainEntity`），不写死。
    - **字段**：标签名=类型；抽 `Key`/`FieldName`/`Name`/`Id`；下拉项取 `ComboField/Items/ComboItem`。
    - **层级**：**建树 + 取直接子节点**，按字段 `ParentId → 分录 Id` 回填（无 ParentId=表头），
      子分录链式回溯。**严禁就近/首个匹配**（坑见验收：会把嵌套 `StatusField` 的 Id 误当操作 oid）。
    - **字段口径（分类全保留）**：`kind = entity(有FieldName) / dynamic(动态表单无DB列) / basedata_prop(带出引用) /
      platform(平台标准) / inherited(继承父模板)`；排除 `TargetField/SrcField/DataDimensionField` 等规则表达式标签。
    - **唯一键**：字段用 `Id` 或 `(实体Key, 字段Key)` 复合键（`basedatapropfield` 的 Key 不唯一）。
  - `metadata/template_loader.py`：解析继承根模板为 `hex oid → {operationKey, name, type}` 映射表，
    **按 `ModelType`/`InheritPath` 选模板**（bill→billtpl、basedata→basetpl）；oid 不是全局，每类一套。
  - **插件**（不遗漏）：界面（`*FormAp/Plugins`）、列表（列表布局/Plugins）、操作（`Operation/Plugins`，含 validator）；
    完整保留 `<ClassName>` 全限定名；每条标 `source = project(cqkd./cqspb.) | platform(kd.bos.*)`；
    操作插件回填其绑定操作：自带 Key 的取自身，继承覆盖型（带 oid）走模板表，解不出记 `unknown`。
  - 统一 `MetaModel` 数据结构 + `to_dict()` JSON 快照 + 分类计数报告。
  - **设计决策（ISV）**：解析 `<Plugin>` 时**完整保留 `<ClassName>` 全限定名**（包路径+类名），
    不得只截末段类名 —— 它是阶段 3 桥接到源码的**唯一钥匙**。

- **阶段 2.2 · 整包双层 zip**【我开发】
  - `metadata/package_loader.py`：外层读 `kdpkgs.xml` 清单 → 解 `dm/*.zip` 内层 →
    遍历 `datamodel/<版本>/main/<appKey>/metadata/*.dym` 批量解析；带进度（758 量级）。
  - 列出全部表单（验收第 4 条）；`appKey` 作模块/应用归属线索；`.dymx` 多语言暂作可选补充。

**【你需提供】**（不阻塞 2.1）：含反写插件的 dym（已搁置）；动态表单继承根模板（`6fb46130…`，
缺则 dynamic 操作 oid 记 unknown）。

**【你测试 ✅】** 三类单 dym 解析：字段数按分类口径对得上、字段层级（表头/分录/子分录）正确、
字段类型与下拉项准确、插件三类无遗漏且操作绑定正确（单据/基础资料 oid 能翻译成操作名）；
