# 下一步规划 · 单据视图升级为「场景 → 事件/操作 → 那个插件」（2026-06-30，待下次做）

> 本文记录 2026-06-29 起一次方向讨论的结论，供下个会话直接接续。
> 背景动机：form_key 解析、MCP 防截断都已优化到红线内天花板，问"下一步最该做什么"。
> 2026-06-30 续讨论纠正了"操作"概念混淆，本文已按**两轴模型**整篇重写（旧"操作级钻取"用词作废）。

## 一、出发点：真实排障是「动作优先」，不是「字段优先」

旗舰能力（`trace` + 全部 form_key 优化）是 **「字段优先」** 入口：先有一个字段，再查谁读写它。
但用户真实排障工作流多是 **「动作优先」**：

> 实施顾问报"**某个具体动作**出问题（保存/提交/某按钮/字段联动/调度…）" →
> 进**对应单据/场景该动作的插件** → 找出**那个确定的插件**改掉。

由此**否定**了之前设想的"穷举某字段所有读写来建验证库"——既不可行，也不是真实排障方式。
真实需求是 **「场景/动作 → 那个确定插件」** 这条链。

> ⚠️ 但"动作"一词要拆清楚，否则方向会带偏——见第二节两轴模型。

## 二、核心模型：「操作/动作」有两层含义，是两条正交的轴

苍穹是**动态领域模型（充血模型）**：单据实体被元数据定义了可执行动作。把"操作/动作"压成一层会带错方向，
实际是两条**正交轴**：

- **轴 A · 场景 / 插件类型**：用户当时处在哪个平台预制业务场景，决定干预用哪类插件——
  打开界面交互→**界面插件(form)**；执行调度计划→**调度插件(task)**；打印/导出→**print/import**；
  接口集成→**webapi**；单据执行元数据预制动作→**操作插件(op)**；还有 list/convert/writeback/workflow。
- **轴 B · 元数据预制"操作"**：充血模型上被元数据定义的可执行动作（save/submit/audit/unaudit/delete…，
  见 `references/base/plugin/plugin-operation.md` 末尾预置清单）。**这条轴只对操作插件成立。**

**最容易错的点**：界面插件的"动作"（打开界面、字段联动、按钮）**不是**元数据操作——它绑的是
**单据/界面 + UI 生命周期事件**（initialize / propertyChanged / itemClick），**没有 operation_key**；
调度/打印/导出/接口同理，各绑各的场景触发点。**只有操作插件这条车道才有 save/submit/audit 这层维度。**

因此正确导航是**先按轴 A 分流，再在操作插件车道上才挂轴 B**：

| 顾问报的"动作" | 轴 A 分流到 | 再按什么定位到那个插件 |
|---|---|---|
| 打开界面 / 字段联动 / 按钮 | 界面插件(form) | **UI 事件**（initialize/propertyChanged/itemClick）——绑 form_key，**无** operation_key |
| 保存 / 提交 / 审核 报错 | 操作插件(op) | **operation_key**（save/submit/audit）+ 事务相位事件（beforeExecute…/begin…/end…） |
| 调度 / 打印导出 / 接口 报错 | task/print/import/webapi | 各自场景触发点（无 operation_key） |

**用户确认**：① 两轴模型对；② 排障里**操作插件 + 界面插件占绝大多数**，其他类型也都会遇到。
→ 优先把这两条主力车道做厚，其他车道至少给正面入口（别让它们在单据视图里隐身）。

> ⚠️ 但轴 A 的多数车道**结构上挂不到某张单据**——非单据绑定的插件类（workflow/task/openapi）没有
> form_key，无法在 `bill` 下分流。详见第四节"插件覆盖边界"，这直接决定第六节交付物的措辞与拆分。

## 四、插件覆盖边界：我们的扫描结构上只覆盖"绑在单据元数据上"的一部分

> 来源：苍穹官方《Java 插件开发流程》（vip.kingdee.com/article/239776518421883648）列 **12 种插件**，
> 按注册位置分两组。对照后坐实——这是**红线 #4 必须显式声明的盲区**，也决定第六节交付物的拆分。

元数据解析器（`metadata/model.py:38`）只产出 **5 种** `plugin_type`：`form/list/op/writeback/convert`。
对照官方 12 种，缺口分两类，**性质不同**：

- **缺口 A · 在摄取范围内、解析器没抽出来（可修的解析漏）**：**套打/打印插件**（绑打印模板业务属性）、
  **报表界面/取数插件**（绑报表页面/查询）。这些 dym 我们**确实摄取**（`ReportFormModel→report` 已解析），
  但解析器**没把它们当作带 form_key 的插件抽出来**。红线内可补。
- **缺口 B · 根本不在摄取范围（独立子系统注册，scope 外）**：**工作流插件**（流程服务云·流程设计）、
  **后台任务插件**（系统服务云·调度作业）、**OpenAPI 插件**（开发服务云·开放平台）——它们
  **不绑任何单据**，注册在我们压根没摄取的子系统里。
  （**转换插件是例外**：注册在转换规则 `.cr`，我们已解析为 `convert`，已覆盖。）

**今天这些"非单据绑定"插件靠什么被看到？只有源码父类。** `plugin_classifier` 父类兜底 → 落成**孤儿插件**
（`source_class.plugin_base` + `orphan_role='plugin'`，**无 form_key**）。`bridge_report.py:41` 已按基类聚合
成 `plugin_orphan_by_base`（`bridge` 报告能看"哪些类型的插件没被绑定"）——**数据在，但没进排障主路径、
没进 MCP / 可信度页**。

**两条结论（直接约束第六节）**：
1. `bill` 的"轴 A 场景分流"**结构上只能覆盖单据绑定的那几类**（form/list/op/writeback/convert，
   外加修复缺口 A 后的 print/report）。task/webapi/workflow **没有 form_key，挂在任何单据下都是空的**。
   → 主力车道（op+form）正是单据绑定的，bill 升级依然成立，但**措辞要降级为"单据绑定插件的分流"，
   不能假装是完整场景清单**。
2. 非单据绑定的插件**另起一条入口，锚在"插件类型目录"而非单据**：把已有的 `plugin_orphan_by_base`
   提成一等排障入口（CLI/MCP/Web 同口径）。"调度跑错 / 接口报错 / 流程节点出错"的排障**起点是
   作业/API/流程，不是某张单据**，按苍穹基类分组列孤儿插件正好对上——便宜、且诚实。

## 五、坐实的代码缺口（核过源码，是展示/拼接缺口，非数据缺口）

`cosmic_kb bill <单据标识>`（`report/bill_view.py`）已覆盖大半：操作集（`★` 标有自定义插件的操作）、
插件清单（`[类型]` + `←operation_key`）、字段触达、桥接风险。但离"镜像顾问路径"差三点：

1. **插件清单没按轴 A 分组**：`report/bill_view.py:35` 查到的 `plugin.plugin_type` 就是
   form/list/op/writeback/convert（`schema.sql:69`）——**单据绑定插件的分流维度本身够用**，缺的只是
   渲染层没按它分组（见第 3 点）。注意：task/webapi/workflow 等**非单据绑定**插件本就**没有 form_key**、
   不会进 `plugin` 表，`bill` 里看不到它们是**结构使然**（第四节缺口 B），不是这里的 bug——它们走第六节
   的独立"孤儿插件目录"入口。
2. **只到"半跳"，无事件/相位**：`bill_view` **完全没 join `plugin_method`**（`event_kind`/`event_phase`
   现成躺在表里，`schema.sql:128`）。插件清单只说"[op] 某类 ←submit"，**不说它处理哪个生命周期事件、
   什么相位**（beforeExecute/begin/end/校验）。而顾问报的"动作"常对应一个**事件**——现在要再对每个插件
   `calls` 一次才知道它处理哪个事件，多跳、易看花。
3. **平铺、无语义**：渲染层（`bill_view.py:332`）插件清单是一把梭平铺，没按场景分组、无事件触发时机说明。

## 六、下一步交付物（两条入口，对应第四节两类插件）

### 主线 · `bill` 插件清单升成「场景 → 事件/操作 → 那个插件」（**只覆盖单据绑定插件**）
数据基本现成，主要是 **分组 + join + 语义标注**，CLI/MCP/Web 同口径。
**措辞要诚实**：这是"**单据绑定插件**的场景分流"，不是完整插件清单（非单据绑定插件见旁路入口）。

1. **轴 A 分流**：按 `plugin.plugin_type`（form/list/op/writeback/convert）分组——操作/界面/列表/反写/转换。
   （这一维度现成；若先修第四节缺口 A，再补 print/report 两类。）
2. **轴 B 只在操作插件车道展开**：op 插件下按 `operation_key`（save/submit/audit）二级分组，
   join `plugin_method`（`schema.sql:128`）按 `event_phase`（beforeExecute→begin→end→after）排序，
   一跳落到"提交时跑的那个插件 + 它的事务事件"。
3. **界面插件车道**：按 UI 事件列（initialize/propertyChanged/itemClick），**不挂 operation_key**。
4. **每插件/事件焊一句语义**：插件类型一句话 + 该事件触发时机（取自 `references/base/plugin/*.md`，
   知识已落盘），让"动作 → 事件"这步不靠大模型猜。

→ `bill` 从"字段优先的旁路"变成**镜像顾问真实路径**（场景→动作→那个确定插件）。

### 旁路 · 非单据绑定插件的"按类型目录"入口（workflow/task/openapi…）
把 `bridge_report.py:41` 已有的 `plugin_orphan_by_base`（按苍穹基类聚合的孤儿插件）**提成一等排障入口**，
进 CLI/MCP/Web。"调度跑错 / 接口报错 / 流程节点出错"从这里按插件类型找到那个类——起点是作业/API/流程、
不是单据。便宜、且诚实地补上 bill 覆盖不到的那半。

### 信任优先 · 声明覆盖边界（红线 #4）
`coverage` 显式写明："插件覆盖只到摄取的元数据范围；workflow/task/openapi 等独立注册、及 print/report 等
暂未解析的插件类**不绑单据**，仅靠源码父类识别为孤儿、按类型在独立目录呈现，**可能不全**。"
不让 `bill` 给人"这张单的插件齐了"的错觉。

### 附带可做（手段①，便宜高频）
新增"源码坐标 → 所属插件/事件/操作/读写字段"反查（给段二一行 `FQN:行号` 即可定位上下文）；
`source` + `plugin_method` 数据已具备，押后于主线。
