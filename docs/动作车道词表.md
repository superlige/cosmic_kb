# 动作车道词表（顾问人话 → 候选插件/事件 的封闭映射）

> 用途：把实施顾问的自然语言反馈（"保存后金额重算了"/"点了按钮没反应"）映射到
> **苍穹事件车道**，驱动 `bill` 升级版的「场景 → 事件/操作 → 那个插件」导航（见
> `docs/下一步-操作级排障验收与钻取.md` 第六节）。
>
> **两端共用一张表**：
> - **段二 LLM** 把顾问人话**只能从本表车道里挑**（封闭选项，防臆造事件名，守红线#1/#4）。
> - **确定性工具** 拿车道 id → 查 `plugin / plugin_method / field_access` 三表做 join 出候选。
>
> **设计原则**：**recall 优先**——宁可多框一两个候选交人确认，绝不漏。落错到隔壁车道才是事故，框宽不是。
> 工具只负责把 N 个插件缩成带证据的短清单，**最终"就是它"仍由人/LLM 读候选确认**，不替你拍板。
>
> 事件名取自 `cosmic_kb/semantics/references/base/plugin/*.md`，与 `plugin_method.event_kind` 同口径。

## 一、数据模型映射（车道怎么落到我们的表）

> ⚠️ **真实库核对（2026-06-30，`D:\kingdee\asset_management_sys\cosmic_kb.db`）后修正**：
> 车道 join 的**主表是 `field_access`，不是 `plugin_method`**。原因见附录 A，结论先放这：
> `plugin_method` 只收**单据绑定插件**（op/form/list/convert），**孤儿车道（task/report）整张表为 0**，
> 且 `beforeF7Select / afterFieldMapping / filterContainerInit` 被漏分类成 `helper`。
> 而 `field_access.event_method` **全都有**（execute 2876、query 918、beforeF7Select 167、afterFieldMapping 136…）。

**主 join（覆盖全部车道，含孤儿）——走 `field_access`：**
```
field_access(form_key, field_key, level, entry_key,
             plugin_fqn, plugin_type, access_class,
             event_method, event_phase, access, line, ...)
-- 已自带 车道所需全部维度：单据 + 字段 + 插件类型 + 入口事件 + 读写 + 行号
查询 = WHERE form_key=单据 AND plugin_type IN(…) AND event_method IN(车道事件集) [AND access='write']
```

**辅 join（仅在三种情况补 `field_access` 取不到的）——走 `plugin_method`：**
1. **纯校验/抛异常事件**（如 `onAddValidators` 只加校验器、不写字段）→ field_access 里没有，只有 plugin_method 能证明"该插件实现了此事件"。
2. **event_phase 相位语义**（beforeExecute→begin→end 排序）。
3. **每方法的精确行号区间**（start_line/end_line）。
> 但 plugin_method 缺孤儿车道，故它只能当**单据绑定车道的补充**，不能当主键。

**孤儿车道（task/report/workflow…）的"类型目录"入口——走 `source_class`：**
```
source_class(fqn, plugin_base, orphan_role)  -- 按苍穹基类聚合（plugin_orphan_by_base）
-- 它们无 form_key、不进 plugin/plugin_method；字段读写仍在 field_access（plugin_type=task 等）
```

## 二、单据绑定车道（出现在 `bill` 下；form/list/op/writeback/convert）

### 2.1 操作类车道（op + form 协同，**带 operation_key**，轴B在此展开）

顾问说"**保存/提交/审核…报错**"或"**…之后某字段重算/状态变了**"。
同一动作横跨**两类插件**：服务端**操作插件(op)** 跑事务逻辑，**界面插件(form/bill)** 跑操作前后 UI 协同。

| operation_key | 顾问会怎么说 | 操作插件(op) 事件组（服务端事务，按时序） | 界面协同(form/bill) |
|---|---|---|---|
| `save` | 保存 / 存盘 / 保存报错 | onPreparePropertys → onAddValidators → beforeExecuteOperationTransaction → beginOperationTransaction → endOperationTransaction → afterExecuteOperationTransaction（+rollbackOperation 回滚后） | beforeDoOperation / afterDoOperation |
| `submit` | 提交 | ↑ 同一组事件 | ↑ |
| `unsubmit` | 撤销提交 / 反提交 | ↑ | ↑ |
| `audit` | 审核 / 审批通过 | ↑ | ↑ |
| `unaudit` | 反审核 / 弃审 | ↑ | ↑ |
| `delete` | 删除 | ↑ | ↑ |
| `saveandnew` | 保存并新增 | ↑ | ↑ |
| `submitandnew` | 提交并新增 | ↑ | ↑ |
| `statusconvert` | 状态转换 / 单据状态变了 | ↑ | ↑ |
| `disable` / `enable` | 禁用 / 启用 | ↑ | ↑ |
| `invalid` / `valid` | 作废 / 生效 | ↑ | ↑ |
| `donothing` | 自定义按钮触发的空操作 | ↑ | ↑ |

**op 事件组的相位语义**（帮顾问的"什么时候出问题"落到具体事件）：

| 事件 | 相位 | 触发时机 / 干什么 | 常见症状对应 |
|---|---|---|---|
| onPreparePropertys | build | 加载数据包前补字段 | "取某字段是空的" |
| onAddValidators | validate | 执行校验前增删校验器 | "保存/提交被拦报错、必填校验" |
| beforeExecuteOperationTransaction | transaction(前) | 校验过、开事务前最后整理 | "提交前数据被改/兜底校验" |
| beginOperationTransaction | transaction(内) | 事务内、写库前 | "事务内同步、级联改数" |
| endOperationTransaction | transaction(内) | 写库后、提交前 | "落库后事务内后置处理" |
| afterExecuteOperationTransaction | none(事务后) | 事务已提交 | "保存成功后发消息/日志/级联其他单" |
| rollbackOperation | none | 提交失败回滚后 | "失败补偿" |

> 子情况说明：顾问**说得出"哪个字段变了"** → 直接 `field_access WHERE form_key=单据 AND field_key=该字段
> AND event_method IN(op事件组)`，基本一跳到那个插件（**trace 已覆盖大半，这里只是按动作过滤**）。
> 顾问**只描述行为、说不出字段** → 列出该 operation_key 下所有 op/form 候选，按"有写操作"排序，缩圈交人确认。

## 三、非单据绑定车道（**不在 `bill` 下**；走"插件类型目录"旁路入口）

> 这些插件**没有 form_key**，结构上挂不到任何单据（缺口B：独立子系统注册）。
> 顾问报这类问题时**起点是作业/流程/接口/报表，不是某张单据**——从 `plugin_orphan_by_base`
> 按苍穹基类分组的孤儿插件目录进。（红线#4：`coverage` 须声明此覆盖边界。）

| 车道 id | 顾问会怎么说 | 插件类型 | 基类 | event_kind |
|---|---|---|---|---|
| `schedule` | "夜间同步没跑"、"定时任务/调度作业出错" | task | AbstractTask | execute / stop |
| `workflow` | "流程审批人算错"、"条件分支走错"、"流程通知没发" | workflow | IWorkflowPlugin | calcUserIds / hasTrueCondition / notify / notifyByWithdraw / formatFlowRecord |
| `import` | "Excel 导入报错/校验/导不进来" | import | BatchImportPlugin | save / beforeSave |
| `print` | "打印模板空白/取数不对/控件输出错" | print | AbstractPrintPlugin | beforeLoadData / loadCustomData / beforeOutputWidget / afterOutputWidget |
| `openapi` | "外部接口调用报错/返回不对" | webapi/openapi | @ApiController(注解) | @ApiGetMapping / @ApiPostMapping 方法体 |
| `report` | "报表查询慢/数据不对/列不对" | report-form / report-data | AbstractReportFormPlugin / AbstractReportListDataPlugin | query / beforeQuery / afterQuery / processRowData / getColumns / packageData |

## 五、给 LLM 的使用契约（封闭选项 + 兜底）

1. 顾问人话 → **只能映射到本表已有车道 id**（封闭集），不得自创事件名。
2. **一句话可命中多个车道**就都列（recall 优先）。例："保存后金额重算了" → `save`(2.1) + `field-linkage`(2.2)。
3. **判不出车道** → 工具退化为"列出该单据全部车道的全景分组视图"，仍胜过盲开 N 个文件。
4. 工具返回是**带证据的候选短清单**（插件 / 事件 / 相位 / 碰了哪些字段+中文名 / 行号）；
   **"就是这个插件"的最终结论留给人读候选确认**，不臆造。

## 附录 B · 真账验收：`cqkd_assetcard`（资产卡片，2026-06-30）

按文档《下一步…》第七节"用已修真 bug 当免费标准答案"法，挑实施顾问最常报的两类反馈跑漏斗。
该单 **绑定 25 个插件**（op 17 / form 4 / list 4），正是"灾难级人工排查"现场。

### B.1 顾问反馈A：「保存后**闲置面积**算错了」（说得出字段）
漏斗：**25 插件 → 2 行命中**（写 `cqkd_idle_area`）：
```
[propertyChanged/memory] AssetCardFormPlugin:96
[propertyChanged/memory] AssetCardFormPlugin:118
```
✅ **一跳到位，25→2 行。** 但 **暴露关键真相**：顾问说"保存后"，真实写入却在 **propertyChanged（字段联动）**——
保存只是把内存里早算好的值落库。**若按顾问说的"save 车道"硬过滤，会把真凶 propertyChanged 滤掉、命中 0。**
→ **设计修正①：字段已知时，"动作"只能当软提示/排序，绝不能当硬过滤**（顾问普遍误判动作）。
此时正确行为 ≈ 现有 `trace` + 写过滤 + 按动作排序。**即子情况A的增量主要是排序呈现，trace 已覆盖主体。**

### B.2 顾问反馈B-写：「保存后某功能触发了/某值变了」（说不出字段）
漏斗：**25 插件 → save 车道候选**，按"是否真写字段"排序：
```
SetCompressImageMarkOp : afterExecuteOperationTransaction(写2字段)   ← 最可能
AssetCardFormPlugin    : propertyChanged(写17字段)
其余(CodeRuleOp/校验类等): save事件无字段写
```
⚠️ **有用但只是短清单（25→约3个真写的），不是定点**。这是诚实的最好结果——人读 3 个而非 25 个。

### B.3 顾问反馈B-校验：「保存/提交**报错**」（不写字段，纯校验拦截）
这类真凶（校验器）以读单据字段 + addErrorMessage 为主。复核真实库（注意：`AssetCardStateValidator`
整文件被注释、是死代码，样本无效，改用真实生效的 `AssetChangeValidator`）：

| 校验器形态 | 真实表现（修复前） |
|---|---|
| `BaseDataCheckRefrenceOp`（操作插件内 onAddValidators） | ✅ `plugin_method` 有 `onAddValidators/validate` |
| `AssetChangeValidator`（**顶层 `extends AbstractValidator`**，op 插件 `new` 挂载） | ❌ `orphan_role='unknown'`、`plugin_base=None`、`field_access.plugin_type='service'`、`plugin_method` 空 |
