| 平台/继承字段特定单据下 mismatch 加 `note` 弱化措辞 | `auditdate` 类平台通用字段纯继承、零 override，该单据的 dym 里本就没有对应 `field` 元素，`resolve_fields` 找不到就报 mismatch，但这不是"限定符写错单据"；命中候选里存在 `field_kind in {platform,inherited}` 时 `mismatched_form[key]` 加 `note` 说明，渲染用 ℹ 而非 ⚠ | ✅ |
| `resolve_fields` `kind="entity"` 两段式分录限定符 fail-closed | 真实翻车：模型传两段式「分录.字段」（如 `cqkd_zdgl.cqkd_zdzfltk`）+ `kind="entity"`，无单据前缀时 `_matches` 不按单据过滤，若该坐标恰好只在别的单据下存在会返回看似钉准实则未经单据校验的候选；改为该分支直接拒绝，顶层新增 `invalid_request[key]`（`reason="missing_form_key"`+`entry_key`/`field_key`/`hint`），`resolved[key]` 给 `None`，强制改传三段式「单据.分录.字段」；三段式/字段侧两段式/裸 key/`kind` 不传时不受影响。MCP `tool_resolve_fields` 的 `kind` 同步改用 `Literal["field","entity","form"]` 类型标注 | ✅ |
| `resolve_fields` 新增 `kind="plugin"`：插件类名反查绑定单据 | 起因：`bill`/`trace` 的 instructions 明令"插件源码解读必须先核对绑定关系"，但只有插件类名（无 `form_key`）时工具箱没有任何一环能完成这次核对，模型被迫走命名惯例猜测；新增分支整串 `key` 当类名处理（跳过点号坐标限定符协议），`plugin` 表精确 `class_name` 命中为主，查不到且为裸简单类名时退化 Python 侧全表扫描按末段类名匹配，`plugin` 表零命中回落 `binding` 表（只标 `binding_status`/`confidence`/`source_relpath`，不臆造 `plugin_type`/`operation_key` 等元数据登记信息）；两表都查不到但 `source_class` 命中 `orphan_role='plugin'` 时给顶层 `unbound_in_source[key]`（诚实区分"钉不出绑定"与"钉不出类本身"两种 unknown）。CLI `--kind plugin`、MCP `tool_resolve_fields(kind="plugin")` 同步；不改 schema、不改 build | ✅ |
| `resolve_fields` `kind` 支持逐 key 列表 | 真实翻车：模型批量传入单据号/分录容器/字段三个不同层级的 key，却用单个 `kind="field"` 广播给全部三个，前两个全部落入 `mismatched_kind`（诚实报错但等于白跑一次）——根因是接口形状允许"整批一个 kind 值"套到分属不同层级的 key 上。`kind` 改为同时接受与 `keys` 等长的列表逐位对应（`None`/CLI `none` 占位表示该位不限定），长度不一致直接 `ValueError` 拒绝静默截断；单个字符串广播的老用法不变（整批同层级时仍适用）。CLI `resolve --kind` 改 `nargs="+"`、MCP `tool_resolve_fields(kind=...)` 类型标注同步扩展为 `Literal | list[Literal\|None] \| None` | ✅ |
| `read_source`（模式 A，含三档消歧/常量解析/表单兜底/防截断/游标提醒等 7 轮迭代） | 🗑 已退役（2026-07-05）——本地源码已是纯 UTF-8，"防乱码"前提不再成立；改走宿主自带 reader + `resolve_fields("实体key.字段key")` 精确核对。CLI `cosmic_kb source` 不受影响，富模式函数原样保留 | 🗑 |
| `method_calls`（方法出向调用导航） | 🗑 已退役（2026-07-05）——用户三轮追问证伪设计动机，唯一窄场景（通配符 import/继承方法）用通用 Grep/Read 即可解决，维护成本不划算，整体删除 | 🗑 |

**验收命令**：`cosmic_kb trace "单据.字段"` / `cosmic_kb bill "<单据标识>"` / `cosmic_kb resolve "<标识> ..."`
