# 苍穹项目 AI 工程知识底座

## 项目定位

本项目不是普通代码搜索工具，也不是单纯的 Skill，更不是完整 Agent。

它的定位是：

```
面向金蝶苍穹历史项目的 AI 工程知识底座
```

目标是把：

```
苍穹元数据
+ Java 插件代码
+ DynamicObject 数据结构
+ 苍穹 SDK 语义
+ GitNexus 代码图谱
```

组织成 AI 可以查询、理解、解释的项目知识上下文。

最终让 AI 能回答：

```
某个业务字段被哪些插件/服务/任务控制？
某个插件事件在什么前端或后端时机触发？
某个字段是主实体字段、分录字段、子分录字段，还是基础资料字段？
某个操作按钮会影响哪些实体和字段？
某个 bug 应该优先排查哪些插件类、普通 Java 类或业务流？
```

```

```

### 4. DynamicObject 路径追踪层

这是项目核心难点之一。

目标不是只知道代码里出现了某个字段，而是知道字段属于哪条数据路径。

需要识别：

```
主实体字段
cqkd_assetcard.cqkd_mortgagestatus

分录字段
cqkd_assetcard.cqkd_entryentity.cqkd_qz

子分录字段
cqkd_assetcard.entry.subentry.field

基础资料字段
cqkd_assetcard.cqkd_customer.name
```

需要处理代码模式：

```
DynamicObjectCollection entry = bill.getDynamicObjectCollection("entryKey");
DynamicObject row = ...
row.set("fieldKey", value);

DynamicObject base = bill.getDynamicObject("baseDataField");
base.getString("name");
```

这层用于解决：

```
不同实体存在相同字段标识时如何区分
字段到底属于主实体、分录、子分录还是基础资料
```

------

### 5. 持久化判断层

负责判断代码修改字段后，是否真的可能写入数据库。

需要识别：

```
确认入库
- set 后同方法 SaveServiceHelper.save

疑似入库
- set 后调用封装 save 方法
- set 后 OperationServiceHelper.executeOperate

事务自动保存
- 操作插件事件中修改 e.getDataEntities()

未知
- 只修改方法参数
- 没找到保存链路
```

输出不应该简单是 true/false，而是：

```
confirmed
likely
unknown
```

并带原因。

------

### 6. 业务流分析层

用于分析实体之间的业务流关系。

目标识别：

```
单据 A 引用基础资料 B
单据 C 下推生成单据 A
单据 A 的提交/审核操作写回单据 B
某个操作通过代码创建或保存另一张单据
某个字段变化触发另一张单据字段变化
```

这层需要结合：

```
元数据引用关系
操作插件绑定关系
DynamicObject load/set/save 链路
GitNexus 调用链
苍穹转换规则 / BOTP 元数据
```

------

### 7. 语义解析层

负责把用户自然语言转换成真实项目查询。

例如：

```
用户：资产卡片抵押状态是谁改的？
```

转换成：

```
{
  "intent": "field_impact",
  "entityKey": "cqkd_assetcard",
  "fieldKey": "cqkd_mortgagestatus",
  "access": "write"
}
```

需要能力：

```
实体中文名匹配
字段中文名匹配
字段别名匹配
操作名称匹配
模糊匹配
候选打分
不确定时反问
```

这层解决“用户必须输入精确字段标识”的问题。

------

### 8. 苍穹 Skill 层

Skill 不存项目事实，只负责教 AI 如何解释苍穹语义。

内容包括：

```
插件类型解释
事件触发时机
SDK 方法含义
页面赋值和数据库写入的区别
操作插件事务阶段
常见业务排查规则
回答格式要求
不确定性表达规则
```

例如告诉 AI：

```
propertyChanged 通常是字段值变化后触发
beforeDoOperation 是前端操作前
afterExecuteOperationTransaction 是后端操作事务后
DynamicObject.set 不等于一定入库
SaveServiceHelper.save 是显式保存
```

------

### 9. AI Context Builder

负责把图谱查询结果整理成 AI 可回答的上下文。

输入是结构化图谱数据，输出是：

```
实体：
字段：
写入点：
读取点：
插件类型：
事件触发时机：
是否可能入库：
代码证据：
GitNexus 调用链：
风险判断：
排查建议：
```

它是 AI 回答质量的关键，不让 AI 直接面对零散 JSON。
