# 跨类调用链解析与编译期符号方案

> 2026-07-15 讨论落库，**本次对话不开发，下次对话继续**。本文档是设计方案 + 已验证的
> 现实依据，供下次会话直接接续，不用重新调研。

## 起因：真实调用反馈暴露的两个根因

用户用 `trace(kind="operation")` 排查 `updatehistory` 操作触发链时，工具正确返回了 3 处
直接触发点，但漏了 `ContractRefundUpdateAssetOp` 通过 `ContractService.updateRlateAssets`
间接触发的一跳。复盘后定位到两个根因（不是 trace 设计错了，是工具链缺一环）：

1. **解析器 gap**：`java/ast_index.py:iter_invocations` 只扫 tree-sitter 的
   `method_invocation` 节点，`Class::method` 这类 `method_reference` 节点完全没被扫描——
   不止 op_trigger，全项目所有"调用点提取"能力都吃不到方法引用。
2. **反查工具缺失**：`java/call_graph.py:build_call_graph` 只建**类内、单向**调用图
   （"我调了谁"），没有跨类反向查询（"谁调了我"）。`op_trigger.py` 设计注释里写的
   "级联链由 agent 递归查 trace 拼出"，隐含假设了"下一跳"有确定性工具可查——但这个
   反查工具根本不存在，agent 只能退化成 Grep，而 Grep 天然漏方法引用语法、也不验证
   类是否真被引用（死代码误判为调用链）。

## 拍板：删除"解析器绝不依赖编译"这条硬约束

`CLAUDE.md` 六条硬约束里"代码是野生的……解析器绝不依赖编译或依赖解析"这条，用户拍板
**彻底删除**（原假设"开发者拿到源码不能编译还怎么排障修 bug"不成立——实际都是先把
build 跑通再排障）。

**范围澄清**：这条红线字面上覆盖全部解析层，但实际只有 `java/` 子包的设计依赖这个假设
（语义级 Java 分析：call graph / field access / plugin 绑定）。另外三个子包不受影响：

- `ingest/`：文件发现 + 编码探测，跟编译无关。
- `metadata/`：解析 dym/zip 结构化元数据，根本不是 Java 源码。
- `bridge/`：靠 `<ClassName>` 全限定名做字符串/路径匹配，不涉及编译。

所以本次架构调整**收窄到 `java/` 子包**，其余三个子包不重写。

## 关键澄清：不需要真的执行构建

苍穹平台依赖（`kd.bos` 等）是**已编译好、完全在本地磁盘的 jar**，不涉及网络拉取或版本
解析。真正要做的只是两件更小的事，都不需要调用 `mvn`/`gradle`/`javac` 跑一次真实构建：

1. **类路径发现**：从工程元数据（IDEA 工程文件 / Gradle 构建脚本）解析出"哪些 jar +
   哪些源码目录"构成类路径——纯文本解析，跟 `metadata/` 解析 dym/zip 同一性质。
2. **符号解析**：把二开源码 + 类路径喂给 `JavaParser + Symbol Solver`
   （`CombinedTypeSolver` = 各模块 `JavaParserTypeSolver`（含依赖的兄弟模块源码目录）+
   `JarTypeSolver` 指向平台 jar farm），这个库本来就是为"不跑 build、直接给源码目录 +
   jar 列表做类型绑定"设计的。

**这意味着"去掉不可编译红线"不等于"我们要执行编译"**——我们自己依然不跑任何构建，只是
从"完全不看类路径"变成"读懂类路径声明、接进符号解析器"，比最初设想的轻量得多，也没有
破坏"本地优先、离线确定性扫描"的气质。

### 共同本质

两代构建体系殊途同归：苍穹平台依赖的本质都是**本地磁盘上一堆已编译好的 jar，按
bos/trd/biz/cus 分类目录平铺**，只是"声明入口"换了格式和位置（IDEA 工程文件 vs
Gradle 脚本）。这让"类路径发现"从"要不要跑通构建/连内网仓库"的高风险问题，降级成
"解析几个规整的配置文件"的低风险问题。

## 新架构设计

```
ClasspathProvider（抽象）
 ├─ IdeaClasspathAdapter        解析 *.iml（orderEntry type=module/library）
 │                              + .idea/libraries/*.xml（jarDirectory）
 ├─ GradleCosmicTemplateAdapter 按验证过的三级回退链解析 cosmic_libs_path
 │                              + 固定子目录名枚举 jar
 │                              + settings.gradle/project(':x') 建模块图
 └─ 显式兜底 --classpath-dir    两种都识别不出时（裸源码导出、无工程文件）手工指定

  ↓ 产出：{ 模块→源码根 }、{ 模块→依赖模块 }、jar farm 目录列表

Java 符号解析微工具（新增 JVM 组件，JavaParser + Symbol Solver）
  CombinedTypeSolver = 各模块 JavaParserTypeSolver（含依赖的兄弟模块源码目录）
                      + JarTypeSolver（jar farm）
  Python 侧 subprocess 调用，喂路径、收结构化 JSON（每个调用点的精确目标 FQN+签名，
  或 unresolved + 原因）

  ↓ 产出喂给 java/ 子包现有 KB 建库流程，替换/增强 call_graph.py 的类内单向匹配
    和 op_trigger.py 的名字匹配
```

需要拍板但本文档先留白（下次会话定）：没有 classpath 或某模块解析失败时，是直接
不可用还是退化回现有 tree-sitter 名字匹配当兜底（精度分级共存）。

## 优化后带来的提升

1. **调用链能追全，不再依赖 agent 手写 Grep**：这次真实翻车的直接原因（漏方法引用
   语法 `Class::method`、不验证类是否被实际引用）由确定性工具从根上解决，agent 不用
   再面对"正则写严了漏调用、写松了引入误报"的两难。
2. **多态/接口实现解析**：现有 tree-sitter 纯 AST 方案对"接口方法被哪个实现类接住"
   完全无能为力（比如 `IOperationServicePlugin` 的抽象方法被谁实现），编译期符号解析
   能做，这是纯文本方案永远补不上的能力缺口。
3. **字段访问精度提升，压低 unknown/ambiguous 比例**：涉及真实字段类型、getter/setter
   精确匹配的场景，编译期类型信息能直接提升字段级排障准确率——这直接影响"信任优先"
   这条硬约束下最核心的几个指标（字段覆盖率、`null_reason` 成因诊断）。
4. **死代码判定从"猜"变成"证"**：现在 agent 要验证一个类是否被实际引用，只能另外
   跑一次 import 搜索；有了跨类反查工具后，"查无调用方"本身就是一个确定性事实
   （0 处调用），不需要额外验证步骤。
5. **不牺牲"本地优先、离线"的气质**：全程不需要执行 `mvn`/`gradle` 构建、不需要连
   网络仓库——只解析本地配置文件 + 绑定本地 jar，跟现有 `metadata/` 解析 dym/zip 是
   同一路数的确定性扫描，只是这次解析对象从"元数据包"换成了"构建依赖声明"。

## Spike 结论（2026-07-15，端到端验证已通过）

用 Gradle 建了个最小 Java 工程（`javaparser-symbol-solver-core:3.25.10`），CLI 小工具
接收 jar 目录 + 源码目录，构建 `CombinedTypeSolver`，对源文件里的 `MethodCallExpr` +
`MethodReferenceExpr` 逐个 `.resolve()`，统计成功率/耗时/失败归因。三项待验证全部通过：

**1. 类路径完整性（两个真实项目都跑通，无需任何真实构建）**

| 项目 | jar 数 | 抽样文件数 | 调用点解析成功率 |
|---|---|---|---|
| IDEA 工程 `asset_management_sys` | 3678 | 1345（全量） | 96.5%（83205/86189） |
| Gradle 工程 `zlgd_ygpt2.0` | 4442 | 457 | 96.7%（24924/25780） |

残留失败按 `UnsolvedSymbolException` 消息归因后，绝大多数是 JavaParser Symbol Solver
自身的已知局限（`javaparser-symbol-solver` GitHub issues 里能查到同类报告），不是类路径
缺失：泛型通配符/约束擦除（`type declaration cannot be found on constraint T`）、
`Collectors`/`Stream` 链式调用里的方法引用类型推断、javassist 反射签名下的重载消歧
（`MethodAmbiguityException`）。**说明"类路径发现"这一步本身是可靠的**——96%+ 的成功率
已经是同类工具（IDE 里的 Java 索引）的正常区间。

**2. jar farm 规模下的加载耗时（结论：完全可接受，热盘 3~5 秒级）**

- IDEA 工程 3678 jar：构建 `CombinedTypeSolver` 耗时 4.4~4.7 秒（多次复测稳定）。
- Gradle 工程 4442 jar：**首次**冷盘/未缓存耗时 142.7 秒（该 jar farm 在
  `D:\kingdee\workspace\...` 首次被本机进程完整读一遍，OS 文件缓存全未命中）；
  **同一目录热盘复测仅 3.2 秒**，比 IDEA 工程还快。结论：耗时几乎全部是磁盘 I/O
  首读成本，`JarTypeSolver` 本身对 4000+ jar 规模的索引开销可以忽略——但也意味着
  **首次对一个新项目建 KB 可能有一次性的"冷盘税"**（几十秒到几分钟级，取决于 jar
  farm 所在磁盘/网络盘速度），后续增量重扫（阶段 11）会自然吃到热盘缓存，不需要
  额外做缓存层。
- 抽样解析（含符号解析）耗时：IDEA 全量 1345 文件 55.6 秒（~41ms/文件），Gradle
  457 文件 17.9 秒（~39ms/文件）——线性，规模可控。

**3. 精确解析验证真实翻车同构场景（结论：符号解析 + scope 兜底可完整解决原问题）**

在 `asset_management_sys` 项目里找到了与 `ContractService::updateRlateAssets` 同构的
真实场景：`getZqzdGroupKey` 这个静态方法名在项目里被**四个不同的类**各自定义一份
（`ContractArrearsYSRQPlugin`、两处 `ContractLeaglFormPlugin`、两处 `ContractService`），
纯名字匹配（Grep/tree-sitter）在这种场景下天然会把四份定义和所有调用点全部混在一起，
无法判断某个 `Class::getZqzdGroupKey` 调用点具体指向哪一份。

实测 `ContractArrearsYSRQPlugin::getZqzdGroupKey`（`RentArrearsManagementQSRQDown.java:156`，
作为 `Collectors.groupingBy(...)` 的分类函数）：
- 表达式级 `.resolve()` 直接失败（`UnsolvedSymbolException: We are unable to find the
  method declaration corresponding to ...`）——这也是 JavaParser 在"方法引用出现在带
  泛型推断的函数式接口上下文"里的已知局限，不是类路径问题。
- **兜底方案验证成功**：单独解析 `ref.getScope()` 的类型（即 `ContractArrearsYSRQPlugin`
  这个 `TypeExpr`）精确落到 `cqkd.am.assets.task.contractinfo.ContractArrearsYSRQPlugin`
  （import 路径 = 源码包路径，唯一 not ambiguous），再在这个已解析类型上按名字找候选
  方法——命中且只有 1 个候选（`getZqzdGroupKey(DynamicObject)`），完全消歧。

**设计结论**：符号解析要做两层——① 表达式级 `MethodCallExpr`/`MethodReferenceExpr`
`.resolve()` 优先（多数场景直接给出精确 FQN+签名）；② 失败时退化到"先解析 scope/限定符
的类型，再在该已确定类型上按名字+参数个数找候选方法"这一层兜底（仍然是确定性的类型
绑定，不是 Grep 式的全项目名字匹配），两层叠加基本覆盖了 JavaParser 自身的推断局限。

## §协议规格 v1（12.1 已实现，12.2 注入管线的施工图）

> 实现：Python 侧 `cosmic_kb/java/symbols/`（classpath.py / runner.py / table.py）；
> JVM 侧 `tools/symsolver/`（构建产物随包分发于 `cosmic_kb/java/vendor/symsolver.jar`，
> 开发者改动后跑 `scripts/build_symsolver.ps1` 重建，使用者只需本机有 java 8+ 运行时）。
> 一次批量不常驻：`build` 是一次性批处理，阶段 11 增量重扫复用本协议即可（热盘重建
> solver 仅 3–5 秒）；解析结果缓存不做（KB 即缓存）。

### 请求（stdin，单 JSON）

```json
{
  "protocol": 1,
  "source_roots": ["D:/proj/am_bd/src/main/java", "..."],
  "jar_dirs":    [{"path": "D:/.../cosmic", "recursive": true}, ...],
  "jar_files":   ["D:/.../single/one.jar", ...],
  "files":       [{"path": "<绝对路径>", "relpath": "<相对源码根POSIX路径>", "encoding": "GBK"}],
  "options":     {}
}
```

- **编码由 Python 侧传**：`ingest/scanner.py` 的探测结果是单一真源，runner 用
  `to_java_charset()` 把 Python codec 名翻成 Java Charset 名（`utf-8-sig`→`UTF-8` 等）；
  Java 按给定 charset 解码（REPLACE 容错），charset 名不认识退 UTF-8 并在 file 事件标
  `encoding_fallback: true`；解码后剥 BOM（U+FEFF）。
- `jar_dirs.recursive`：IDEA `jarDirectory` 的 recursive 属性照传；Gradle 模板
  fileTree(include:'*.jar') 是平铺语义 → false。
- `relpath` 是全协议的文件主键（响应按它回，SymbolTable 按它索引）。

### 响应（stdout，JSONL 流式；stderr 只放人读日志）

事件时序：`start` → `solver_ready` → 逐文件 `file`（穿插 `progress`/`warning`）→ `summary`。

| 事件 | 字段 | 说明 |
|---|---|---|
| `start` | `protocol` | 建 solver **前**发出；版本不符 Python 侧立即击杀拒收 |
| `solver_ready` | `jar_count` / `jar_failed` / `elapsed_ms` | 单 jar 损坏跳过计入 jar_failed，不崩 |
| `file` | `relpath` / `status`(ok\|parse-error\|io-error\|internal-error) / `note?` / `encoding_fallback?` / `sites[]` | 每文件一条，**发出即有效**（流式即恢复边界） |
| `progress` | `done` / `total` | 每 25 文件一条 |
| `warning` | `message` | 进 SymbolTable.warnings |
| `summary` | `files` / `sites` / `by_resolution` / `elapsed_ms` | 收到即 status=ok |

### site 对象（file.sites[]）

| 字段 | 说明 |
|---|---|
| `line` / `col` | 方法名标识符起点，1-based **字符列**（非字节列；12.2 与 tree-sitter 字节列换算对齐时以此为准） |
| `name` | 方法简单名 |
| `kind` | `invocation` \| `method_reference`（后者含构造器引用 `X::new`，`argc` 无） |
| `argc` | 实参个数；method_reference 缺省（Python 侧 None） |
| `resolution` | `expr`（表达式级 resolve 直接成功）\| `scope`（scope 类型兜底唯一命中，仍是确定性类型绑定）\| `failed` |
| `declaring` / `signature` / `static` | 解析成功时必有：目标方法声明类 FQN / 全限定签名 / 是否静态 |
| `target_kind` | `project`（源码 TypeSolver）\| `jar`（JarTypeSolver/javassist）\| `jdk`（Reflection）——按解析结果的实现类归类 |
| `reason` | failed 时必有：`unsolved-symbol` \| `ambiguous` \| `generic-inference` \| `parse-error` \| `io-error` \| `internal` |

### 两层解析（JVM 侧 Resolver）

① `MethodCallExpr/MethodReferenceExpr.resolve()` 优先 → `expr`；
② 失败退化：解 scope 类型（无 scope 的裸调用用所在类兜底，含祖先 `getAllMethods`），
按 名字+参数个数（方法引用不按 argc、不筛 static——`Class::instanceMethod` 合法）筛候选：
唯一→`scope`，多→`failed/ambiguous`，零→`failed/unsolved-symbol`。
TypeSolver 顺序：Reflection(jdkOnly) → 各源码根 JavaParserTypeSolver（项目源码优先，
防旧构建产物 jar 遮蔽）→ 各 jar。每调用点/每文件 try/catch（含 StackOverflowError）。

### Python 侧容错语义（runner.RunOutcome，永不上抛）

| status | 含义 |
|---|---|
| `ok` | 收到 summary（退出码非零仅记 warning） |
| `partial` | 收到部分 file 事件后进程死亡/看门狗击杀——已收部分照常可用 |
| `unavailable` | 找不到 java（COSMIC_KB_JAVA → JAVA_HOME → PATH，`java -version` 10s 验活缓存）或随包 jar 缺失 |
| `failed` | 进程跑了但零 file 事件（协议不符/启动即崩/请求写入失败） |

看门狗两段阈值：start→solver_ready **600s**（jar farm 首读"冷盘税"实测可达 140s+，
热盘 3–5s），之后事件间隔 **120s** 静默即击杀。非 JSON 输出行忽略并记 warning
（微工具已把 System.out 重定向 stderr 防第三方库污染，双保险）。

### SymbolTable 对齐口径（12.2 消费）

主键 `(relpath, line, name)`；同行同名多调用用 `col` 消歧；对不齐走
「唯一性兜底 → 否则 None」诚实路径，绝不冒认。`stats()` 出 files/sites/coverage/
by_resolution/by_failure_reason，进 build stdout 与 kb_meta（`symbol_resolution` JSON，
含 classpath attempts 诊断轨迹——"为什么没有符号层"必须能从 KB 里读出来）。
