# 字段名核对工具方案（`resolve_fields`）+ 分录层级识别

> 本文是**方案设计 + 待办**，本次（2026-06-25）只记录、不实现，下次按本文继续做。
> 起因：段二大模型读 Java 源码时，靠**命名惯例猜字段中文名**翻车——`cqkd_zjjnqk`
> 被猜成"资金缴纳情况"，真实是"租金缴纳情况"（一字之差）。整轮解释没调过任何 cosmic_kb
> 取证工具核对字段标识。
> 目标：让大模型读代码时**顺手、低成本地拿真实元数据字段名比对**，把"按命名惯例臆断"
> 从默认行为里挤掉。

---

## 1. 根因诊断：两件事同时缺位

**① 没有"便宜"的字段名查询工具。**
现有能查字段名的工具全是"重"的：`trace`（全量读写追溯）、`bill`（整单钻取）、
`ask`（NL→意图）。模型读源码碰到 `cqkd_zjjnqk`，**不会**为确认一个中文名去调 `trace`——
payload 大、语义是"谁改了它"而非"它叫什么"。于是走阻力最小的路：按命名惯例猜。
**缺一个 O(1)、专做"标识→真实中文名"的轻量工具。**

**② MCP `INSTRUCTIONS` 里没有那条硬反射。**
`cosmic_kb/mcp/server.py:22` 的 instructions 说了"绝不臆造字段名"，但**臆造 ≠ 按命名惯例
推断**——模型主观上把"租金/资金"当成"识别"而非"臆造"。缺一条明确纪律：凡引用
`<isv>_xxx` 并要陈述其中文名，必须先比对元数据，命名惯例不算证据。

**关键**：词典层 `Lexicon.fields_by_key()`（`cosmic_kb/semantic/dictionary.py:195`）
**已经有这份数据**，只是没暴露成 MCP 工具。

---

## 2. 方案（按杠杆从高到低）

### 2.1 核心：新增 `resolve_fields(keys: [...])` MCP/CLI 工具

便宜到模型愿意"读一段代码就顺手批量核一次"。直接打词典，回最小包，钉不出标 `null`
（诚实留白，符合红线）。草图：

```python
def tool_resolve_fields(keys: list[str]) -> dict[str, Any]:
    """字段标识 → 真实元数据中文名+实体坐标（比对元数据、防命名惯例臆断）。
    批量传 key，回 {key: [{name, form_key, entity_key, level, kind}] | null}。
    专治『读源码顺手核字段名』：比 trace 便宜得多，钉不出回 null（不臆造）。"""
    conn = _open()
    try:
        lex = build_lexicon(conn)
        return {k: [dataclasses.asdict(f) for f in lex.fields_by_key(k)] or None
                for k in keys}
    finally:
        conn.close()
```

> ⚠️ 见 §3：这版草图**只覆盖 `field` 表**，分录容器 key 会漏，必须补 `entity` 表。

### 2.2 配套：`INSTRUCTIONS` 加一条强制纪律

加到 `server.py` 的 `INSTRUCTIONS`，例如：

> 【字段名纪律】凡在源码中引用 `<isv>_前缀` 字段标识并要陈述其中文名/业务含义，
> **必须先 `resolve_fields` 批量比对元数据**——命名惯例（`zjjnqk`→租金还是资金？）
> 不算证据，是幻觉高发区。resolve 回 null 的标 unknown，不猜。

两条合起来才管用：工具让"核对"成本趋零，纪律把"核对"变成读码时的默认反射。
光有纪律没便宜工具→嫌贵照猜；光有工具没纪律→想不起来调。

### 2.3 可选增强：`Read` 钩子自动提醒

要做到字面意义"读代码时**同时**触发"：挂 `PostToolUse` 钩子在 `Read` 上，正则扫文件
内容里的 `\b[a-z]+_[a-z0-9_]+\b` 字段 token，命中就回一句 `additionalContext`：
"本文件含字段标识 X、Y，陈述其含义前请 `resolve_fields` 比对"。钩子不能替模型调工具，
但能把提醒推进上下文，压低"想不起来"的概率。配 `settings.json`。**第二步，可选。**

## 3. 分录层级识别：分两个层面，结论不同（本次重点澄清）

KB 字段建模（`cosmic_kb/graph/schema.sql`）：
- `field` 表（:52）每条带 `entity_key` + `level`(header/entry/subentry) + `name`——
  **分录里的字段是带实体坐标存的**，同 key 在表头/分录A/分录B 各一行，名字各自独立。
- `entity` 表（:42）存**分录容器本身**（分录实体 key，如 `cqkd_zdfl`「退租后账单」），
  带 `level` + `parent_key`。
- `field_access` 表（:140）带 `level` + `entry_key`，记录每次代码访问的实体坐标。

### 结论表

| 层面 | 分录字段能否正确识别 | 说明 |
|------|-------------------|------|
| 分录内**字段名**解析 | ✅ 准确 | `field` 表带 entity_key+level+name，resolve 原样返回全部坐标 |
| 分录**容器 key** 解析 | ⚠️ 需补 | 草图只打 `field` 表，分录容器 key 在 `entity` 表，会漏 |
| 同 key 跨多分录消歧 | ✅ 给全坐标 / ⚠️ 靠上下文 | 工具诚实返回 list 不替选，模型需用实体上下文选对 |
| 代码访问绑定到精确分录 | ◐ 部分 | 溯到来源实体→精确钉分录；溯不到→落「层级/分录存疑」桶（诚实留白） |

### 三个分录特有的坑（实现时必须处理）

**坑 1 —— 同 key 跨多分录，resolve 返回 list 不替选。**
分录字段常出现"一个 `field_key` 在多个分录都有定义、名字还可能不同"。工具诚实摆出
所有坐标（红线哲学），**消歧靠模型读代码时的实体上下文**——它得知道当前
`row.get("cqkd_xxx")` 的 `row` 是从哪个 `getDynamicObjectCollection(...)` 来的，才能对上
正确那条。模型必须会用，否则分录场景仍可能对错坐标。

**坑 2 —— 分录容器标识不在 `field` 表，在 `entity` 表。**（§2.1 草图的硬伤）
模型读到 `getDynamicObjectCollection("cqkd_zdfl")`——这是**分录 key**不是字段 key——
草图的 `fields_by_key` 解析不出。**要让分录识别闭环，`resolve_fields` 必须同时查
`entity` 表**，回 `{kind: "entry"/"subentry", name, parent_key}`。

**坑 3 —— 代码访问的分录归桶不保证 100% 精确（设计如此）。**
`field_access` 溯不到 DynamicObject 来源分录时记 `level=unknown / entry_key=None`，落进
`trace` 的「可能命中（层级/分录存疑）」桶（`field_trace.py:155`）。这是红线·信任优先的
诚实留白，不是 bug——宁可标"存疑"也不硬钉错分录。`resolve_fields` 本身不受影响
（它查的是元数据定义侧 `field`/`entity`，不是代码访问侧 `field_access`）。

## 5. 设计红线对照（自检）

- **不臆造**：钉不出回 `null`/`unknown`，不猜（红线·处处 unknown）。
- **KB 是契约**：纯读 `field`/`entity` 表，零 schema 改动，CLI/MCP 同口径（红线 #6）。
- **复用不重写**：打现成词典层 `Lexicon`，不新造解析（红线·资产复用）。
- **本地优先**：纯本机查 KB，无外发（红线 #1）。
