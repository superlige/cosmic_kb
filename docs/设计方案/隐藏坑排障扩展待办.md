### 2. 非插件入口点识别

- **隐藏坑**："字段半夜被改了"——源头常是**定时任务（AbstractTask）、事件订阅、
  WebApi/OpenAPI 入口、消息消费者**，不绑在任何单据的插件配置上，设计器查无此人。
- **现状**：跨类调用回溯（`java/project_graph.py`）从插件事件方法出发；这几类入口
  类中的字段写入即使被扫到，也缺"从什么入口触发"的归属标注。
- **方案**：按纯静态特征（继承基类/实现接口/注解）识别入口点类型，给类打
  `entry_kind = task / event_subscriber / webapi / mq_consumer` 标签入 KB；
  调用回溯把这些入口方法也纳入出发点，写入证据附"触发入口"维度。
- **产出**：`trace` 证据里非插件来源的写入点标明入口类型（如「定时任务」），
  不再是无归属的孤立写入。
- **验收要点**：构造/真实的定时任务改字段样例，trace 能标出入口类型；
  普通 service 类不误标。

### 3. 同字段覆盖冲突检测

- **隐藏坑**："我的插件明明 set 了，怎么没生效"——同一事件车道上另一个插件在
  后面又写了同一字段，后写覆盖先写。执行顺序取决于元数据里的插件注册顺序，
  肉眼比对多个插件源码几乎不可能发现。
- **现状**：`trace` 已按坐标分组列出所有写入点；KB 已存元数据插件注册信息。
  缺的只是一层分析：同单据 + 同事件 + 同字段的多写入点没有被标记为冲突。
- **方案**：纯 KB 增量分析层，不动扫描器——对 trace 结果做「同单据+同事件+同字段
  多写入点」聚合，标 `conflict_candidate`，附元数据注册顺序（能取到则给顺序，
  取不到标 unknown）。
- **产出**：`trace` 返回里冲突候选写入点带显式标记 + 顺序线索，agent 可直接
  回答"你的 set 被谁覆盖了"。
- **验收要点**：两个插件同事件写同字段的样例被标记；不同事件/不同字段不误报。

### 4. 时序反模式风险标注

- **隐藏坑**："改了没生效"的另一半：set 的**时机**不对（如在会被平台重算/重载
  覆盖的事件里写入）。代码本身看一百遍都没错，只有懂苍穹事件时序才知道是坑。
- **现状**：`semantics/` 随包权威文档里已有事件时机与反模式知识，但目前是
  **被动查询**——agent 主动调 `cosmic_semantics` 才看得到，排障时容易漏问。
- **方案**：把反模式知识里"事件 × 写入"类规则整理成确定性规则表（写入发生在
  事件 X → 附风险提示 Y），`trace` 组装证据时按写入点所在事件查表**主动标注**。
  只做确定性规则命中，不做推断；规则不命中不标。
- **产出**：`trace` 写入点证据附 `timing_risk` 风险提示（含指向 semantics 文档的
  出处），高频"没生效"问句一次命中。
- **验收要点**：已知反模式时机的写入样例被标注且提示语出自权威文档；
  正常时机写入不标。

### 5. 错误文案反查

- **隐藏坑**：用户手里只有一句报错弹窗文案。文案常由拼接/`MessageFormat` 占位符
  生成，普通 grep 搜不全，野生 GBK 编码再干扰一层——定位抛错点是纯体力活。
- **现状**：KB 已有 FTS5 全文索引；源码字符串字面量在扫描时可及，但没有
  面向"报错文案 → 抛错位置"的检索入口。
- **方案**：新取证工具 `whence "文案片段"`（CLI + MCP）：在源码字符串字面量
  （建库时入 FTS5，含占位符归一化，如 `{0}`/`%s` 打洞后模糊匹配）+ 元数据
  校验器提示/操作确认文案里检索，返回命中位置（类/方法/行号 或 元数据坐标）
  按匹配度排序。
- **产出**：报错文案一步定位到抛错的插件行号或元数据校验器，替代全局搜索。
- **验收要点**：带占位符的拼接文案能命中；GBK 源文件里的中文文案能命中；
  多处命中按匹配度排序全摆出，不替用户挑。
