# 苍穹开发排障 Agent 规划（讨论稿 v2）

> 2026-07-20 首次讨论沉淀；2026-07-22 大改。目标：在 cosmic_kb（排障取证引擎）之上做一个
> **专门的苍穹开发+排障 agent**，第一阶段只做排障与提测质量，第二阶段整合金蝶官方 vibecoding
> 开发组件形成"排障→开发→提测"闭环。本文是方向规划，未立项、未排期；与
> `docs/核心/开发计划.md`（cosmic_kb 本体蓝图）互不替代。
>
> **本版核心修订（对上一版最重要的三条，先读这段再读全文）**：
> 1. **换轴**：封装 agent 的价值不是"预制提示词/skill/MCP"，而是**把本该确定性完成的决策从模型手里拿走**
>    （§4.1）。四条模糊卖点收敛为一条可判定的边界线。
> 2. **边界线（本版灵魂）**：**入口（自然语言 → 该调哪个工具 / 走哪条车道）归模型；出口（工具跑完后：
>    证据完整吗、结构不变量守住吗、盲区标了吗）归 harness**（§4.3）。回路里有"读懂自然语言"的地方
>    交给模型，回路里只做结构化检查、没有 NL 的地方才轮到 harness。
> 3. **一条死路立碑**：**"分诊"（把顾问那句"某动作出错"静态路由到 op/form 车道）= 已退役的阶段 9 `ask`
>    的重生，不要再提**（§4.5）。NL→意图静态识别是 ask 栽的地方，宿主大模型比关键词分类器更擅长选工具。

## 1. 背景与动机

cosmic_kb 现状是"引擎"形态：MCP 五工具 + Skill 挂在用户自备的通用宿主（Claude Code / Qoder / …）上，
前提是用户**自己有 agent、会配 MCP、会提问**。这个前提在真实目标用户面前不成立（见 §3）。

同时，金蝶官方推出了 vibecoding 开发组件（见 §2），只管"从需求生成新代码"，
对存量老项目是半盲的——不懂项目里的调用链、字段读写、隐藏触发链。
一读一写恰好互补，具备做成闭环的条件。

## 2. 官方 vibecoding 组件调研结论

来源：`http://101.132.39.238/install`（安装部署手册-2.0 + 用户指引，2026-07 抓取）。

**关键发现：官方组件不是 agent，而是一套 Claude Code 格式的 Skill 包**（官方也选择寄生在现有宿主上）：

- **客户端侧**：`cosmic-dev` skills 目录（含 `cosmic-meta-modeling` 建模 / `cosmic-meta-modifier` 改元数据 /
  `pdm-automation` / `cosmic-gen` 插件代码生成 四个技能 + Python 脚本 + `config.json`），
  装到 Claude Code / Qoder / Kiro 的 skills 目录，`/cosmic-dev` 触发。
  官方原话"技能文件与 Claude Code 完全相同，无需额外适配"。
- **服务端侧**：开发环境苍穹需部署 `bos-designer-ai` jar + 开 `metadata.debug` 调试接口 +
  导入 OpenAPI（`ai-meta-openapi.dts`）并注册第三方应用；技能经 OpenAPI 在线拉元数据、调 `buildMeta` 写回苍穹。
- 与 cosmic_kb 的关系：**方向相反**——cosmic_kb 做"读"（排障取证），官方组件做"写"（建模/生成代码/写回元数据）。

## 3. 必要性：目标用户画像拍板了这件事

**用户画像（已确认）**：分公司开发本部正职 15 人（半数 30+，对 AI 开发范式理解不深）；
大量项目由外包伙伴承接（老项目运维 + 新项目分包/整包），伙伴**平均水平极低、流动性巨大**（多为应届专科）。
典型案例：资产系统由多个外包团队堆砌成巨型遗留系统。

**工作流现状（已确认）**：项目制，一/几个本部开发带多个伙伴；新伙伴不懂平台/项目规则直接问本部；
实施顾问口头/在线表登记问题 → 开发排障或开发新功能 → 通知实施测试 → 上生产；
实施时间极紧张，测试高度依赖开发自测，开发水平参差 → **线上问题频发、反复修改**。

**这个画像意味着**：

1. **培训投入归零的用户群**：流动性大 → 任何依赖"用户会用"的价值都无效，价值必须全部固化在工具侧。
2. **KB 是唯一的机构记忆**：知识只在已离职者脑子里 → `/接手` 项目地图从加分项变杀手级场景。
3. **agent 顺带是外包治理工具**：结论带证据行号 + 报告格式统一 → 伙伴的排障/改动汇报从口头变成本部可审计的交付物。

**这个画像也解释了为什么价值必须在"减法"上**：模型供给是弱国产模型、用户是低水平且流动的伙伴，
所以"模型不配合 / 伙伴不会补救"是**默认情况**，不是边缘情况。凡是靠"模型听话 + 伙伴会用"才成立的价值，
在这个画像下天然打折；凡是"不依赖模型行为、不依赖用户水平"的确定性约束，价值反而被放大。这是 §4 全部论证的地基。

**诚实的反面**：若目标用户只是自己 + 少数同水平同事，现有 MCP+Skill 路线已够，不必做 agent。
本规划成立的前提就是上面的画像。

## 4. 核心论证：封装 agent 的实质增量到底在哪

> 这一节是全文的重心，也是相对 v1 改动最大的地方。v1 用四条并列卖点（SOP 固化 / 门禁 / KB 生命周期 /
> 受众扩大）论证，被判"说服力不够、像四条各自为战的话术"。v2 把它收敛成**一条可判定的边界线**，
> 并砍掉一条混进来的死路。

### 4.1 换轴：不是"给模型加能力"，是"从模型手里拿走决策"

prompt / skill / MCP 三者的共同上限是——**它们只能给模型"加"东西**（加指令、加工具、加知识），
然后**由模型决定怎么用**。它们全是建议性的。

封装 agent（拥有 harness / 掌握 agent loop 与工具执行层）唯一的结构性增量是**反方向**的动作：
**把本该由代码确定性完成的那部分决策，从模型的自由裁量里拿走**，只把"理解语义 / 念成人话"留给模型。

门禁（工具执行前 `block`）只是这个动作里**最显眼的一种**——拿走"要不要放行"这个决策。
但它远不是全部，也不是最值钱的那部分。抓住"拿走决策"这个轴，能拿走的东西是一整片（§4.4）。

**对本项目画像（§3）**：价值几乎**全在减法**（拿走裁量），不在加法（加能力）——因为加能力那部分
（更多工具、更好提示词）通用宿主 + MCP 本来就能给。

### 4.2 一把尺子：模型完全不配合时，这个能力还成立吗

判断一个卖点是"真 harness 增量"还是"只是把提示词写好了帮你省事"，用这把尺子：

> **设想模型完全不配合**——不调工具、查到一个 writer 就停、忘标 unknown、按拼音瞎猜字段中文名。
> 这个能力**还成立吗**？
> - **成立** → 真 harness 增量（因为它不依赖模型行为）。
> - **不成立、要靠"模型听话 + 好提示词"** → 那是 prompt/skill 就能干的，封装 agent 在这里只是省事，
>   撑不起立项。

对本项目：如 §3 所说，"模型不配合"是默认情况，所以**过尺子的能力，价值不是打折而是放大**。

### 4.3 关键边界：入口归模型，出口归 harness（这条是 v2 的灵魂）

**来自一个已经付过学费的教训**：阶段 9 `ask`（NL → 意图 → 确定性证据包）已整体退役。退役原因是
**自然语言 → 意图 → 该调哪个工具**是多对多的模糊映射，**静态识别太难**，宿主大模型反而比关键词分类器
更擅长选工具，所以把选工具的权力**还给了模型**，KB 收缩回纯确定性取证。

这个教训划出的**不是**"harness 帮不上排障"，而是一条**很精确的分界**：

> **凡是回路里有"读懂自然语言"这一步的 → 交给模型（`ask` 的坟）；**
> **凡是回路里只在结构化数据上做检查、根本没有 NL 的 → 才轮到 harness。**

**为什么这条线是硬的**：`ask` 的失败模式（NL 太模糊、静态抓不住）**只在有 NL 的地方复发**。而 harness
真正该接管的那些检查——writers 与入口链的差集（KB 行的集合运算）、callers 的 verdict（entry_analysis
的枚举值）、bill 覆盖边界（"哪几个 plugin_type 结构上进得了 bill"的静态事实）——**回路里压根没有
自然语言**，是对**工具已经跑出来的结构化结果**做检查。ask 的病在这里**结构上不可能复发**。

**一句话定位**：

| 环节 | 内容 | 归谁 | 依据 |
|---|---|---|---|
| **入口** | 自然语言 → 该调哪个工具 / 走哪条车道 / 这是哪类问题 | **模型** | ask 已证明 harness 在这里没有优势 |
| **出口** | 工具跑完后：证据完整吗？该车道的结构不变量守住了吗？盲区标了吗？ | **harness** | 回路里没有 NL，全是结构化数据运算 |

**注意门禁天然就在出口/执行侧**——"未经 resolve 核对的中文名不放行"拦的是"是否存在 resolve 记录"这个
结构化事实，发生在工具执行层，没有 NL。所以门禁能活、分诊会死，不是偶然：**它们分别在这条线的两侧**。

### 4.4 落在这条线"出口侧"的增量清单（全部过 §4.2 的尺子）

| # | 增量（都是出口/执行侧动作，回路里无 NL） | 苍穹场景 | 过尺子 |
|---|---|---|---|
| 1 | **出口门禁** | 未 `resolve` 核对的字段中文名不允许标"已确认"；改公共方法没查过 `callers` 不放行；模型想跑 mvn/gradle 构建直接禁止（守红线 #2） | 模型想把猜测包装成结论，门禁不放行 ✅ |
| 2 | **定论完整性** | writers 与入口链的差集**必须清空或被显式解释**才算排障完成，不让模型查到一个 writer 就收工；`callers` 的 verdict + 符号覆盖率达标才允许输出"可改/可删" | 模型想在证据不全时下定论，出口条件拦住 ✅ |
| 3 | **覆盖边界常开盖章** | `bill` 每次输出**无条件**标注"本视图仅含单据绑定插件（form/list/op/writeback/convert）；task/webapi/workflow 见孤儿目录，可能不全"——落地红线 #4，防"这单干净"的假象 | 模型想把盲区渲染成"干净"，盖章已在 ✅ |
| 4 | **确定性后处理注入** | 每个字段结果自动注入真实中文名；每份回答头部盖"KB 代码分析落后 N 文件，截至 XX:XX" | 模型忘标新鲜度/中文名，harness 已盖章 ✅ |
| 5 | **确定性预检索 + 上下文塑形** | 大遗留系统上，模型锚定单据/字段坐标后，harness 把该次排障需要的证据（该单 schema、命中 field_access、相关 callers 子图）确定性预取进上下文，逐出无关项——把模型推理面从"整个系统"收缩到"一张单据的证据包" | 模型不主动探索，harness 已把该看的喂到面前；**这是让弱模型在大遗留系统上可用的关键杠杆** ✅ |
| 6 | **确定性兜底 / 自愈** | `resolve` 返回 null → 自动做一次字典同步 → 重查 → 仍无才拦（与门禁同时上线，见 §7.3）；`callers` 返回 0 → 自动挂符号覆盖率 caveat，不让模型欢天喜地说"能删" | 模型不会自救，harness 替它救 ✅ |
| 7 | **KB 生命周期自动化** | 查询时按需检查新鲜度并分层刷新（§7.3，前置依赖阶段 11 增量重扫）；用户心智里不再存在"KB"概念 | 模型/用户忘了重建，harness 按水位线自动处理 ✅ |
| 8 | **会话即审计物** | 强制输出契约：事实/推测/unknown 三段分离，每个中文名带"已核对/未核对"位，每步调了哪个工具、命中哪行源码落成不可篡改的会话记录 → 本部可抽查、可回溯、能喂进 `/提测检查` | 伙伴想把"我猜的"写得跟"我查证的"一样圆，schema 里的位抹不掉 ✅ |

> ⚠️ 关于 #5 的诚实边界：**"锚定哪张单据/哪个字段"如果来自顾问的自然语言，那一步仍是模型的（入口）**；
> harness 只在模型锚定**之后**做确定性预取。别把预检索误当成"从 NL 猜该看哪张单"——那又滑回分诊/ask。

### 4.5 一条死路的墓碑：分诊 = 阶段 9 `ask` 的重生，不要再提

> 立此碑，防止后续会话（包括未来的 Claude）再次"发明"这条已经付过学费的死路。

**什么是分诊**：把顾问那句自然语言"保存的时候报错 / 字段联动出问题"**静态路由**到
op 车道 / form 车道 / 该调哪个工具。参见 `docs/设计方案/下一步-操作级排障验收与钻取.md` 的两轴模型
与四手段——那两棵决策树**字面上是确定性的**，很容易让人以为 harness 可以接管。

**为什么是死路**：分诊的第一步是 **NL → 车道**，这**正是 `ask` 栽的地方**（NL→意图静态识别太难，
宿主大模型更擅长）。把它做进 harness，会死得和 ask 一模一样。

**正确做法**：**车道/工具选择留给模型**（入口归模型）。harness 只在**模型选定车道之后**，在出口强制该车道的
**结构不变量**——例如"只有 op 车道才有 operation_key"这类**对已知车道的领域规则约束**，以及 §4.4 的
覆盖边界盖章。这是对**已确定的结构**做渲染/校验（无 NL），**不是从 NL 猜车道**（有 NL）。两者差一个字，
命运差一整个阶段。

## 5. 工作流嵌入点与产品命令面

### 5.1 三个断点

| 断点 | 现状痛点 | agent 插入方式 |
|---|---|---|
| ① 新伙伴上手/答疑 | 习惯性问本部，本部当人肉文档，每换一批伙伴重耗一遍 | 平台规则走 `cosmic_semantics`；项目现状走 KB（`/接手`/trace/callers）；**项目约定规则做成每项目一份"项目规则 skill"**；问不出来的才升级到人 |
| ② 排障/开发 | 排障靠翻源码/切设计器/猜 | 命令产品面（§5.2）+ §4.4 的出口机制 |
| ③ 提测→上线 | 实施没时间测 → 依赖开发自测 → 水平参差 → 线上问题频发 | **`/提测检查`：自测清单不由开发凭自觉写，由 KB 确定性推导**（§5.3） |

### 5.2 命令是"产品外壳"，别把它误当成 harness 增量

`/接手`、`/字段排障`、`/影响面`、`/提测检查` 这些命令，**外壳部分**（把几个工具串成一个入口、统一格式）
本质是 prompt/skill 就能做的——它们的价值是**受众扩大 + 易用 + 一致性**（对 §3 画像很重要，但不是
结构性增量）。**真正的 harness 增量是命令背后那批出口机制**（§4.4：门禁、完整性、盖章、预检索）。
论证立项时，**卖点要落在出口机制上，不要落在"我们做了几个命令"上**——后者通用宿主自己也能配。

- `/接手` —— 项目地图/理解报告（KB 确定性派生的报告，属取证不属分诊；价值在受众/易用）。
- `/字段排障` —— resolve → trace → 读源码。**序列本身是外壳**（省事）；**"没有 resolve 记录不许标中文名已确认"
  才是 harness 增量**（门禁，§4.4#1）。
- `/影响面` —— callers → 入口可达性 → 敢不敢改。取证是外壳；**verdict + 覆盖率达标才放"可改"是 harness 增量**（§4.4#2）。
- 项目规则 skill 机制 —— 每项目一份，治人肉文档。

### 5.3 `/提测检查`（优先级最高，且稳稳在边界线的正确一侧）

对 git diff 改动的方法逐个 `callers` 反查全部调用点 → 回溯插件入口 → 反查绑定单据/操作 →
叠加 `trace` 字段读写面 → 确定性产出「影响面 + 建议回归测试点 + unknown 项」报告。

**为什么它是旗舰、又不违反 §4.3 的线**：它的**输入是 git diff（结构化，不是自然语言）**，
管线是确定性编排，**回路里没有 NL 意图识别**——因此它是"确定性编排"而非"分诊"，稳在出口/结构化侧。
模型只负责最后把结果**念成人话**。

一份报告同时给三方：伙伴拿到不依赖水平的自测硬清单；实施把紧张时间花在精确到单据/操作的测试点上；
本部有了抽查审计入口。"改了 A 把 B 弄坏"类反复修改，大比例死在"不知道 A 还被谁用着"，正是 callers 主攻场景。
**该功能几乎不依赖模型能力**（影响面是 KB 静态推导），恰好避开国产弱模型最不可靠的部分；
ROI 也最好论证（线上问题次数是现成考核指标，前后对比即数据）。

对 cosmic_kb 的新要求：callers 目前是单方法查询，需要"按 git diff 批量取改动方法 → 聚合影响面"编排层。
先在 agent 侧用现有工具循环调用实现，验证有价值后再下沉为 KB 原生能力。

## 6. 技术路线

### 6.1 三条实现路线

| 路线 | 做法 | 工作量 | 结论 |
|---|---|---|---|
| A. 自研 agent loop | 自己写模型调用+工具循环+CLI | 数月级 | ❌ 重复造宿主轮子，维护黑洞 |
| B. Claude Agent SDK 专用 CLI | SDK 带 loop/MCP/Skill/权限，只写提示词+门禁+打包 | 周级出原型 | 备选；绑 Claude API，国产模型供给冲突 |
| C. 开源宿主"发行版" | 不造 agent，深度预置：插件+skills+MCP+agent 定义+安装包 | 小 | ✅ 推荐起步 |

### 6.2 二开方式决定升级成本（能否跟随上游更新的答案）

- **第一档·纯扩展点发行版（推荐）**：不改上游一行核心代码，定制全部放官方扩展机制
  （插件 hook / skills / MCP / agent 定义 / 系统提示词）。自己的仓库 = 上游版本 pin + 定制层 + 安装脚本。
  上游发新版 → 改版本号 + 回归验证，**上游新能力自动到手，零 merge 成本**。
- **第二档·薄 fork**：仅当扩展点不够时动核心，patch 控制在个位数，CI 自动 rebase 上游 release + 冒烟测试。
- **第三档·硬 fork**：❌ 不走。反面教材：Roo Code（Cline 的 fork，2026-05 已归档）；Qwen Code（须专人追上游 merge）。

> §4.4 的出口机制（门禁/完整性/盖章/后处理注入）**全部落在"工具调用前/后 hook"这一个扩展点上**——
> 这正是"第一档纯扩展点"能不能兑现的命门。宿主选型（§6.3）第一优先级就是看这个 hook 稳不稳。

### 6.3 宿主选型：OpenCode 与 pi 两个平级候选（2026-07 格局）

五个硬需求：① 接 cosmic_kb 五工具；② 二阶段整合官方 cosmic-dev skill；③ **出口门禁（工具调用拦截）**；
④ 打包分发；⑤ 国产模型。其中 ③ 是 §4 论证的命门，选型第一看它。

#### 候选一 · OpenCode

GitHub 最热开源编码 agent（18 万+ star，MIT，月活约 800 万）。

| 需求 | OpenCode 扩展点 | 契合度 |
|---|---|---|
| ① 接 cosmic_kb 五工具 | 原生 MCP | 零改造 |
| ② 整合 cosmic-dev | 原生 Agent Skills（SKILL.md） | 高，需实测（Claude plugin 打包格式可能要薄适配） |
| ③ 出口门禁 | V2 插件 API 命名空间 hook，工具调用层可拦截 | **关键增量**，不改核心即可硬拦截；但 V2 有过 V1→V2 breaking change 历史 |
| ④ 打包分发 | agent 定义/提示词/配置全文件化 | 一个仓库全装下 |
| ⑤ 国产模型 | 75+ provider，供应商无关 | GLM/DeepSeek/Qwen 可接 |

#### 候选二 · pi（`earendil-works/pi`，libGDX 作者 badlogic，MIT，OpenClaw 的底层 harness）

2026-07 调研结论——**在 ③ 门禁这条命门上比 OpenCode 更干净**：

| 需求 | pi 的情况 | 结论 |
|---|---|---|
| ③ 出口门禁 | **核心内置** `tool_call` 事件，`return { block: true, reason }` 直接拦截并阻断工具执行，`event.input` 可改写参数；另有 `tool_result` 事件可确定性改写结果（正好做 §4.4#4 后处理注入） | ✅ **一等公民、签名三行、核心可审计**，比 OpenCode 依赖 V2 插件 API 更稳、更贴"信任优先" |
| ⑤ 国产模型 | DeepSeek/Qwen/Kimi 原生 + OpenAI 兼容自定义端点 + 可指私有网关 | ✅ 完美命中 §8 合规约束（公司统一/私有化模型接入） |
| ① 接 cosmic_kb | **无原生 MCP**（Issue #563 才在提"做个 MCP 扩展"） | ⚠️ 唯一缺口，但**不是硬伤**：cosmic_kb 五工具**都是 CLI 命令**，MCP 只是外壳；pi 核心自带 Bash，官方推荐模式就是"CLI + skill/README"，直接挂即可，还更能精确控制暴露给模型的工具面（对门禁是加分） |
| ② 整合 cosmic-dev | 有 Agent Skills（SKILL.md 同源），同 OpenCode 需实测兼容 | ➖ 打平 |
| ④ 打包分发 | pi packages 打包扩展+skill+prompt；npm/安装脚本；有 windows.md | ➖ 略 DIY，可用 |

pi 的风险：生态成熟度低（约 11.5k star vs OpenCode 18 万）、实战检验少、**强单作者依赖**（bus-factor 高）。
好在 MIT + 极简（"whole core fits in your head"，可审计整条信任链）+ 可 fork，真出事迁移成本低。
它的"极简核心 + 扩展层"架构与 §6.2 第一档发行版策略贴合度**更高**。

#### 选型倾向

**对一个卖点是"信任优先 + 确定性取证 + 出口门禁"的项目，pi 用生态成熟度换极简和可审计性、且把门禁做成
核心原语，这笔交换划算**——建议把 pi 列为**平级候选甚至首选**，OpenCode 作为"要成熟生态时"的备选。
最终由 §9 spike 的门禁验证结果拍板。

已知共性风险：① 两者都迭代快 → **pin 版本 + 固定升级窗口（如每月），不追 latest**；
② 用 Claude 须 API key 计费（OpenCode 与 Anthropic 有纠纷不支持订阅登录；pi 走 BYOK）；
③ **最大风险不在工程在模型**：国产弱模型工具纪律差，门禁能拦但"改对"与否须真实案例实测（§9）。

## 7. 阶段划分

### 7.1 第一阶段：排障 + 提测质量（为闭环搭骨架）

四个命令 + 一批出口机制：

1. `/接手` —— 项目地图/理解报告（治新伙伴上手）
2. `/字段排障` —— resolve 核对 → trace 取证 → 读源码 → 统一格式证据报告
3. `/影响面` —— callers → 入口可达性 → 敢不敢改的结论
4. `/提测检查` —— **优先级最高**，见 §5.3
5. 项目规则 skill 机制 —— 每项目一份，治人肉文档

横切（**这批才是 §4 论证的 harness 增量本体，不是上面的命令外壳**）：
§4.4 的出口门禁 / 定论完整性 / 覆盖边界盖章 / 后处理注入 / 预检索塑形 / 兜底自愈 / KB 生命周期 / 会话审计物 +
统一安装包。

> 注意：单独看第一阶段相对现状增量偏薄，其定位是"为二阶段闭环搭骨架 + 用试点数据换立项弹药"，不是独立目标。

### 7.2 第二阶段：整合官方 vibecoding 组件（形成闭环）

`trace/callers` 查清现状 → 判定影响面 → `cosmic-gen` 改代码 → `enableCompile` 编译验证 → `/提测检查` 出报告。

融合的真正工作量不在"接入"（两边都是标准 skill/MCP 格式），在四件融合设计：

1. **元数据访问路径错位（非数据分叉）**：官方 `buildMeta` 写回最终落的就是 KB 直读的那批
   `t_meta_formdesign`/`t_meta_entitydesign` 表——数据本体不分叉，冲突体现在三层：
   - **时效性**：KB 是建库快照，官方 skill 在线拉 + `spec/` 缓存。`buildMeta` 加完字段的瞬间 KB 必然过期，
     `resolve_fields` 查不到新字段 → **过期 KB + 硬门禁 = 误拦合法操作，侵蚀门禁公信力**（伙伴学会绕过就全塌）。
     ⚠️ 现实约束（已确认）：十几人团队一天改几十个元数据+插件、随时有人在改，且别人在设计器里的改动
     agent 收不到任何事件 → **事件驱动"写了就推"不成立，采用 pull-based 查询时按需拉**（见 §7.3 分层新鲜度）。
     门禁误伤的精确兜底（= §4.4#6）：`resolve_fields` 查不到字段 → 自动做一次元数据字典同步 → 重查 →
     仍查不到才拦。**约束：该兜底必须与硬门禁同时上线，不能后补。**
   - **合并口径**：底层表存设计期原始 XML（扩展走 fmasterid 关联母体），扩展合并是 KB 自己实现的
     （`merge_vendor_extension` 父子并集）；官方 debug 接口大概率返回平台运行时合并视图。边界情况
     （字段禁用/删除/属性覆盖）两套逻辑可能答案不同 → 权威规则写死在系统提示词：**排障取证以 KB 为准
     （附建库时间戳）；生成代码的目标元数据以官方接口为准**，不让模型自己仲裁。二阶段前置验证：
     同一张扩展单据 KB 视图 vs debug 接口视图 diff 一次，量化口径差异。
   - **环境一致性**：KB 的 db-config 与官方 `config.json` 两套凭证各配各的，可能指向不同苍穹环境
     （最隐蔽的脚枪）→ bootstrap 强制校验两套配置指向同一环境（比对租户/数据中心），不一致拒绝启动。
   - 长期选项：KB 元数据源加"OpenAPI 适配器"（官方 debug 接口作可选第三源，统一读取口径、免 DBA 库权限），
     但不可完全替代——依赖官方服务端已部署，且 `t_botp_convertrule` 等表未必经接口暴露；DB 直读 + dym/zip 仍是兜底。
2. **门禁编排**：cosmic-gen 生成代码前强制过 `resolve_fields`（字段核对）+ `callers`（影响面）——融合的灵魂，纯拼装做不出来。
3. **信任边界分离**：排障侧维持只读；官方组件要连开发环境 OpenAPI **写回**元数据，"写"操作须单独确认机制，两套信任边界不混。
4. **一阶段不受影响**：纯排障只读、不碰官方组件，元数据路径错位仅在二阶段显现，且三层均有明确解法，不构成方案级障碍。

前置依赖：开发环境苍穹能否部署 `bos-designer-ai` jar + 开 `metadata.debug` + 注册第三方应用（需环境管理员配合）。
装不进去则二阶段降级为"排障 + 只出代码不写回元数据"。

### 7.3 分层新鲜度策略（应对高频改动的团队节奏）

**现实约束（已确认）**：项目上十几人开发+实施，一天可改几十个元数据+插件代码，随时有人在改；
"元数据一改 KB 就重建"负担不可接受，且别人的改动收不到事件。

**原则**：从"写事件驱动推"改为"查询时按需拉"（水位线 + TTL 节流）；新鲜度分层，各层成本与策略不同；
来不及刷新不阻塞回答，**在报告头部诚实标注过期警告**（= §4.4#4）——
新鲜度本身就是一种置信度，是"处处置信度 + unknown"哲学的自然延伸。

| 层 | 内容 | 刷新策略 | 成本量级 |
|---|---|---|---|
| 元数据字典层 | resolve_fields 用的字段/实体/单据名称字典 | 查询时按需拉（`t_meta_*` 修改时间水位线 + TTL 如 5 分钟）+ resolve 查不到时兜底同步重查 | 秒级，随便刷 |
| 桥接/语义层 | 元数据↔源码桥接、form_key 归属 | 跟随元数据同步或定时（每小时/每次会话开始） | 中，需实测量化 |
| Java 分析层 | tree-sitter + 符号解析 + call_edge | **增量重扫**（只重分析变更文件）+ 会话开始检查，过期过多提示后台重建 | 重，依赖阶段 11 |

**对 cosmic_kb 本体的硬结论：阶段 11（增量重扫）从"待开发"升级为本 agent 项目的前置依赖。**
没有增量重扫，团队高频改动节奏下 Java 层要么天天全量扫（不可接受）要么长期过期（结论不可信）。

待验证：① `t_meta_*` 是否有可靠修改时间列做水位线；② 元数据同步连带桥接层刷新的实际耗时——
若偏重则先只热更新字典层（治门禁误伤已够），分析层允许滞后并标注。

## 8. 硬约束与诚实边界

**画像带来的硬约束**：

1. **模型供给倒向国产 + 公司统一供给**（伙伴无 Claude 订阅、token 不能伙伴自付）。
   ⚠️ 待确认合规红线：源码经 API 出机器给模型厂商，公司/客户是否允许？公有云 API 还是必须公司自建/私有化模型？
   此答案影响整个架构。（cosmic_kb 红线#1 是"不发布到公网站点"，接大模型必然出机器，二者不冲突但须明示。）
   pi/OpenCode 都支持 OpenAI 兼容自定义端点 + 私有网关，无论合规最终落在哪一档都接得上。
2. **分发适应伙伴机器**：内网、受管控、装软件要审批 → 离线可装、免管理员权限、开箱即用；
   现有"对话式安装口令"对没有 agent 的伙伴不成立，须退回传统一键安装包。
3. **防工具被拿去糊弄**：门禁目标 = 没有核对证据不放行产出（§4.4#1）；unknown 项显眼——
   宁可伙伴回来问本部，不让 agent 把不确定的事说圆。

**agent 治不了的（立项时先讲明，防被将军）**：

- **入口选工具/理解意图的质量**：这是模型的活（§4.3），弱模型选错工具、读错业务语义，harness 拦不住，
  只能保证它没漏查、没瞎标、证据齐（出口）。**别承诺"agent 保证选对工具"**。
- 需求理解错误（实施口头登记 → 开发理解偏差）：沟通问题，最多靠结构化工单模板缓解。
- 运行时/数据问题（并发、脏数据、环境差异）：KB 是静态分析，不在覆盖面。
- 对外话术："消灭**静态可预防**的那一类线上问题"，不许诺降多少故障率。

## 9. 落地节奏

```
① 模型供给方案确认（合规：公有云 API vs 内部网关/私有化）
        ↓
② 宿主 spike（1–3 天，pi 与 OpenCode 各跑门禁验证，见下）
        ↓
③ 试点：资产系统 + 1–2 个在维伙伴，只放 /接手 /字段排障 /影响面（/提测检查 就绪则加入），跑一个月
        ↓
④ 看两个数：伙伴排障工单平均解决时间；伙伴结论被本部打回比例 → 决定全推 + 二阶段立项
```

**spike 验证清单（按 §4 的边界线校准，两个"决定性"项直接回答"做 agent 有没有意义"）**：

1. 宿主配 cosmic_kb（pi 走 CLI+skill，OpenCode 走 MCP），跑通真实 `trace` 排障问答；
2. **【决定性·出口门禁】** pi `tool_call` 的 `block` / OpenCode V2 hook：验证"未 resolve 核对的中文名不放行"
   "改方法没查 callers 不放行"能否**稳定拦住弱模型**——这直接验证 §4.4#1，是"专门 agent 比现状强在哪"能否兑现的命门；
3. **【决定性·/提测检查 ROI】** 用资产系统 **3–5 个真实引发过线上问题的历史改动**，手工走
   "diff → callers 聚合 → 测试点清单"，统计"若当时有此工具，N 次故障能圈住几次"——命中率数字就是立项弹药（§5.3）；
4. **【可用性·入口质量】** GLM / DeepSeek 各跑同一排障案例，看**入口选工具**的质量衰减（§4.3 入口归模型，
   这是弱模型可用性关键；注意：这一项衰减 harness 治不了，只能观测，决定是否需 §10#5 的 Claude 备选通道）；
5. 官方 `cosmic-dev` 技能目录直接丢入 pi/OpenCode，看兼容缺口（二阶段前置）。

> ⚠️ spike 清单里**没有"验证分诊状态机"这一项**——分诊已立碑作废（§4.5），不要在 spike 里复活它。
