# 扩展元数据（扩展二开）识别方案

> 本文是**方案设计 + 待办**，本次（2026-07-02）只诊断、记录，不实现，下次按本文继续做。
> 起因：用户读 `samples/bill/cqkd_bd_customer_ext.dym` 时发现，苍穹的**扩展元数据**
> （对原厂标准单据/基础资料做二开：加字段、加操作、加页面插件）在我们的工具里是**结构性半盲**。
> 目标：让工具**知道自己不知道**——识别"这是扩展单据"、记录它扩展了哪个原厂实体、把
> 插件读写的原厂字段从"扫不出"里诚实拆出来，绝不臆造，符合红线 #2/#4。

---

## 0. 背景：苍穹扩展元数据是什么

真实项目大量对原厂页面（如 `bd_customer` 客户、`bd_supplier` 供应商）做扩展开发：
新增扩展字段、新增操作、在页面上绑自研插件。**扩展元数据只包含扩展部分，不含原厂字段**；
插件却会读写原厂实体的标准字段（`name`/`number`/`org`/`status`/`createtime`…）。
运行时该扩展**挂在原厂实体上**（本例插件实际跑在 `bd_customer` 上），扩展单据本身
（`cqkd_bd_customer_ext`）只是一层"补丁"。

---

## 1. 实测：`cqkd_bd_customer_ext.dym` 现在解析出什么

`ModelType=BaseFormModel`，`Key=cqkd_bd_customer_ext`，`InheritPath` 链末端指向
`MasterId=a86c9c130002f7ac`。解析**不崩**，且拿到不少东西：

| 项 | 现状 |
|---|---|
| 扩展新增字段（`cqkd_*`，有 `fk_cqkd_*` 落库列） | ✅ 全部解析到（30 个 `entity` 字段） |
| 插件 / 操作（含项目插件） | ✅ 全部解析到（26 插件 + 5 操作） |
| 原厂字段本体（`name`/`number`/`org`…） | ❌ 本机 KB 里不存在（原厂 dym 不在导出包内） |
| 原厂字段"占位残根"（`action=edit oid=…`） | ⚠️ 仅 9 个、**无 Key**、`oid` 无法反解到标准 key |
| 插件读写原厂字段 | ❌ 字段级 trace 必然定位失败 → 落 `form_key=None`/误绑 |
| 扩展关系建模（是扩展、扩展了 `bd_customer`、运行时挂 `bd_customer`） | ❌ 完全没建，`trace` 两头对不上 |

原厂字段残根实测长这样（`cosmic_kb/metadata/dym_parser.py` 把它们分类成 `inherited`，`key=None`）：

```
MuliLangTextField  action=edit oid=8oq6R4m9CF   # 实为标准字段 name(客户名称)，但我们不知道
TextField          action=edit oid=h736cFwZ6e   # 实为标准字段 number(客户编码)
OrgField           action=edit oid=Nf3Ci3g19l   # 实为标准字段 org
ComboField         action=edit oid=BAWmwio8tC   # 实为标准字段 status/enable
...（共 9 个，均只带 oid，无 Key，多数无 Name）
```

---

## 2. 三个硬伤

**① 原厂字段在本地 KB 里根本不存在。**
扩展元数据只有 9 个原厂字段的"占位残根"，只带 `oid`、无 `Key`；`oid=8oq6R4m9CF → name`
的映射我们无从知道。连残根都没有的其它标准字段（`creator`/`createtime`/`modifier`…）
在 KB 里彻底缺席。

**② 插件读写原厂字段 → 字段级排障引擎必然定位失败。**
插件里 `getModel().setValue("name", …)` / 读 `number`，trace/resolve 拿 `"name"`
去 KB 查——查不到（KB 里只有 `cqkd_*`），落 `form_key=None` 或误绑。**这是结构性缺口，
不是识别算法问题**；之前降 NULL 率的努力在扩展单据上撞的就是这堵墙。

**③ 扩展关系本身没建模。**
存的 `key` 是 `cqkd_bd_customer_ext`，但插件运行时挂在 `bd_customer` 上。于是
`trace bd_customer.name` 找不到这些插件，`trace cqkd_bd_customer_ext.name` 找不到字段——
**两头都对不上**。既没标"这是扩展"，也没记"扩展了哪个原厂单据"。

---

## 3. 根因

- **一半是数据可得性**（非解析 bug）：原厂 `bd_customer` 完整元数据由平台内置，
  **通常不在 ISV 导出包里**。除非项目方也导出了原厂 dym，标准字段的 key/中文名本机就是没有——
  与红线 #2「代码是野生的、信息天生不完整」一致。
- **一半是缺概念**：我们从未把"扩展"建模成一等对象，也没利用 `InheritPath`/`MasterId`/
  `action=edit oid` 这些现成信号。

---

## 4. 待办方向（按红线内、从确定到理想）

1. **识别 + 标记扩展**（确定、低风险）
   信号：`InheritPath` 非空 + 存在 `action=edit oid=…` 残根字段（+ key 命名如 `*_ext`）。
   产物：`MetaModel.is_extension=True`，记录 `extends`（扩展的原厂实体标识/MasterId），
   让插件/操作能挂回运行时单据。需扩 schema（新字段）+ `dym_parser` 打标。

2. **原厂字段引用诚实标注**（确定、符合红线 #4）
   给"读写了本机无元数据的标准字段"一个专门 null_reason 码
   （建议 `standard-field-not-local` / `扩展·原厂字段·元数据不在本地`），
   在 `cosmic_kb/java/null_reason.py` 新增互斥码，暴露到 trace/coverage/web/MCP。
   把它从"扫不出/未识别"里拆出来，告诉段二/人「这行读的是原厂字段、不是识别失败、不必追」。

3. **trace 双向对齐**（中等）
   扩展与原厂实体建立映射后，`trace bd_customer.xxx` 与 `trace cqkd_bd_customer_ext.xxx`
   能互相找到对方的插件/字段（至少给出"该字段属原厂、由扩展 X 的插件读写"的提示）。

4. **原厂 dym merge 补全**（理想，依赖数据齐全）
   若导出包里**也**含原厂 `bd_customer.dym`，用 `oid`/`MasterId` 把扩展残根 merge 回原厂
   字段定义，标准字段的 key/中文名即可补全，`oid=8oq6R4m9CF → name` 打通。数据不齐时留 unknown。

   > ✅ **已被更强方案替代/落地（2026-07-02）**：直连平台**底层库**取元数据，比"碰运气看导出包里有没有
   > 原厂 dym"彻底得多——`t_meta_formdesign` / `t_meta_entitydesign` 两张设计表按 `fnumber` 各存一份
   > fdata（明文 XML），组合即拿回**全部**原厂/扩展字段。已实现只读 DB 元数据源 `cosmic_kb/dbmeta/`
   > （`assemble` 把两表 fdata 套回 dym 骨架 → 复用 `parse_element`，解析器零改动；`connection` 三重只读
   > 防线；CLI `cosmic_kb db-meta`）。实测 `bd_customer` 落库标准字段/操作/原厂插件全解析到。详见
   > `docs/阶段验收.md`「扩展元数据 · 底层库元数据源」。**下一步**：把 DB 取回的原厂元数据并入 KB，
   > 打通硬伤 ①②（标准字段 `form_key` + trace 双向对齐）。

---

## 5. 验收要点（实现时补测试）

- `parse_file(cqkd_bd_customer_ext.dym)` → `is_extension=True`，`extends` 指向原厂实体。
- 9 个 `action=edit oid` 残根被识别为"原厂字段覆盖"，不误当作扩展新增字段。
- 插件读写标准字段（如 `name`）在 trace 里落**专门 null_reason 码**，不混进普通"未识别"。
- 原厂 dym 缺席时全程不臆造、不崩；若提供原厂 dym，merge 后标准字段可核对出中文名。
- 样例文件：`samples/bill/cqkd_bd_customer_ext.dym`（本机已有）。
