# `read_source` 字段名取值逻辑现状（2026-07-05 梳理，供下一步决策参考）

> 本文**只梳理现状、不改代码**。起因：讨论"能否让大模型传入
> `实体名.分录名.字段名` 这种带层级的复合 key 来唯一锁定字段"时，发现需要先把
> `read_source` 现在到底怎么把一个裸标识（token）变成"中文名 + 坐标 + 置信档位"
> 讲清楚，才能判断这个提案的增量价值在哪、边界在哪。落成文档供反复对照，不占对话
> 上下文。涉及文件：`cosmic_kb/report/read_source.py`。

> `form_ctx` 文件级数据流收敛）随 MCP `read_source` 工具一并退役。用户复盘后判断
> "工具自动消歧"的复杂度收益比不划算（§5 量化已证实结构性漏判修复收益 <1.1% 且有

> CLI `cosmic_kb source`（人工终端排障）不受影响，本文档描述的三档逻辑在那条路径

## 1. 入口：`keys` 参数三态

`read_source(conn, relpath, *, keys=None)` 用一个参数区分两条完全不同的取证路径：

| `keys` 取值 | 使用场景 | 行为 |
|---|---|---|
| `None`（缺省） | CLI/Web 富投影 | **全文盲扫**：正文所有 token 与 KB 全量已知 key（`field`+`entity`+`form` 三表）取交集，交集里的每个 key 都走分类 |
| `[]`（显式空） | MCP 压缩投影首次调用 | 不扫描，`field_names` 直接为空，只给源码正文 |
| 显式非空列表 | MCP 压缩投影二次调用（模型自报陌生标识后） | 只核对列表里这些标识，形如 `Xxx.CONST_NAME` 的按限定常量引用处理，其余按普通 key 处理 |

无论走哪条路径，**最终喂进分类逻辑的都是一批裸字符串 key**（`scan_keys`），下游处理
完全一致——这一点是本文档要讲的重点：不管 key 从哪来，分类算法本身没有区别。

## 2. 单个 key 怎么被判定：`_classify_key`（`read_source.py:229`）

```
_classify_key(key, candidates, form_ctx)
```

- `candidates`：`resolve_fields.resolve_fields(conn, [key])` 查出的**该 key 在全库范围
  内的所有候选**——即元数据里所有 `key` 相同的字段/容器定义，可能来自多张不同单据。
  这是纯字符串匹配，不带任何上下文。
- `form_ctx`：**本文件**（`relpath`）的 `field_access` 解析结果，形如
  `{key: {form_key: {lines, src}}}`——是段一建库时**对这一个源文件**做数据流追踪
  （ORM `load`/事件入参绑定/跨实体传播）得到的"这个 key 在这个文件里实际访问的是
  哪个实体"。这是唯一带"归属证据"的输入。

判定逻辑三步：

1. **候选只落在 ≤1 个单据** → `unique`，直接照抄，不需要 `form_ctx` 介入。
2. **候选跨多个单据**，但 `form_ctx[key]` 里有该 key 在本文件解析到的实体
   → `resolved`：用 `form_ctx` 收敛到具体单据，坐标上标 `resolved_lines`（本文件命中
   行号）+ 依据来源（`data_flow` 数据流证明 / `metadata_*` 字段归属反查）。
3. **候选跨多个单据，且 `form_ctx` 没有该 key 的记录**（本文件数据流没追到）
   → `ambiguous`：`names` 留空，给出消歧方向提示（看接收变量来源），**不擅自选一个**。

**关键点**：第 2 步用来消歧的证据（`form_ctx`）来自**段一静态分析对整个文件做的数据流
追踪**，不是查询时才算的。也就是说——不管调用方是老版本"全文盲扫"传进来的 key，
还是新版本"模型自报"传进来的 key，能不能消歧只取决于**这个 key 在这个文件里的数据流
是否在建库时被追踪到**，与"谁发起了这次查询"无关。

## 3. "实体名.字段名复合 key" 提案在这套逻辑里处在什么位置

讨论中提出的思路：如果模型能从源码里读到实体/表单 key 字面量（比如
`.load("cqkd_contractbill", ...)`），把它和字段 key 拼成复合 key 传入，工具按
`(form_key, field_key)` 精确查库，天然唯一，不需要再靠 `form_ctx` 猜。

对照上面的流程看，这个提案等价于：**让调用方在查询时手动提供 `form_ctx` 本来要
提供的那份"归属证据"**。由此得到两个边界判断：

- **模型能看出实体归属的场景，`form_ctx` 通常已经看出来了**——因为两者用的是同一
  类证据（`.load(...)`/事件入参这类源码里的字面量数据流），我们的静态分析器在建库
  时就做过一遍，且能追踪跨方法调用链，比"模型临时读一遍代码"更系统。这部分场景
  现在已经落在 `resolved` 档，复合 key 提案在这里没有增量。
- **真正落在 `ambiguous` 档的**，是 `form_ctx` 追不到的场景（动态反射、深层间接
  调用、纯常量定义无直接访问）。这部分场景模型凭肉眼读代码大概率也看不出实体
  归属——不是分析器偷懒，是代码本身没把线索留在能看到的地方。复合 key 提案在这里
  能不能生效，取决于这批 ambiguous 样本里，"模型能看出但分析器看不出"的比例有多
  少——这是个经验问题，需要抽样看真实 ambiguous 样本才能回答，不能纯靠推理下结论。

**如果要做**，正确的落地方式不是取代三档逻辑，而是加一条并行通路：调用方可传
`"实体key.字段key"` 形式的复合 key，工具按 `(form_key, key)` 精确查库——查到即
`unique`（比 `resolved` 更强，不需要 `form_ctx` 介入）；查不到就诚实报错（"该实体下
无此字段"），不静默接受、不误导。**前提仍是模型传的必须是它从源码字面量直接读到的
实体 key，不能是猜测的中文名或臆断的标识**，否则问题从"字段名瞎猜"变成"实体名
瞎猜"，红线 #4 要防的东西原样复现，只是换了个位置。

## 4. 其余两条支线（与本次讨论关系不大，附带记录现状）

- **限定常量引用**（`Xxx.CONST_NAME`）：先查 `java_constant` 表把常量解回字面值，
  解出的字面值再走上面同一套 `_classify_key` 三档逻辑。字面值本身仍可能跨单据同名，
  一样受第 3 节的边界限制。
- **表单标识兜底**：字段/容器分类都钉不出时，再查 `form` 表看是不是表单标识
  本身（如 `.load("xxx")` 里的字面量）。表单没有 `resolved` 档——同 key 多义直接
  standing `ambiguous`，不靠 `field_access` 收敛（表单标识本身不是"字段访问"，
  `field_access` 不记录它）。

## 5. 抽样与量化结论（2026-07-05，真实库 `D:\kingdee\asset_management_sys\cosmic_kb.db`）

### 5.1 ambiguous 占比

对真实库 `field_access`（20,710 行）按 `_classify_key` 同口径复算：涉及"跨多单据
同名 key"的 `(文件, key)` 组合共 **7354** 个，其中：

| 档位 | 数量 | 占比 |
|---|---|---|
| resolved（本文件数据流已收敛） | 5125 | 69.7% |
| ambiguous（本文件数据流没追到） | 2229 | 30.3% |

### 5.2 抽样 20 条真实 ambiguous 样本，发现三类不同性质

1. **假性歧义**——key 本身是通用系统字段（`id`/`number`/`billstatus`），几乎每个
   实体都隐式带，元数据候选列表本身不构成有意义的业务归属问题。给复合 key 也无益，
   该识别的是"这压根不需要消歧"，不是"消歧不出来"。
2. **真性不可归**——报表/汇总类文件（如 `StatementAcreageReport.java`，单个 key
   候选高达 60 张单据）本来就在一段代码里横跨多张单据，不存在唯一归属，`ambiguous`
   本身就是正确答案。
3. **结构性漏判**——数据流理论上可推、但现有静态追踪器没覆盖到的模式，是唯一
   可能有增量的一类，往下两节量化了两个具体候选模式。

### 5.3 量化候选模式 ①：`new DynamicObject(局部变量)` 且变量来自
`X.getDataEntityType().getProperties().get(key).getItemType()` 链

现有 `field_access.py:59-60` 的 `_NEW_DO_OF_COLL_RE` 只认内联写法
`new DynamicObject(coll.getDynamicObjectType())`；真实样本
`PushSkFormPlugin.java:214-225` 是分几行、经 `getProperties().get("cqkd_sonbill")`
存局部变量再 `new DynamicObject(变量)` 的写法，属于确认存在但未覆盖的模式。

全项目扫描结果：**17 个站点、14 个文件**命中；交叉 `field_access` 后，这些文件里
`form_key IS NULL` 的行合计约 **53 条**（这是文件级上界，非精确到"确系此链导致"）。
相对全库 6903 条未定位 / 2229 条 ambiguous，**占比不到 1%**。

**结论：规模太小，不值得为此单独扩正则、加测试。**

### 5.4 量化候选模式 ②：常量类名 → 主导 `form_key` 推断

思路：若某常量类（如 `ContAttr`）里绝大多数常量的字面值都落在同一张单据的字段上，
可推断该类是该单据的"专属常量类"，作为消歧信号。

量化做法：对 `java_constant` 按 `class_name` 分组，要求可解析常量 ≥3 条、主导
form_key 占比 ≥80% 才算"有主导实体"——全库满足条件的常量类仅 **11 个**。用这 11
个类的映射去对 2229 条 ambiguous 样本："文件里出现了某个有主导实体的常量类，且该
实体在这个 key 的候选集里" → 记为可收敛。结果：**25 / 2229，1.1%**。

**且这个信号本身不安全**：命中最多的 `TransBillCon -> cqkd_zkd`，在
`ContractService.java`/`ReportQueryService.java` 这类横跨多单据的 service/report
文件里命中——"文件里提到过这个常量类"不能证明"文件里*这一条*歧义字段访问就属于
该类对应的实体"，文件级粒度的关联在报表/service 类里大概率产出**错误的**唯一
答案，比留 `ambiguous` 更糟（直接违反红线 #4：宁可 unknown 不可臆造）。

**结论：不仅规模小于 1.1%，还有引入误判的风险，不值得做。**

### 5.5 最终结论

两个"结构性漏判可能有增量"的候选方向，量化后均不成立，与
`docs/设计方案/数据包来源与form_key解析合并.md` §3.1 已下的"红线内已无安全的高收益 form_key
提升空间"结论互相印证，不是绕过了那轮工作找到新路子。

对 30.3% 的 ambiguous，现实结论是：**在"用确定性方法收敛"这条路径内，已接近能力
边界**。`ambiguous` 档给出全部候选 + 消歧方向提示（看接收变量来源），交模型结合
调用链自行判断，已经是红线内能做到的最优解。「复合 key（`form_key.field_key`）
精确查库」这条并行通路本身逻辑没问题（真传对了 key 就能精确查中），但它依赖"模型
能看出分析器看不出的归属"这个前提——抽样显示这类真实场景占比极低（≤1.1%），**不
建议实现**，除非未来出现新的、经过量化验证的高频结构性漏判模式。

若要继续提升，应回到 `docs/设计方案/数据包来源与form_key解析合并.md` §6 尚未做的优先级
（ORM 动态实体常量实参 / 反射目标解析 / 动态字段静态集合子集 / `unknown` 桶补证据 /
扩充 API 名单）——那份清单是基于全项目扫描已验证的方向，而非本文档探索的这两个
零散模式。
