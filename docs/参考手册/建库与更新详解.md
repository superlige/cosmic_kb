# 建库与更新详解（第一步：初始化你的项目）

> 本文是 [`README.md`](../../README.md)「对话式完成」里"建/重建 KB"那一步的详细版——供你核对
> agent 做得对不对，或想自己在终端逐条跑时参考。日常推荐直接用对话式安装口令让 agent 代劳。

`build` 需要两样输入：**你的二开 Java 源码根**，和**元数据**。元数据**推荐直连底层库现取**，
登录苍穹开发平台手动导出 dym/zip 只作为没有库权限时的备选方案（见下）。

## 推荐做法：直连底层库，元数据不用手动导出

找底层库管理员申请一个**只读**账号，配好之后 `build` 时**不用给任何 dym/zip 文件**，元数据
（包括你自己的二开单据，也包括扩展单据继承的原厂标准字段）全部由工具自动现取：

```powershell
cosmic_kb db-meta --init-config                      # 生成配置模板 cosmic_db.json，填 host/port/账号/schema
$env:COSMIC_DB_PASSWORD = "..."                       # 口令走环境变量，不写进配置文件
cosmic_kb db-meta --check --config cosmic_db.json     # 测试只读连接

cosmic_kb build "D:\项目源码根" --db-config cosmic_db.json   # meta 位置参数留空，纯连库建库
```

- 连接**强制只读**（多重防线，见 `cosmic_kb/dbmeta/`），只取 `t_meta_formdesign` /
  `t_meta_entitydesign` / `t_botp_convertrule` 三张表的数据，不落公网、结果照旧只进本机 KB——
  不违背"本地优先"红线。
- 为什么推荐这条路而不是手动导出：手动导出的 ISV 包**只含你自己二开的部分**，继承来的原厂
  标准字段根本不在包里，跳过直连库这一步，扩展单据的字段级排障会**结构性半盲**（原厂那部分
  字段连 `unknown` 都标不出来，因为元数据里压根没有这条记录）；直连库能把这两类一次性都补
  齐，还免去每次去开发平台手动导出、解压、归类文件的麻烦。

## 更新逻辑是什么

> **当前限制：不支持增量同步**。本项目自己的二开元数据（`fisv` = 你项目自己的实施方标识，
> 也就是 isv 开头这部分）每次 `build` 都会**全量重新拉取**该 ISV 下当前完整的
> form/entity/转换规则集合，不是"只拉自上次同步以来变更过的部分"。原厂标准单据不受此
> 影响，仍按三类确定性信号**精确发现**代码里引用到的 key 再现取，不会整表拉取库里所有
> 原厂表单。

直连库拉元数据分两类，规则不一样：

| 类别 | 拉取范围 | 更新逻辑 |
|---|---|---|
| **你自己的二开**（form/entity/转换规则，`fisv` = 你项目自己的实施方标识） | 每次 `build` 都拉该 ISV 下**当前完整**的记录集合 | **同 key 整条替换**（不是字段级合并，删掉/改名的字段不会残留），不支持只拉增量变更 |
| **原厂标准单据**（继承母体，如 `bd_customer`） | 每次 `build` 都按三类确定性信号（扩展母体 / ORM 查询 / 操作执行）**自动发现**代码里实际引用到的 key，现取这些 key 的最新值 | 不会不加区分地拉库里所有原厂表单（原厂页面太多，会引入大量噪声）；只精确现取代码里真正引用到的那几个 key |

也就是说：
- **日常重扫**（源码或元数据有更新后），原地重跑同一条命令即可，自己的二开部分每次都会
  全量校验一遍最新内容、原厂部分照旧精确现取：
  ```powershell
  cosmic_kb build "D:\项目源码根" --db-config cosmic_db.json
  ```
- **库里有多个二开 ISV、工具无法唯一确定该同步谁**时会报错，显式指定：
  ```powershell
  cosmic_kb build "D:\项目源码根" --db-config cosmic_db.json --isv <你的ISV标识>
  ```
- **代码里有原厂标准单据被自动发现漏掉**（没被三信号命中），用 `--vendor` 手动追加：
  ```powershell
  cosmic_kb build "D:\项目源码根" --db-config cosmic_db.json --vendor bd_customer bd_supplier
  ```
  想先看会自动拉哪些 key、不实际摄取：
  `cosmic_kb db-meta --discover "D:\项目源码根" --db cosmic_kb.db`。

## 原厂标准单据是怎么被"发现"的（三类确定性信号）

上面表格里"原厂标准单据"那一行说"按三类确定性信号自动发现"，具体是什么、为什么这么设计，
展开说一下。

**先说为什么不能简单粗暴地"整库全拉"**：苍穹平台底层库里原厂标准单据页面成百上千（`bd_customer`、
`bd_supplier`、`bos_user` ……），真正和你这个二开项目有关系的可能只有十几个。整表拉取会引入
大量与本项目无关的噪声，拖慢 `build`、污染 KB。所以工具的策略是：**只拉代码里真正用到的那几个
key**，靠静态分析在你的 Java 源码里找"确定性证据"，证据不够就宁可不拉（红线：宁可 unknown，
不可臆造）。

三类信号命中即视为"必摄取"（不是"可能相关"的模糊打分，是确定性判断）：

| 信号 | 触发条件 | 典型场景 |
|---|---|---|
| **① 扩展母体** | 本地元数据里存在一个扩展模型（`InheritPath` 非空 + 命名匹配你项目 ISV 前缀，如 `cqkd_bd_customer_ext`），它扩展的母体 fnumber（如 `bd_customer`）必摄取 | 你在客户资料基础上加了自定义字段，做了"基础资料扩展" |
| **② ORM 查询** | 代码里调用 `BusinessDataServiceHelper.load` / `loadSingle` / `loadFromCache` / `loadSingleFromCache` / `newDynamicObject`，或 `QueryServiceHelper.query` / `queryOne` / `exists`，去查询一个**不在本地元数据里**的实体 | 插件里 `BusinessDataServiceHelper.loadSingle(id, "bd_customer")` 查了原厂客户资料 |
| **③ 操作执行** | 代码里对一个**不在本地元数据里**的实体执行操作：`OperationServiceHelper.executeOperate`/`execOperate`、`DeleteServiceHelper.delete`、`SaveServiceHelper.save`/`update`/`saveOperate` | 插件里对某个原厂单据触发了"提交"/"删除"/"保存"操作 |

几个容易问到的细节：

- **信号②③怎么知道该拿哪个参数**：工具按这几个方法各自的**重载签名**精确定位"实体名"参数
  在第几位（比如 `loadSingle(pk, entity)` 是第 2 位，`loadSingle(pk, entity, fields)` 又是
  第 2 位，但 `loadSingle(entity, fields, QFilter[])` 变成第 1 位），参数值可以是字符串字面量，
  也可以是引用到的常量（自动解析 `ConstantTable` 里的常量定义）。**分不清是哪个重载**（比如
  `load` 的 `(pks, type)` 重载，参数位形状对不上）就直接跳过，不猜。
- **`save(var)` 这种传变量的怎么办**：只在**同一个方法体内**往回找 `var` 是不是刚从某次
  ORM 查询/构造赋值来的（比如 `DynamicObject var = BusinessDataServiceHelper.loadSingle(...)`），
  能对上就取那次查询的实体名；跨方法、跨类的数据流**不追踪**——这是为了不做"全程序数据流分析"
  这种成本和不确定性都太高的事，宁可漏掉几个候选。
- **过滤链**（避免把噪声当成候选）：
  - 已经在本地元数据里的 key（表单/实体/字段三个维度都算）直接排除，不会重复摄取；
  - 值的形状必须像一个 key（小写字母数字 + 下划线，且至少带一个下划线），排除掉长得不像
    key 的普通字符串；
  - 以你项目自己 ISV 前缀开头的（比如 `cqkd_xxx`）排除，那是你自己的二开实体不是原厂的；
  - 苍穹平台物理列/表名前缀 `fk_`/`tk_`（外键列/物理表名，形状和实体 key 一样但语义不同）
    排除。
- **候选怎么排序**：命中信号①（扩展母体）的排最前面，因为这是最确定的信号；其余按信号②+③
  命中次数从高到低排。每个候选最多附 5 条证据行号（`文件路径:行号`），方便你核对工具为什么
  认为这个 key 该拉。

**怎么在实际摄取前预览会拉哪些 key**（不连库、不摄取，纯扫描）：

```powershell
cosmic_kb db-meta --discover "D:\项目源码根" --db cosmic_kb.db
```

会列出每个候选 key、命中了哪些信号、命中次数、证据行号。确认没问题后，`build --db-config`
会自动跑同一套发现逻辑并直接摄取；如果发现有该拉但三信号都没命中的（比如代码里用了工具
识别不了的写法），用 `--vendor` 手动追加即可（见上面"更新逻辑是什么"一节）。

## 备选方案：本地导出 dym/zip（没有库权限 / 离线环境时用）

拿不到底层库账号（网络隔离、权限申请不下来等），退回到登录苍穹**开发平台**手动导出：

- 找到开发平台的应用导出功能，把你要排障的**二开应用全量导出**，会得到整包 `.zip`。
- 找到单据转换开发导出二开的**转换规则**（单据间取数映射，后缀 `.cr`）——本工具同样支持解析，
  建议**一并全量导出**，覆盖率会更完整。
- 把导出的这些文件（`.dym` / `.cr` / `.zip`，不管几个）统一放进**同一个文件夹**，不用手动
  解压、不用分类。

```powershell
cosmic_kb build "D:\项目源码根" "D:\元数据"
```

- 第一个参数是 Java 源码根；第二个参数直接指向**刚才存放导出文件的那个文件夹**（也可以是单
  个 `.zip` 包路径）——工具会自动扫描识别里面的 `.dym` / `.cr` / 压缩包，不需要你手动区分类型
  或逐个指定。
- **这条路径拿不到原厂标准字段**，扩展单据的排障结论只能诚实标"半盲/unknown"——如果之后能
  申请到只读账号，随时可以在这份 KB 基础上补一句 `--db-config` 重跑 `build`，两条路径可以
  叠加使用，不冲突。
- KB（`cosmic_kb.db`）**默认随源码根落盘**，一项目一库、互不覆盖。项目源码或元数据有更新时，
  重新导出一次、原地重跑 `build` 即可重新扫描（这条路径没有时间戳增量，每次都是用当次导出的
  文件重新解析）。
