# 返回值字段词典（工具到底回了什么）

> 面向对象：想直接读 CLI `--json` / MCP 原始返回的人（写脚本、调试、或想搞清楚 agent 为什么
> 这么答）。日常用 agent 提问不需要读这份文档——agent 会自己解释证据，这里是给"想看原始证据
> 长什么样"的人看的参考。

工具返回（CLI `--json` / MCP）刻意用了一批专有字段名（`unlocated` / `dynamic_writers` /
`possible` …）。它们都不是随手起的——每个对应「老项目分析天生不完整」里的一种**确定性程度**。
读懂它们，才能分清哪些是确诊、哪些是「值得人工/大模型去核的工作单」、哪些被截断需翻页。下面按
工具逐一解释。

## 贯穿所有工具的通用约定

| 字段 | 真实含义 |
|------|---------|
| `summary` | **真实总数（永远全量）**。无论返回怎么被裁剪，这里的计数都是库里的真值——判「完整 vs 截断」先看它。红线 #4。 |
| `*_total`（如 `groups_total` / `fields_total`） | 该列表的真实条数。返回里只展示了一部分时，全量数在这。 |
| `*_capped` / `capped` / `sites_capped` / `methods_capped` | **被设计内 cap 截掉的条数**（>0 表示这一段没列全）。不是丢数——总数在 `*_total`，被截条目可翻页取回。 |
| `next_cursor` / `*_next_cursor`（如 `"unlocated@4"`） | **翻页游标**：把这个值原样作 `cursor=` 再调一次同一工具，即取回该段下一页（`page.items` + 新 `next_cursor`），循环到 `null` 即把被截条目**一条不漏取全**。 |
| `note` | 人读提示：为何没命中、被截了该怎么收窄/翻页、tree-sitter 没装等。 |
| `confidence` / 三态 | `confirmed`（确诊）/ `likely`（很可能）/ `unknown`（判不准，标 unknown 不臆造）。 |
| `persists`（落库三态） | `yes`=落库（写进数据库，CLI 显示 ✅落库）；`no`=仅内存（—内存）；`unknown`=存疑（❓存疑，缺保存链路一律判这个，不臆造）；`na`=不适用。 |

## `trace` —— 字段排障旗舰（参数最多，重点看这个）

输入字段、输出「谁读写它/在哪个事件/是否落库」。返回里**三种"命中桶"按确定性递减**排列：

| 字段 | 真实含义 |
|------|---------|
| **`status`（可能是 `"need_clarification"`）** | 只在**字段跨单据歧义**时出现：字段 key 在元数据里挂在 ≥2 个不同单据下、且调用没指定 `form_key`。此时返回体只有 `field_key`/`status`/`filter`/`occurrences`/`note` 五个键，`groups`/`summary`/`unlocated`/`dynamic_writers`/`coarse` 都不存在——先看 `occurrences` 挑一个单据，加 `form_key` 或用 `单据.字段` 点号格式再查。同单据内跨层级/分录不算这种歧义，不触发。 |
| `field_key` / `field_name` | 被查字段标识 + **已核对的真实中文名**（来自元数据，引用时照抄，**别按拼音/命名惯例猜**）。同一 key 跨多坐标且名字不同时 `field_name` 为 `null`，消歧看 `occurrences`。 |
| `filter` / `precise` | 本次查询的坐标过滤（form/entry/level）、是否精确到单一坐标。 |
| `occurrences` / `occurrences_total` | 该字段在**元数据里的定义坐标**（单据·层级·分录）——即"这个字段名在哪些单据上有定义"的**消歧菜单**，不是读写记录。 |
| `groups` / `groups_total` / `groups_capped` | **确诊命中**：按坐标 (单据·层级·分录) 分组，每组列该坐标下读写它的类。字段定义单一但被跨单据插件读写（`plugin_cross_form`）时证据仍可能命中多张单据，故 `groups` 可能被裁——真实组数在 `groups_total`。若字段本身在元数据里跨多张单据**定义**且未指定 `form_key`，`trace` 会直接反问（见上方 `status`），不会走到这里。 |
| **`entry_chains`** | 字段读写的**插件事件入口目录**，按 `(access_class, access_method)` 去重。读写节点只带短 `entry_ref`，到本目录找“入口事件 → … → 实际读写方法”的链；目录被裁时用 `entry_chains@offset` cursor 翻页，避免同一链随每个源码行重复导致返回爆炸。 |
| `groups[].writers` | 该坐标的**写入**，按写入类合并：`classes[]` 每类一份，行号/落库/via 列在该类的 `sites`。`total`/`capped`/`sites_capped` 同通用约定。 |
| `groups[].readers` / `readers_overview` | 读取。默认只回**按类计数概览** `readers_overview`（最省字节）；要读取明细另调 `access='read'`，回 `readers`（类→方法，直接读源码）。 |
| `possible` | **存疑命中**：同单据同字段、但**层级/分录对不精确**（碰哪级分录判不准）的读写。不丢弃、降一档放这（"宁可多列也不漏"）。 |
| **`unlocated`** | **「反推来源单据」工作单**：这些读写**确实碰了被查字段**，但来源 DynamicObject **属于哪张单据没钉出**（`form_key=None`）。按方法去重，每条带 `plugin_form_label`（该插件注册的单据，**只是线索非确诊**，别当确定来源）+ `null_reason`（见下）。要确认它改的是哪张单据，读源码反推。 |
| **`dynamic_writers`** | **「字段钉不出」的动态写入候选**：代码对运行时/配置决定的字段集做泛化写入（循环遍历/字符串拼接 key/引用外部常量），静态钉不出**具体改了哪个字段**，但可能正改本字段。按"该读方法"去重列出（`methods[]` + `calls` + `by_cause`），交人/大模型读源码定性。⚠️ 与 `unlocated` 对称：`unlocated` 是钉不出"哪张单据"，`dynamic_writers` 是钉不出"哪个字段"——**两者都是工作单不是结论，别直接当成"谁改了它"的答案**。 |
| `coarse`（仅 `access='read'`） | **粗扫疑似盲点**：纯正则把字段标识当源码字面量搜到、但高精度分析没覆盖的位置（`coarse_only` + `locations` 带行号）。候选非确诊，供人工核对扫描盲区。 |
| `note` 里的 ⚡ 提示 | 写入点来自**操作插件**且该操作存在**程序化触发点**（别处代码 `executeOperate`/`invokeOperation` 触发该操作，设计器不展示）时，note 点名确切的操作坐标 + 触发点计数——排查"没人手动执行操作字段怎么变了"时，照提示转 `trace(该操作坐标, kind="operation")` 查触发链（明细不内联，按需查询）。 |
| `summary` | 真实总数集合：`writers`/`readers`/`persisting_writers`(落库写入数)/`uncertain_writers`(存疑写入数)/`plugins`/`forms`/`coords`(坐标组数)/`possible`/`unlocated`/`dynamic_writers`/`annotation_writers`(注解反射映射写入数) + `unlocated_by_reason`(未定位成因直方图)。 |

### `null_reason` —— `unlocated` 每条/每段的成因码（该不该追）

`form_key=None` 的行为何 None，决定了"值不值得顺源码反推"。8 个**互斥**成因码：

| 成因码 | 含义 | 该追吗 |
|--------|------|-------|
| `basedata-ref` | **读取**基础资料引用对象自身的字段，本就无业务单据坐标 | **正确 None，无需追** |
| `basedata-write-suspect` | **写到**基础资料引用对象——苍穹不会取基础资料再保存，疑似来源变量被误绑、真实来源单据未定位 | **继续追**（扫描器待修，非正确 None） |
| `dynamic-entity` | ORM 实体名是运行时变量/拼接，静态不可钉 | **正确 None，无需追** |
| `helper-caller-unknown` | helper 的 DynamicObject 入参，调用方未安全收敛 | 值得读源码反推 |
| `local-or-container-source` | 本地 new/Map/返回值等容器来源未识别 | 值得读源码反推 |
| `model-context` | `getModel()`/模型形参写入，但插件未注册绑定单据 | 读源码/补元数据可定（多为未注册表单插件） |
| `field-key-undeterminable` | 字段 key 本身就钉不出（动态/拼接/外部常量/歧义） | 来源讨论无意义 |
| `unknown` | 暂无足够证据归因 | 先补证据再判断 |

全量成因分布在 `summary.unlocated_by_reason`（真实总数恒在此）。

### `trace kind="operation"` —— 操作坐标追踪（程序化触发链，隐藏坑 #1）

`trace("单据.操作key", kind="operation")`（CLI `trace "单据.操作" --kind operation`）。坐标判别
**纯显式**：不传 `kind="operation"` 一律按字段坐标解释，不做自动猜测（同一 key 可能既是字段又是
操作）。裸操作 key 唯一命中自动定位，跨单据返回 `need_clarification`（`candidate_forms` 列候选）。

| 字段 | 真实含义 |
|------|---------|
| `operation` | 操作元信息（key/名称/类型）。操作 key 有触发点但**不在元数据操作集**（平台默认操作等）时为 `{key, in_metadata: false}`，note 注明不可考——证据照给，本体不臆造。 |
| `plugins` | 本操作绑定的操作插件（触发链的"执行体"）。 |
| `triggered_by` | **谁的代码触发了本操作**：`executeOperate`/`invokeOperation` 调用点，每条带 `caller_class`/`caller_method`/`line`/`source_relpath`/`via`/`op_key_resolution`/`caller_forms`（调用方插件绑定的**上游单据**——对上游继续 `kind="operation"` 递归即拼出 A→B→C 跨单据链；KB 只存单跳事实，多跳链由 agent 递归查询拼出）。 |
| `unresolved_inbound` | **无法静态排除是本操作**的嫌疑触发点全量（非空才出现），每条带 `suspect_reason` 成因：`op_unresolved`=目标已钉本单据但操作 key 真正解不出；`target_unresolved`=操作 key 正是本操作但**目标单据解不出**（表单插件绑多张单/目标动态拼接的外发即此形态——挂不上操作坐标的外发已并入，无需补查 `bill.outbound_triggers`）；`both_unresolved`=两者都解不出（可能是任何操作的调用，最弱嫌疑排最后）。已解析为其他 key 的调用无论是否进入元数据操作集，都必须排除，不得串成本操作嫌疑。`triggered_by`+此段合起来即对本操作程序化调用的完整答案；排"没人碰操作怎么执行了"时必查，读源码定性。 |
| `triggers_downstream` | 本操作插件的代码**又对外触发了谁**（级联链下行）。每条带目标坐标 + `target_form_name`（已核对目标中文名）+ `next_trace`（下一跳 trace 坐标，目标/操作解不出时为 null 不臆造）。非空才出现。 |
| `summary` / 分页 | 三段真实总数恒在 `summary`；被 cap 的段带 `*_next_cursor`（`"triggered_by@20"` 等），翻页协议与字段模式同一套（顶层 `pagination` 门）。 |

## `bill` —— 单据钻取

| 字段 | 真实含义 |
|------|---------|
| `form` / `stats` | 单据元信息 + 计数（实体数/字段数/操作数/插件数/被触达字段数）。 |
| `entities` / `fields` / `operations` / `plugins` / `bindings` | 该单的实体、字段元、操作集、插件清单、动态绑定（各有 `*_total`，被截带 `*_next_cursor`）。 |
| `operations[].programmatic_trigger_count` | **最小发现性信号**（隐藏坑 #1）：该操作被别处代码程序化触发的次数（非零才出现）。明细不内联——用 `trace("单据.操作key", kind="operation")` 按需查（见下方 `trace kind="operation"` 一节）。`stats.programmatic_trigger_count`=指向本单据的触发点总数（含解不出的），`stats.unresolved_trigger_count`=其中操作 key 解不出/不在操作集的条数。 |
| `outbound_triggers` | 本单据绑定插件的代码**对外**触发了哪些别的单据的操作——单据级**影响面**视图（改本单插件前评估会波及谁）。`target_form_key=null` 表示目标单据解不出——"不知道会炸到谁"本身就是要暴露的风险；该切片已同时并入对应操作坐标追踪的 `unresolved_inbound`，查"谁调用了某操作"用 `trace(kind="operation")` 一次即完整，本节不是必查补充。 |
| `entity_touch` | 按实体分组的**字段触达**：每字段被触达的逐条事件已折叠为「写/落库/读」计数（`writers`/`persisting`/`readers`）。每行带 `trace` 锚点（`"trace 单据.字段"`）——要看"某字段谁改的/在哪个事件/是否落库"，照它对该字段 `trace`。 |
| `risk_bindings` | 桥接有风险的绑定（如插件类没在源码命中），通常很少、整列内联。 |

## `resolve_fields` —— 字段名 O(1) 核对

手上有一串字段/分录/单据标识、想核对真实中文名时用（大模型自读源码识别出标识后，用本工具精确
核对，不凭命名惯例猜）。

返回 `{"resolved": {key: [item, ...] | null}}`，每个 item 带 `kind` 判别：

- `kind:"field"`：字段定义 —— `{name, form_key, entity_key, level, field_kind, field_type, access}`。
  `access` 是派生取值语义：**多选基础资料字段（MulBasedataField）也用 `getDynamicObjectCollection()` 取集合，不是分录行**——取分录还是基础资料取决于 key，别凭 API 名当分录。
- `kind:"entry"/"subentry"/"header"`：**分录容器** key（不是字段 key）—— `{name, form_key, level, parent_key, access}`。
- `kind:"form"`：**单据(表单)标识** —— `{name, form_key}`，字面量是表单 key（如
  `BusinessDataServiceHelper.load("cqkd_invoic_apply", ...)`）而非字段/分录 key 时命中。
- 支持复合限定符：`"单据.字段"` / `"分录.字段"` / `"单据.分录.字段"`，与 `trace` 同一套点号
  坐标写法。限定符不含该字段时返回 `mismatched_form`，诚实指出字段真实所在单据/分录，不悄悄回退。
- 同 key 跨多坐标 → 回 list 全摆出（**不替你选**，消歧靠上下文）；**钉不出回 `null`——标 unknown，绝不臆造**。

## `callers` —— 调用链反查 / 苍穹入口可达性 · 死代码判定

输入 `Class.method`（简单类名跨包重名会像 `trace` 一样反问），先给该方法的**全部静态调用点**，
再自动沿 `call_edge` 反向 BFS **一路回溯到插件事件入口**，反查元数据是否绑定单据/操作、绑定是否
启用——"0 个调用点"不直接等于死代码，必须结合入口注册状态才能下结论（红线 #4）。

| 字段 | 真实含义 |
|------|---------|
| `status: "need_clarification"` | 简单类名跨包重名，返回体只有 `candidates`（每条 `locator` + `call_site_count`），挑一个完整 locator 再查，不按调用数猜选。 |
| `target` | 消歧后落定的目标：`target_fqn`/`method`/`source_relpath`/`target_kind`（`project`/`jar`/`jdk`，后两者不跑入口分析）。 |
| `callers` | 全部静态调用点：每条 `caller_fqn`/`caller_method`/`line`/`col`/`source_relpath`/`kind`（`call`/`method_reference`）/`resolution`（`expr`/`scope`/名字兜底）/`confidence`。 |
| `summary` | 真实总数：`call_sites`/`caller_methods`（去重后的调用方法数）/`method_references`（`Class::method` 引用计数）/`by_resolution`。 |
| `resolution_coverage` | 建库时符号解析层质量：`status`/`coverage`/`strong_zero_evidence`（覆盖 ≥95% 且无失败文件才算强证据）——0 个调用点时先看这个，覆盖不足不能断言"查无调用方"。 |
| **`entry_analysis`** | **入口可达性判定**，verdict 五值（`not_analyzed` 仅平台/JDK 目标）： |
| └ `entry_reachable` | 至少一个入口/边界插件类已确认 `registered_enabled`，静态可达。 |
| └ `entries_inactive` | 找到的入口均已禁用或未注册，**疑似不可达**（likely，永不 confirmed——不排除反射/动态注册这类静态分析看不到的入口）。 |
| └ `entry_unverifiable` | 注册/启用状态无法确认；或本应判 `entries_inactive` 但搜索被截断/符号层弱化，**降级**为无法确认（红线 #4：负向结论不带截断冒充完整）。 |
| └ `no_entry_found` | 沿 `call_edge` 向上追不到任何插件事件入口；`resolution_coverage.strong_zero_evidence` 且无截断才给 `likely`，否则 `unknown`。 |
| `entry_analysis.confidence` | `confirmed`/`likely`/`unknown` 三态，语义同通用约定，但此处 `entries_inactive`/`no_entry_found` 永远到不了 `confirmed`。 |
| `entry_analysis.entries[]` | 回溯到的入口/边界插件类，每条 `class`/`terminal`（`self`/`entry`/`plugin_boundary`）/`events`（命中的事件+阶段，非空才有）/`registration`。 |
| `entries[].registration` | **元数据注册状态反查**，`status` 六值： |
| └ `registered_enabled` | 已注册且至少一处有效绑定已启用；`bindings[]` 列出绑定明细（表单/操作/转换规则，convert 类型的启用状态查的是 `convert_rule.enabled` 而非 `plugin.enabled`）。 |
| └ `registered_disabled` | 已注册，但全部有效绑定已禁用。 |
| └ `registered_enabled_unknown` | 已注册，启用状态未知（NULL 或转换规则悬空）。 |
| └ `orphan_unregistered` | KB 有该类插件种类的注册表（`form`/`list`/`op`/`writeback`/`convert`），但查无绑定——元数据范围内未注册（不排除动态注册）。 |
| └ `orphan_unverifiable` | KB **未接入**该类插件种类的注册配置表（如调度计划/开放平台/工作流/校验器），如实报"无法确定是否平台注册"，不臆造。 |
| └ `no_registration_evidence` | 既非元数据绑定插件，也非孤儿插件基类命中，无注册证据兜底。 |
| `entry_analysis.chains` | 代表性入口链（触发点→…→入口，最多列 `_EA_SHOW_CHAINS` 条），MCP 分页首页外的页面只留 verdict/confidence，全链看首页。 |
| `entry_analysis.chains_truncated` / `search_truncated` | BFS 是否被 cap 截断——非零/`true` 时负向 verdict 已按上面规则自动降级，此处只是留痕。 |
| `note` | CLI/MCP 富投影里追加的入口可达性一句话小结（中文 verdict + confidence）；`entry_analysis.note` 是该判定本身的成因说明。 |
| `pagination`（仅 MCP `callers_compact`） | 与通用约定同一套 `complete`/`next_cursor`（`callers@N`）协议；首页外的 `entry_analysis` 精简为 `{verdict, confidence, note}`，避免热点方法把入口证据挤爆预算。 |



## `cosmic_semantics` —— 苍穹领域语义文档
