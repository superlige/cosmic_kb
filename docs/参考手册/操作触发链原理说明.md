# 操作触发链原理说明（`trace "单据.操作key" --kind operation`）

> 面向：想知道"这条链到底是怎么查出来的""哪些写法能查到、哪些查不到"的人工排障者/大模型宿主。
> 不重复讲返回字段怎么读（那部分见 `docs/参考手册/trace返回详解.md` 的通用截断/分页机制），
> 本文只讲**这个功能的识别规则边界**——它是"隐藏坑 #1"（设计器不展示的程序化操作触发）
> 的取证手段，源码见 `cosmic_kb/java/op_trigger.py`（提取）+ `cosmic_kb/report/op_trace.py`
> （三段查询）+ `cosmic_kb/report/entry_chain.py`（入口回溯）。

---

## 一、要解决什么问题

苍穹设计器里"这单的 audit 操作谁会执行"只能看到界面按钮/工作流配置。但真实项目里大量操作是
**代码里硬调**的：

```java
OperationServiceHelper.executeOperate("audit", "cqkd_b", bills, OperateOption.create());
```

这类调用不出现在任何设计器视图里，接手者只能全局搜源码硬拼。`trace --kind operation` 把这类
调用点结构化成"谁（类·方法·行号）触发了哪张单据的哪个操作"，按需查询，不预拼多跳链。

---

## 二、识别规则：目前认哪几种调用写法

`op_trigger.find_operation_triggers` 只认**语法上是方法调用**（`method_invocation`，
不含 `方法引用`/`Class::method`、不含反射 `Method.invoke`）里的两类调用：

| 写法 | 操作 key 在哪 | 目标单据在哪 | 备注 |
|---|---|---|---|
| `OperationServiceHelper.executeOperate(opKey, entityNumber, ...)` | 第 1 个实参（下标 0） | 第 2 个实参（下标 1） | 平台标准触发入口，真实项目里能查到的绝大多数写法 |
| `xxx.invokeOperation(opKey, ...)` / `invokeOperationXxx(...)` | 第 1 个实参（下标 0） | **不在实参里**——由 enclosing 类的元数据绑定单据推断 | 触发的是"当前视图绑定的单据"自己的操作 |

> 扫描规则曾多识别一个 `execOperate`（当作 `executeOperate` 的"同签名别名"），经用户核实
> **苍穹平台并没有这个方法**后，2026-07-20 已从 `op_trigger.py`/`persistence.py`/
> `dbmeta/discover.py` 的识别规则里一并删掉，只保留 `executeOperate`。

判断"是不是 `OperationServiceHelper` 的调用"分两层证据（后者优先）：

1. **符号解析证据**（阶段 12 编译期符号解析命中时）：接收者的声明类型精确等于
   `kd.bos.servicehelper.operation.OperationServiceHelper`，`receiver_source="symbol"`。
2. **文本兜底**（符号解不出、或项目未接入 classpath 时）：接收者表达式的最后一段文本
   等于字面 `OperationServiceHelper`（如 `OperationServiceHelper.executeOperate(...)`），
   `receiver_source="text"`。

`invokeOperation` 系列反过来——**默认相信是平台调用**，只有符号解析明确证明它是"项目内某个
非 `kd.bos.*` 类自己写的同名方法"时才排除（避免自建工具类里凑巧叫 `invokeOperationXxx` 的方法
被文本匹配误伤；符号解不出时无法证伪，按平台调用处理，见下节"假阳性"）。

### 参数值怎么解出来（`ConstantTable.resolve_arg`）

实参不一定是字符串字面量，苍穹项目常把 key 定义成常量。解析按以下优先级/置信度落档：

| 实参写法 | 解析方式 | `resolution` | 置信度 |
|---|---|---|---|
| `"audit"`（字符串字面量） | 直接取值 | `literal` | 1.0 |
| `BillConst.OP_SUBMIT`（`类.常量`限定引用，类在本工程内可见） | 按类精确查常量表 | `constant` | 0.95 |
| `OP_SUBMIT`（裸常量名，工程内只有一处同名常量） | 按常量名唯一匹配 | `constant` | 0.85 |
| `OP_SUBMIT`（裸常量名，工程内多处同名但字面值不同） | 无法消歧 | `ambiguous` | 0.3 |
| 未知标识符（引用不到任何已扫到的常量，如跨模块/外部依赖里的常量） | 查无 | `unknown` | 0.0 |
| 局部变量 / 方法返回值 / 三元表达式 / 字符串拼接等**非字面量非标识符**实参 | 静态解不出该表达式的值 | `dynamic` | 0.0 |

`invokeOperation` 的目标单据不来自实参，而是"该调用所在类，在元数据里绑定了几张单据"：

| enclosing 类的元数据绑定情况 | `target_resolution` | 置信度 |
|---|---|---|
| 唯一绑定一张单据 | `binding` | 0.85 |
| 绑定了多张单据（同一插件类服务多个单据） | `ambiguous` | 0.3 |
| 元数据里查无绑定 | `unknown` | 0.0 |

**处处置信度，解不出一律留 `unknown`/`dynamic`，绝不按命名习惯猜字面值**——这是红线 #4 在本
功能上的落地。

## 三、返回的三段是怎么拼出来的

`operation_trace` 对 `"单据.操作key"` 坐标做的其实是三条独立的 SQL 查询 + 一次入口回溯，不是
一次"追踪算法"：

1. **`triggered_by`（入站命中）**：`operation_trigger` 表里 `target_form_key=本单据 AND
   op_key=本操作` 的行，即操作 key 与目标单据**都**静态解出且精确匹配的调用点。
2. **`unresolved_inbound`（入站嫌疑）**：**无法静态排除**是本操作调用的行，按解不出的
   那一侧分三种成因（谁调了本操作的完整答案 = ①+②合起来看）：
   - `op_unresolved`：目标已钉本单据，但操作 key 是 `dynamic`/`unknown`（解不出，不能排除）；
   - `target_unresolved`：操作 key 已精确匹配本操作，但目标单据解不出（`invokeOperation`
     绑多张单据的 `ambiguous`、或 `executeOperate` 目标实参是拼接/变量的 `dynamic` 都属此类，
     常见形态是"表单插件外发但目标钉不出"）；
   - `both_unresolved`：操作 key 与目标单据**都**解不出——这条对**任何**操作坐标都排除不掉，
     是最弱的嫌疑，排序永远垫底。

   已解出且**明确不等于**本操作 key 的调用点会被静态排除、不进这个列表（哪怕它没在元数据操作
   集里）——即"能证明不是你"就不算嫌疑，不会把 `refresh` 的调用点错算成 `updatetax` 的嫌疑。
3. **`triggers_downstream`（下行外发）**：本操作绑定的操作插件类，在它们的代码里又调了
   `executeOperate`/`invokeOperation`（目标≠本单据）的行——即"这个操作执行完，它自己又拉起了
   谁"。每条带 `next_trace` 坐标，级联链靠对 `next_trace` 反复 `trace --kind operation` 逐跳查，
   KB 不预拼多跳（避免环/组合爆炸）。

**`entry_chains`（入口回溯）** 是查询期在 `triggered_by`/`unresolved_inbound` 每一条上，沿
`call_edge`（阶段 12 编译期调用图）反向 BFS 找"谁调了这个 caller_method"，一路往上直到碰到
插件事件方法（`plugin_method` 事件表命中）为止。`terminal` 五种取值：

| terminal | 含义 |
|---|---|
| `entry` | 确认到达插件事件方法（事件表命中，`confirmed`/`likely` 取决于链上调用边是否全是编译期强证据） |
| `plugin_boundary` | 到达一个插件类（元数据绑定或孤儿插件基类命中），但该方法不在事件表——多是 task/workflow 等事件表未覆盖的插件种类 |
| `no_static_caller` | 向上追不到任何静态调用方——**不代表没有入口**，只是反射/定时任务注册/OpenAPI 派发这类动态派发方式，`call_edge` 天生追不到 |
| `depth_capped` | 超过回溯深度上限（默认 10 层）截断 |
| `callers_all_visited` | 上游成环，或汇入已探索过的更短路径 |

`entry_chains` 本身也是**这次查询才拼**，不是建库时预算好的多跳链。

## 四、目前支持回溯的写法（能查到的）

- `OperationServiceHelper.executeOperate("audit", "cqkd_b", ...)` —— 字面量双参，最标准形态。
- `OperationServiceHelper.executeOperate(BillConst.OP_SUBMIT, BillConst.ENTITY_B, ...)` —— 操作
  key/目标单据都走 `类.常量` 精确引用。
- `OperationServiceHelper.executeOperate(OP_SUBMIT, "cqkd_b", ...)` —— 裸常量名（工程内唯一）。
- `this.getView().invokeOperation("save")` —— 界面插件触发自身绑定单据的操作（目标靠元数据
  绑定推断，不用实参）。
- 操作 key 是局部变量/表达式解不出、但目标单据是字面量 —— 仍会入库（`op_unresolved` 嫌疑），
  只是拿不到具体 key，需要读源码定性。
- 表单插件绑定多张单据时 `invokeOperation` 触发 —— 目标 `ambiguous`，进 `target_unresolved`
  嫌疑，不会漏报也不会瞎猜是哪张单据。
- 装了阶段 12 编译期符号环境（classpath 齐全、JavaParser 能解出）时，**跨类多层转发**只要每一
  跳都落在 `call_edge` 里（即每一跳都是显式方法调用，非反射/动态代理），`entry_chains` 能一路
  BFS 回溯到最初的插件事件方法，哪怕触发点本身埋在 service/helper 深处、和插件类隔了好几层。

## 五、目前查不到 / 会漏报或标 unknown 的写法

- **反射调用**：`method.invoke(helper, "audit", "cqkd_b")` —— 语法上不是
  `OperationServiceHelper.executeOperate(...)` 的方法调用节点，扫描器根本不识别，静默漏掉
  （不会出现在任何列表里，包括 `unresolved_inbound`——因为它连"疑似调用点"都算不上）。
- **方法引用/Lambda**：`OperationServiceHelper::executeOperate` 这种 `Class::method` 引用——
  `op_trigger` 用的 `ax.iter_invocations` 默认不扫方法引用（`include_refs=False`），同样不识别。
- **静态导入后裸调用**：`import static ...OperationServiceHelper.executeOperate;`，随后裸调用
  `executeOperate("audit", "cqkd_b", ...)` —— 接收者表达式为空，文本匹配失效；若当前环境没有
  编译期符号解析（未装 java 环境/解不出 classpath）会**整条漏判**；有符号解析时能靠声明类型
  识别出来。
- **参数位置不同的自定义封装**：项目里包一层 `MyOpHelper.exec("audit", "cqkd_b")`
  再内部转调 `OperationServiceHelper.executeOperate`——只有内层真正调用平台方法的那一行会被
  记为触发点，外层封装方法本身不算（这是设计使然：只记"最终触点"，调用方是谁靠 `entry_chains`
  向上追）。若外层封装恰好和内层在同一个 `call_edge` 里能连上，向上追踪能看到；追不到则
  `no_static_caller`。
- **动态拼接的操作 key / 目标单据**：`executeOperate(opCode + "_v2", entityVar, ...)`——
  非字面量非标识符的表达式，静态解不出具体值，只能落 `dynamic`/`unknown`，需要人工读源码。
- **常量值来自外部依赖/未随包源码**：引用的常量类没有源码（如平台内置常量、未导出的 jar），
  常量表里查无此常量名，落 `unknown`。
- **跨工程边界的调用链**：`entry_chains` 只在 KB 已扫到的源码范围内沿 `call_edge` 走，调用链
  一旦跳出扫描范围（如调了外部 jar 里的方法、该方法内部再回调本工程）无法追踪。
- **定时任务 / 开放平台(OpenAPI) / 工作流触发的入口本身**：这类派发是平台运行时动态完成的，
  没有一条"谁调用了插件方法"的静态代码边，`entry_chains` 只能停在 `no_static_caller` 或
  `plugin_boundary`（如果那个类本身被识别为孤儿插件基类），**不能把它当成"死代码"或"无人调用"
  的证据**，必须读源码/查配置表定性。

### 一个需要警惕的假阳性来源

`invokeOperation` 系列**默认相信是平台调用**——没有编译期符号解析、或符号解不出时，任何名字
以 `invokeOperation` 开头、且至少有一个实参的方法调用都会被记成触发点。如果项目里自己写了一个
同名但语义完全不同的方法（如 `MyDispatcher.invokeOperation(...)`），在无符号解析兜底的情况下
会被误判为程序化操作触发，需要人工核实 `receiver_source`：`"text"` 且当前 KB 未装编译期符号
解析（`cosmic_kb doctor` 查 java 环境）时，对这条更要留一分怀疑。

## 六、排障建议

- 查"这操作没人手动点怎么执行了"：先看 `triggered_by`，**再看 `unresolved_inbound`**——
  两者合起来才是完整答案，只看前者会漏掉目标/key 解不出但很可能就是它的调用点。
- `entry_chains` 里 `no_static_caller`/`plugin_boundary` 不代表"查不到就是没有"，是"静态分析
  的边界在这，读源码继续查"的信号。
- 想确认某条触发点证据是否可靠：看 `receiver_source`（`symbol` 优于 `text`）和各字段的
  `resolution` 档位（`literal`/`constant` 优于 `ambiguous`/`dynamic`/`unknown`）。
- 反射、方法引用、动态拼接这几类写法本工具原理上查不到，怀疑有这类隐藏触发时只能人工全局
  搜索源码关键字（如搜 `"audit"` 字符串本身，或搜 `getMethod`/`invoke` 附近的操作相关常量）。
