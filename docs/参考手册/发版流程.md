# 发版流程（Codex 代劳版）

> 目标：以后发版由 Codex 全程代劳，人工只提出目标版本；机械步骤统一交给
> `scripts/release.ps1`，不再在对话里逐条拼测试、提交、打包、上传和打 tag 命令。

## 标准发版：一条命令

Codex 审查完本地改动并写好 `docs/核心/V<版本>发版说明.md` 后，只执行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\release.ps1 `
  -Version 0.2.2 `
  -Summary "一句话说明本次改动"
```

想先看计划、不修改任何内容：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\release.ps1 `
  -Version 0.2.2 `
  -Summary "一句话说明本次改动" `
  -DryRun
```

## Codex 在运行前只做三件事

1. 审查全部本地改动，确认范围、版本类型和是否含敏感信息。
2. 若改了 KB schema，在发版说明里明确升级后必须重建 KB；未改 schema 不制造强制重建要求。
3. 新建 `docs/核心/V<版本>发版说明.md`，写清主要变化、升级注意和版本号。

随后把所有机械步骤交给脚本，不再重复执行针对性测试、第二套安装环境或多轮手工状态检查。

## 脚本自动完成什么

`release.ps1` 固定执行五段：

1. **准备**：检查 `main`、远端同步状态、版本号、发版说明和 tag；更新
   `cosmic_kb/__init__.py`，并调用 `make_dist.ps1 -RefreshOnly` 同步 README / 安装说明的固定版本口令。
2. **验证并提交**：只跑一次 `pytest -q`，通过后做 `git diff --check`，统一暂存、中文提交并推送 `main`。
3. **构建**：在干净 HEAD 上生成离线 zip 和 wheel，执行 `twine check` 并计算 SHA256。
4. **发布**：通过 Windows keyring 读取 Token，使用 UTF-8、无进度条模式上传 PyPI；随后读取 PyPI
   公共 JSON，核对公开 wheel 与本地 SHA256 完全一致。
5. **收尾**：创建并推送 `v<版本>` tag，输出 commit、PyPI 地址、产物路径和摘要。

## 为什么比旧流程快

- 全量测试只跑一次；不再先跑一轮针对性测试、最后又跑一轮全量测试。
- 不再手工调用两次 `make_dist.ps1`；`-RefreshOnly` 负责提交前同步口令，提交后只正式打包一次。
- 不再创建“本地 wheel 验证环境 + PyPI 公开安装环境”两套 venv。全量测试、`twine check`、包内资产测试和
  PyPI SHA256 一致性已经覆盖本项目发版的关键风险。
- 不再在对话里逐条执行和解释 `git add/commit/push/tag`、版本查询、PyPI 查询与哈希命令，减少工具往返和 token。
- 中途失败后直接重跑同一条命令：目标版本已提交时跳过测试/提交，PyPI 已有同摘要 wheel 时跳过上传，
  本地 tag 已创建但没推送时继续推送。

## 保留的安全门禁

以下检查不能为了省步骤删除：

- 只允许从 `main` 发布，且不能落后于 `origin/main`。
- 目标版本必须有发版说明，远端 tag 不得已存在。
- 有未提交改动时必须通过一次全量测试和 staged diff 格式检查。
- 离线包必须来自已提交的干净 HEAD，关键 Skills、semantics、模板和 `symsolver.jar` 仍由打包脚本校验。
- PyPI 公开 wheel 的 SHA256 必须与本地产物一致，摘要不同立即停止，不能覆盖或继续打 tag。
- PyPI 成功后才推 tag，避免“Git 显示已发布但 PyPI 实际没有包”。

## 本机只需配置一次 PyPI Token

```powershell
keyring set https://upload.pypi.org/legacy/ __token__
```

粘贴包含 `pypi-` 前缀的完整 Token。Token 存在 Windows 凭据管理器，不写入仓库、`.env` 或
`.pypirc`。后续由 `release.ps1` 自动调用 Twine，不再人工输入。

## 失败时怎么处理

- **测试、格式或版本门禁失败**：修复问题后重跑同一条发布命令。
- **上传中断**：先重跑同一命令；脚本会查询 PyPI，已有同摘要 wheel 就继续打 tag，没有才重新上传。
- **凭据缺失**：重新执行上面的 `keyring set`，然后重跑同一命令。
- **PyPI 已存在但摘要不同**：停止，不自动换版本、不移动 tag，由 Codex查明是否版本号已被占用。

发版完成的判定只有四个：PyPI 版本存在且摘要一致、远端 tag 指向发布提交、`main` 已推送、工作区干净。
