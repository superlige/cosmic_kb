# 发版流程（工具改完 → 发一个新版本）

> 本文回答一个问题：**二开工具本身（cosmic_kb 包）改完之后，我要怎么发一个新版本给使用者。**
> 和 [`分发给同事.md`](分发给同事.md) 的区别：那篇讲"把某个版本的包/整包 zip 递给别人"这一个
> 动作；本篇讲**从改完代码到对外可安装的完整发版链**——含版本号、安装口令刷新、PyPI 发布、
> 打 tag。日常只改文档、不动包代码时不需要发版。

## 为什么发版这件事有固定流程

对话式安装口令（README「对话式完成」里那段、`INSTALL-TOKEN` 标记对之间的内容）**把版本号写死**成
`cosmic-kb==<版本>`，让使用者的 agent 从 PyPI 装**这个确定版本**。所以"发版"本质是让三件事同时对上：

1. 包里的 `__version__`；
2. README / 安装说明.md 里安装口令写的版本；
3. **PyPI 上真的存在这个版本**（不发到 PyPI，使用者按口令 `pip install cosmic-kb==<新版本>` 会失败）。

`scripts/make_dist.ps1` 负责把①自动同步到②（按 `__version__` 改写口令块，无需人肉改占位符），
③要你手动 `twine upload`。下面按顺序把三步都串起来。

## 总览：一次发版做哪些事

| # | 步骤 | 命令 / 改动 | 谁做 |
|---|---|---|---|
| 1 | 确认改动已全绿 | `pytest -q` | codex |
| 2 | **Bump 版本号** | 改 `cosmic_kb/__init__.py` 的 `__version__`（唯一版本源，`pyproject.toml` 动态读取，不用改两处） | Claude |
| 3 | 更新验收台账 | `docs/核心/阶段验收.md`（有新功能点时） | Claude |
| 4 | 刷新安装口令 + 打离线整包 | `scripts/make_dist.ps1`（按新版本自动改写 README / 安装说明.md 的 `INSTALL-TOKEN` 块，并产出 `dist\cosmic_kb_dist_v<版本>_<日期>.zip`） | Claude / codex |
| 5 | 构建 wheel | `python -m build --wheel` → `dist\cosmic_kb-<版本>-py3-none-any.whl` | codex |
| 6 | **发布到 PyPI** | `twine upload dist\cosmic_kb-<版本>*` | 你 / codex（需 PyPI 凭据） |
| 7 | 提交 + 打 tag + push | `git commit` → `git tag v<版本>` → `git push --tags` | codex |

> 分工按项目惯例：Claude 只改版本号 + 台账（第 2/3 步），不 `git commit`、不跑全量测试；codex 跑
> `pytest -q` 与提交打 tag；`twine upload` 需要 PyPI 账号凭据，由你本人或 codex 执行。

## 详细步骤

### 1. 定版本号

语义化版本：修 bug → 补丁位（`0.1.0` → `0.1.1`）；加向后兼容的新命令/新返回字段 → 次版本位
（`0.1.0` → `0.2.0`）；有破坏性改动 → 主版本位。**只改一处**——`cosmic_kb/__init__.py`：

```python
__version__ = "0.2.0"   # 从 0.1.0 起
```

`pyproject.toml` 用 `[tool.setuptools.dynamic] version = { attr = "cosmic_kb.__version__" }` 动态读取，
不用同步改。

> 若本次改动动了 **KB schema**（`cosmic_kb/graph/schema.sql` 的 `schema_version`），在发版说明里提醒
> 使用者升级后要重跑一次 `cosmic_kb build`（或对话式的 `bootstrap apply` 续跑）重建 KB。

### 2. 刷新安装口令 + 打离线整包

`make_dist.ps1` 读 `__version__`，把 README / 安装说明.md 里 `INSTALL-TOKEN` 标记对之间的口令块整块
按新版本改写。**因为刚 bump 过版本，第一次跑会改写口令块 → 工作区变脏 → 脚本的脏检查会拦下让你先提交**，
这是设计好的（`git archive` 只导出 HEAD，未提交的口令改动不会进离线包）。所以顺序是：

```powershell
# ① 第一次跑：它改写 README/安装说明.md 的口令块，然后因为脏树停下
powershell -ExecutionPolicy Bypass -File scripts\make_dist.ps1

# ② 提交版本号 + 被改写的口令块（连同本次代码改动一起，由 codex 提交）
#    见下面第 4 步的提交

# ③ 提交后再跑一次：干净树，正常产出 dist\cosmic_kb_dist_v<版本>_<日期>.zip
powershell -ExecutionPolicy Bypass -File scripts\make_dist.ps1
```

急着本地验证、还没提交时可用 `-AllowDirty` 跳过脏检查——脚本会兜底把 staging 里 README/安装说明.md
的口令刷成当前版本，保证发出去的离线包版本一致；但正式发版仍应走"先提交再打包"，别把未提交状态发出去。

口令块只在版本号变化时才被改写，同版本重复跑 `make_dist` 幂等、不会脏树。

### 3. 构建 wheel

```powershell
pip install build
python -m build --wheel     # 产出 dist\cosmic_kb-<版本>-py3-none-any.whl
```

wheel 已是自包含（语义文档、模板、两份 Skill、`ok-cosmic-docs.db` 均随包），装了就能跑 `bootstrap`。

### 4. 发布到 PyPI（让安装口令真的能装到）

```powershell
pip install twine
twine upload dist\cosmic_kb-<版本>-py3-none-any.whl
# 或先传测试站验证：twine upload --repository testpypi dist\cosmic_kb-<版本>*
```

发布完，使用者把 README 里那段安装口令发给 agent，就会从 PyPI 装到这个新版本。

> **首次发布**需要先在 PyPI 注册 `cosmic-kb` 这个项目名并配好 API Token；在此之前，对外分发只能走
> [`分发给同事.md`](分发给同事.md) 的 wheel / 整包 zip（README 里也保留了"包尚未上 PyPI 时先手动安装"
> 的兜底说明，指向 [`手动安装详细教程.md`](手动安装详细教程.md)）。

### 5. 提交 + 打 tag + push

由 codex 执行（把本次代码改动、版本号、被 `make_dist` 改写的口令块一起提交）：

```powershell
git add -A
git commit -m "发布 v<版本>：<一句话改动>"
git tag v<版本>
git push && git push --tags
```

## 一页速记

1. 改 `cosmic_kb/__init__.py` 的 `__version__` → 2. 跑 `make_dist.ps1`（刷口令，脏树停）→
3. 提交版本号+口令 → 4. 再跑 `make_dist.ps1` 出离线 zip → 5. `python -m build --wheel` →
6. `twine upload` 发 PyPI → 7. `git tag v<版本> && git push --tags`。
